## TL;DR

- Clean Architecture and **The** Clean Architecture are different
  - **The** Clean Architecture ⊂ Clean Architecture
- The famous concentric-circle diagram represents The Clean Architecture ⊂ Clean Architecture
- What matters is the philosophy of Clean Architecture
- In Japanese, both Clean Architecture and **The** Clean Architecture become "Clean Architecture"
- This article is meant to **organize the distinction**; it does not explain the methodologies of Clean Architecture or The Clean Architecture

## The root of the confusion

In English, Clean Architecture and **The** Clean Architecture can be distinguished by whether **The** is present.
In Japanese, however, both Clean Architecture and **The** Clean Architecture become "Clean Architecture."
Although they look similar but mean different things, Japanese speakers have a hard time telling them apart unless the English notation is shown explicitly.

## The Clean series

Clean Architecture is one of the books written by [Robert C. Martin](https://amzn.to/40MCSdi) in what is known as the Clean series.

- [Clean Code: A Handbook of Agile Software Craftsmanship](https://amzn.to/4boZtCO)
- [The Clean Coder: A Code of Conduct for Professional Programmers](https://amzn.to/3PqXVj1)
- [Clean Architecture: A Craftsman's Guide to Software Structure and Design](https://amzn.to/4uORTsQ)
- [Clean Agile: Back to Basics](https://amzn.to/4suJIjM)
- [Clean Craftsmanship: Disciplines, Standards, and Ethics](https://amzn.to/4cZ2UBn)

My understanding is that it is called **Clean** Architecture because it is the architecture discussed in the book about architecture within the **Clean** series.
As [The merits and demerits of Clean Architecture](https://zenn.dev/adwd/articles/5d4a89262f4fc5) also mentions, the name Clean Architecture is easy to misunderstand.

## Clean Architecture

### Looking back at Clean Architecture

Clean Architecture is so strongly associated with the famous concentric-circle diagram of **The** Clean Architecture that the broader book is often overlooked.
Before reaching that point, however, the book explains general software development practices and programming techniques that are widely known.

Part I covers the impact and value of design and architecture in software development.
Part II covers structured programming, object-oriented programming, and functional programming.
Part III covers the five design principles known by the acronym SOLID.
Part IV covers component principles.
Then Part V explains Clean Architecture and leads into the famous The Clean Architecture.

Of these, Parts I through IV are not specifically about Clean Architecture.
They are closer to **common knowledge for developing maintainable software**.
Also, if you read Part II, you can see how meaningless it is to criticize a programming paradigm by pointing only to features of a programming language that claims to support some particular "XXX-oriented" style.
As I wrote in an earlier article, Java's language features do not lead to a serious argument that "object orientation is..."
Nor should we be misled by claims that reduce object orientation to encapsulation, inheritance, and polymorphism.

### What we can learn from Clean Architecture

As I wrote in "Looking back at Clean Architecture," [Clean Architecture: A Craftsman's Guide to Software Structure and Design](https://amzn.to/4uORTsQ) contains only very basic things.
However, for people who already treat those things as basic knowledge, it is probably not a book they would deliberately read in order to learn something new.
They know that there are books that explain each individual topic in more depth, so they would likely decide to read those instead.

I think I read this book around 2019 or 2020.
After working as an engineer for about seven years, what I now feel is that ordinary coders should write code with the ideas in Clean Architecture already in mind.
Whether the practices described in Clean Architecture should be applied depends on the organization and the system.
Still, I would like the basic ideas described in Clean Architecture, before it reaches The Clean Architecture, to be shared as assumed knowledge.
