## 抽象メソッド ― 実装を強制する

抽象クラスの本領は、**抽象メソッド**（Abstract Method）にあります。
抽象メソッドは、「**中身を書かず、『このメソッドがある』という約束だけを定めた**」メソッドです。

---

## 中身のないメソッド

第15章までの `Animal` では、`cry()` に「（動物の鳴き声）」という標準のふるまいを書いていました。
ですが、はじめに述べたとおり、「動物そのもの」の鳴き声は決められません。
本当に言いたいのは、「**すべての動物は `cry()` を持つべきだ。でも、中身は、それぞれの動物が決める**」ということです。

これを表すのが、抽象メソッドです。
`abstract` を付け、中身（`{ }`）を書かずに、**セミコロン `;` で終わらせます**。

```java
abstract class Animal {
    String name;

    void eat() {                  // ふつうのメソッド（中身あり）
        IO.println(name + " はごはんを食べる");
    }

    abstract void cry();          // 抽象メソッド（中身なし、; で終わる）
}
```

`abstract void cry();` が抽象メソッドです。
`{ }` がなく、「`cry()` というメソッドが、ある」という約束だけを書いています。中身は、子クラスに任せます。

> **抽象メソッドがあるクラスは、抽象クラスにする**
>
> 抽象メソッド（中身がないメソッド）を1つでも持つクラスは、必ず `abstract` を付けた抽象クラスにしなければなりません。
> 中身のないメソッドが残っている＝「未完成」なので、実体を作れない（`new` できない）のは当然、というわけです。

---

## 子クラスは、実装を強制される

抽象メソッドを持つ抽象クラスを継承した子クラスは、**その抽象メソッドを必ずオーバーライド（実装）しなければなりません**。
実装しないと、コンパイルエラーになります。

```java
abstract class Animal {
    String name;
    abstract void cry();
}

class Dog extends Animal {
    // cry() を実装していない！
}
```

```text
エラー: Dogはabstractでなく、Animal内のabstractメソッドcry()をオーバーライドしません
```

「`Dog` は抽象クラスではないのに、`Animal` の抽象メソッド `cry()` をオーバーライドしていない」と教えてくれます。
`Dog` を具体的なクラス（`new` できるクラス）にするなら、約束された `cry()` を、きちんと実装する義務があるのです。

正しくは、こうします。

```java
abstract class Animal {
    String name;
    void eat() { IO.println(name + " はごはんを食べる"); }
    abstract void cry();          // 約束だけ定める
}

class Dog extends Animal {
    @Override
    void cry() { IO.println(name + ": ワン"); }   // 約束を果たす
}

class Cat extends Animal {
    @Override
    void cry() { IO.println(name + ": ニャー"); }  // 約束を果たす
}

public class Main {
    public static void main(String[] args) {
        Animal[] animals = { new Dog(), new Cat() };
        animals[0].name = "ポチ";
        animals[1].name = "タマ";
        for (Animal a : animals) {
            a.cry();
        }
    }
}
```

```text
$ java Main.java
ポチ: ワン
タマ: ニャー
```

`Dog` も `Cat` も、`cry()` を実装したので、`new` できます。
そして、`Animal` 型でまとめて `cry()` を呼べば、それぞれの鳴き声が出る ―― 第15章のポリモーフィズムが、より自然な形になりました。

## 抽象メソッドの何がうれしいのか

抽象メソッドには、2つの大きな利点があります。

### 1. 実装し忘れを防げる

「すべての動物は鳴くべき」というルールを、`abstract void cry();` という形で**強制**できます。
もし `Dog` で `cry()` を書き忘れたら、コンパイルエラーで教えてくれます。
「うっかり実装し忘れ」を、動かす前に防げるのです。

### 2. 「共通の枠組み」と「個別の中身」を分けられる

`Animal` は、

- **共通の部分**（`name`・`eat()`）は、具体的に書く
- **動物ごとに違う部分**（`cry()`）は、抽象メソッドにして、子に任せる

と、役割を整理できます。
「共通の土台は親が用意し、違うところだけ子が埋める」という、きれいな分担ができるのです。
これは、第15章で見た「変更に強い」設計を、さらに支えてくれます。

---

## まとめ

- **抽象メソッド**は、`abstract` を付け、中身（`{ }`）を書かず `;` で終わるメソッド（約束だけ定める）
- 抽象メソッドを持つクラスは、必ず**抽象クラス**にする（`new` できない）
- 抽象クラスを継承した子クラスは、抽象メソッドを**必ず実装**しなければならない（しないとエラー）
- 利点: (1) 実装し忘れを防げる、(2) 「共通の枠組み」と「個別の中身」を分けられる
