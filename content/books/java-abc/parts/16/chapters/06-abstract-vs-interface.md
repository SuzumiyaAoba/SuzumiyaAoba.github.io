## 抽象クラスとインターフェースの使い分け

抽象クラスとインターフェースは、どちらも「実装を子に任せる」「まとめて扱える」という、似た性質を持っています。
そのため、「結局、どちらを使えばいいの？」と迷いがちです。
この節では、両者の違いと、使い分けの考え方を整理します。

---

## 違いを表で整理する

まず、2つの違いを表にまとめます。

| | 抽象クラス（`abstract class`） | インターフェース（`interface`） |
|---|---|---|
| 表すもの | **何であるか**（is-a） | **何ができるか**（can-do） |
| 状態（フィールド） | 持てる | 基本は持てない（定数は持てる） |
| 実装ありのメソッド | 持てる | デフォルトメソッドで持てる |
| コンストラクタ | 持てる | 持てない |
| 継承・実装の数 | 1つだけ（`extends`） | **複数**できる（`implements`） |
| キーワード | `extends` で継承 | `implements` で実装 |

最も大きな違いは、次の2点です。

1. **状態（フィールド）を持てるか** … 抽象クラスは持てる、インターフェースは（基本）持てない
2. **複数を組み合わせられるか** … インターフェースは複数実装できる、抽象クラスは1つしか継承できない

---

## 使い分けの考え方

この違いをもとに、次のように使い分けます。

### 抽象クラスを使う場面

- **共通のフィールド（状態）や、共通の実装を、子に持たせたい**とき
- 「**〜は、…の一種である**」（is-a）という、強い親子関係があるとき

`Animal` は、`name` というフィールドや、`eat()` という共通の実装を、すべての動物に持たせたいので、抽象クラスが向いています。
「犬は動物の一種」「猫は動物の一種」という、はっきりした is-a の関係もあります。

### インターフェースを使う場面

- 「**〜できる**」という、**能力・役割**を表したいとき
- **継承関係にないクラスにも、共通の約束を持たせたい**とき
- **複数の能力を組み合わせたい**とき

`Drawable`（描画できる）や `Savable`（保存できる）は、特定の親子関係に縛られず、いろいろなクラスに持たせたい「能力」です。
「文書も図形も描画できる」のように、種類のまったく違うものに共通の約束を持たせるには、インターフェースが向いています。

---

## 迷ったら、まずインターフェース

現代の Java では、「**迷ったら、まずインターフェースを検討する**」のが、よく知られた指針です。
理由は、インターフェースのほうが**柔軟**だからです。

- インターフェースは複数実装できるので、後から能力を足しやすい
- 抽象クラスを継承してしまうと、その1つの「継承枠」を使い切ってしまう（ほかの親を継承できなくなる）

そのうえで、「**どうしても共通の状態（フィールド）や実装を、まとめて持たせたい**」場合に、抽象クラスを使う、と考えるとよいでしょう。

> **両方を組み合わせることもできる**
>
> 抽象クラスとインターフェースは、対立するものではなく、**組み合わせて**使えます。
> たとえば、「`Animal`（抽象クラス）を継承しつつ、`Swimmer`（泳げる、というインターフェース）も実装する `Dog`」のように書けます。
>
> ```java
> class Dog extends Animal implements Swimmer { ... }
> ```
>
> `extends`（1つ）と `implements`（複数）は、同時に書けます。
> 「`Dog` は動物の一種であり（is-a）、かつ泳ぐこともできる（can-do）」を、自然に表現できるのです。

---

## まとめ

- 抽象クラスは「**何であるか**（is-a）」、インターフェースは「**何ができるか**（can-do）」を表す
- 抽象クラスは**状態（フィールド）や実装を持てる**が、**1つしか継承できない**
- インターフェースは**複数実装できる**が、基本的に状態を持てない
- 指針: **迷ったらまずインターフェース**。共通の状態・実装が必要なら抽象クラス
- `extends`（1つ）と `implements`（複数）は、組み合わせて使える
