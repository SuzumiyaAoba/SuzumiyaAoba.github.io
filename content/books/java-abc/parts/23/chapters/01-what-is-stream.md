## ストリームとは ― データの流れ作業

**ストリーム**（Stream）は、「データを、流れ作業のように処理する」しくみです[^java-stream-api]。
この節では、その考え方と、処理の3つの工程をつかみます。

---

## 流れ作業のイメージ

ストリームは、**ベルトコンベアの流れ作業**にたとえると分かりやすいでしょう。

1. ベルトコンベアに、材料（データ）を乗せる
2. 途中の工程で、加工する（不要なものを取り除く、形を変える、並べ替える…）
3. 最後に、できあがったものを受け取る（箱に詰める、数える…）

ストリームも、まったく同じ流れで処理します。

```text
コレクション → [作る] → [加工 → 加工 → …] → [結果を得る]
              ↑              ↑                  ↑
           ストリーム化    中間操作            終端操作
```

データの集まり（コレクションなど）をストリームに乗せ、いくつかの工程で加工し、最後に結果を取り出す ―― この一連の流れが、ストリーム処理です。

---

## 3つの工程

ストリームの処理は、つねに次の**3つの工程**でできています。

| 工程 | 役割 | 例 |
|---|---|---|
| **① 作る**（生成） | データをストリームに乗せる | `list.stream()` |
| **② 中間操作** | 加工する（何回でも） | `filter`・`map`・`sorted` |
| **③ 終端操作** | 結果を得る（最後に1回） | `toList`・`count`・`sum`・`forEach` |

- **① 作る** … コレクションなどから、ストリームを作ります（`.stream()`）
- **② 中間操作** … 絞り込んだり、変換したり。**何個でもつなげられます**。結果はまたストリーム
- **③ 終端操作** … 最後に1回だけ呼び、**最終結果**（リストや数値など）を取り出します

「**作って、加工して、受け取る**」 ―― この3工程を、いつも意識してください。

---

## 実例で、3工程を見る

章の冒頭（はじめに）で見た例を、3工程に分けて見てみましょう。

```java
List<Integer> scores = List.of(85, 40, 70, 55, 90);

List<Integer> passed = scores.stream()       // ① 作る
    .filter(s -> s >= 60)                     // ② 中間操作（絞り込む）
    .sorted(Comparator.reverseOrder())        // ② 中間操作（並べ替える）
    .toList();                                // ③ 終端操作（リストにする）

IO.println(passed);
```

```text
[90, 85, 70]
```

- `scores.stream()` で、リストをストリームに乗せる（**① 作る**）
- `filter` で60点以上に絞り、`sorted` で高い順に並べる（**② 中間操作**を2つ）
- `toList()` で、結果をリストとして受け取る（**③ 終端操作**）

`.メソッド()` を上から下へつなげていくこの書き方を、**メソッドチェーン**（method chain、メソッドの連鎖）と呼びます。
処理の流れが、そのまま縦に並ぶので、とても読みやすくなります。

---

## 中間操作と終端操作の、大切な違い

ストリームを理解するうえで、**中間操作と終端操作の違い**は、とても重要です。

- **中間操作**（`filter`・`map` など）は、ストリームを返す。**まだ実行されない**（次の操作の準備をするだけ）
- **終端操作**（`toList`・`count` など）が呼ばれて、**初めてすべてが実行される**

つまり、`filter` や `map` を並べただけでは、まだ何も起きていません。
最後の `toList()` のような**終端操作が呼ばれた瞬間**に、データが流れ始め、すべての中間操作がまとめて実行されるのです。

この性質は「**遅延評価**（lazy evaluation）」と呼ばれます。
いまは深く考えなくて大丈夫ですが、「**終端操作がないと、ストリームは動かない**」という点だけ、覚えておいてください。これは、よくあるつまずきの元になります（第7節）。

---

## 元のコレクションは、変わらない

もう1つ大切な性質があります。
ストリームの処理は、**元のコレクションを変更しません**。

```java
List<Integer> scores = List.of(85, 40, 70, 55, 90);
List<Integer> passed = scores.stream().filter(s -> s >= 60).toList();

IO.println(passed);   // 処理結果（新しいリスト）
IO.println(scores);   // 元のリストは、そのまま
```

```text
[85, 70, 90]
```

```text
[85, 40, 70, 55, 90]
```

`filter` で絞り込んでも、元の `scores` は `[85, 40, 70, 55, 90]` のまま、変わっていません。
ストリームは、元のデータには手を触れず、**処理結果を新しく作って返す**のです。
これは、第13章で学んだ「不変オブジェクト」の考え方とも通じる、安全な性質です。

---

## まとめ

- **ストリーム**は、データを**流れ作業**のように処理するしくみ（ベルトコンベアのイメージ）
- 処理は、**① 作る → ② 中間操作 → ③ 終端操作** の3工程でできている
- `.メソッド()` をつなげる書き方を**メソッドチェーン**と呼ぶ
- **中間操作はまだ実行されず、終端操作が呼ばれて初めて全体が動く**（終端操作がないと何も起きない）
- ストリームは**元のコレクションを変更せず**、結果を新しく返す

次の節では、3工程の最初 ―― ストリームを**作る**方法を学びます。

[^java-stream-api]: Java SE 25 API Specification, `java.util.stream.Stream`, [https://docs.oracle.com/en/java/javase/25/docs/api/java.base/java/util/stream/Stream.html](<https://docs.oracle.com/en/java/javase/25/docs/api/java.base/java/util/stream/Stream.html>)。Stream API は Java SE 8（2014年）で JSR 335 と共に導入された。"A sequence of elements supporting sequential and parallel aggregate operations" として定義され、遅延評価（lazy evaluation）と終端操作のトリガが規定されている。
