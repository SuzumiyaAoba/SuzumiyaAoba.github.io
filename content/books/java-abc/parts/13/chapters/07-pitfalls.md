## よくあるつまずき

コンストラクタで、初心者がつまずきやすいポイントを、まとめて確認します。

---

## 1. コンストラクタに戻り値の型を書いてしまう（最大の罠）

コンストラクタは、**戻り値の型を書かない**のがルールでした。
ところが、うっかり `void` を付けてしまうことがあります。

```java
class Person {
    String name;
    int age;

    void Person(String name, int age) {   // ✕ void を付けてしまった
        this.name = name;
        this.age = age;
    }
}
```

これは、エラーにはなりません。**だからこそ、やっかい**です。
`void` を付けると、これは**コンストラクタではなく、「`Person` という名前のふつうのメソッド」**として扱われてしまうのです。

すると、コンストラクタは1つも書いていないことになるので、Java は**デフォルトコンストラクタ**（引数なし・空）を用意します。
その結果、`new Person("佐藤", 25)` はエラーになり、`new Person()` で作っても、名前や年齢が設定されません。

```java
Person p = new Person("佐藤", 25);   // エラー（引数つきコンストラクタは存在しない）
```

「コンストラクタを書いたはずなのに、引数つきの `new` が使えない」「値が設定されない」と感じたら、**戻り値の型（`void` など）を付けていないか**を、まず疑いましょう。
コンストラクタには、`void` も含めて、戻り値の型は**書きません**。

---

## 2. コンストラクタの名前が、クラス名と違う

コンストラクタの名前は、**クラス名とまったく同じ**にする必要があります（大文字・小文字も含めて）。

```java
class Person {
    Person(String name) { ... }   // ◯ クラス名と同じ
    person(String name) { ... }   // ✕ これはメソッド扱い（p が小文字）
}
```

名前が1文字でも違うと、コンストラクタではなく、ふつうのメソッドとみなされます（罠1と同じ結果になります）。
クラス名とコンストラクタ名は、コピーするくらいの気持ちで、正確にそろえましょう。

---

## 3. コンストラクタを書いたら、引数なし new が使えなくなった

第5節で学んだとおり、**自分でコンストラクタを書くと、デフォルトコンストラクタは自動では用意されません**。

```java
class Person {
    Person(String name, int age) { ... }   // 引数つきだけを書いた
}

Person p = new Person();   // エラー: 引数なしコンストラクタはない
```

引数なしの `new Person()` も使いたいなら、**引数なしのコンストラクタも自分で書きます**。
「コンストラクタを書いた瞬間、引数なしは消える」と覚えておきましょう。

---

## 4. final フィールドを初期化し忘れる

`final` フィールドは、**必ずコンストラクタ（または宣言時）で初期化**しなければなりません。
初期化しないと、コンパイルエラーになります。

```java
class Point {
    final int x;
    final int y;

    Point(int x) {
        this.x = x;
        // y を初期化していない！
    }
}
```

```text line-numbers=false
エラー: 変数yは初期化されていない可能性があります
```

`final` フィールドは「あとから入れる」ことができないので、コンストラクタを抜けるまでに、必ず値を入れる必要があります。
複数のコンストラクタがある場合は、**そのすべてで** `final` フィールドを初期化しなければなりません。

---

## 5. final フィールドを、可変だと思い込む

不変オブジェクトのつもりで `final` を付けても、**`final` は「その変数の再代入を禁止する」だけ**です。
基本型（`int` など）なら値そのものが固定されますが、フィールドが**別のオブジェクトを指している**場合、その**指している先の中身**までは固定されません。

```java
class Team {
    final int[] scores;   // 配列を指す参照は変えられないが…
    Team(int[] scores) { this.scores = scores; }
}
```

この `scores` は、別の配列に**つなぎ替える**ことはできません（`final` なので）。
しかし、`team.scores[0] = 999;` のように、**配列の中身を書き換える**ことはできてしまいます。
本当に「中身まで変わらない」不変オブジェクトにするには、こうした参照先の扱いにも注意が必要です（初心者のうちは、`int` や `String` など、それ自体が不変な値を `final` にする、と覚えておけば十分です）。

---

## まとめ

- コンストラクタに**戻り値の型（`void` など）を書かない**（書くとメソッド扱いになる最大の罠）
- コンストラクタ名は、**クラス名と完全に一致**させる
- 自分でコンストラクタを書くと、**引数なしの `new` は使えなくなる**（必要なら自分で書く）
- **`final` フィールドは、すべてのコンストラクタで初期化**する
- `final` は再代入を禁止するだけ。配列などの**中身**までは固定しない

次は、この章で学んだ言葉を、用語集としてまとめます。
