## インストールを確認する ― java --version と PATH

前の節で、JDK をインストールしました。

ここで大切なのは、**「本当に正しくインストールできたか」を、自分の目で確かめる**ことです。

インストーラーで「完了」と表示されても、それだけでは安心できません。
コンピュータがちゃんと Java を見つけられる状態になっているか、必ず確認しておきましょう。

この確認には、**コマンドライン**を使います。
聞き慣れない言葉かもしれませんが、心配いりません。
まずは「文字でコンピュータに命令する画面」だと思ってください。

---

## コマンドラインとは

これまで、コンピュータは主に**マウスでアイコンをクリック**して操作してきたと思います。
このような操作方法を **GUI**（Graphical User Interface、グラフィカル・ユーザー・インターフェース）と呼びます。

一方、**文字でコンピュータに命令する**操作方法もあります。
これを **CUI**（Character User Interface）や **CLI**（Command Line Interface、コマンドライン・インターフェース）と呼びます。

そして、その文字命令を入力するための画面・ソフトのことを、**ターミナル**（Terminal）や **コマンドライン** と呼びます。

最初はとっつきにくく感じるかもしれませんが、プログラミングでは、文字で命令するほうが速くて確実な場面がたくさんあります。
この章でも、これから何度か登場します。少しずつ慣れていきましょう。

### ターミナルの開き方

使っている OS に合わせて、ターミナルを開いてください。

| OS | 開き方 |
|---|---|
| **Windows** | スタートメニューで「ターミナル」または「PowerShell」を検索して開く |
| **macOS** | 「アプリケーション」→「ユーティリティ」→「ターミナル」を開く（または Spotlight 検索で「ターミナル」と入力） |

ターミナルを開くと、黒や白の画面に、文字を入力できる状態（カーソルが点滅している状態）が表示されます。
ここに命令を打ち込んでいきます。

> **補足: インストール前から開いていた画面は閉じておく**
>
> JDK のインストール前から開いていたターミナルでは、まだ設定が反映されていないことがあります。
> 確認の前に、ターミナルをいったん閉じて、**新しく開き直して**おきましょう。
> この「開き直す」というひと手間が、トラブルを防ぐ意外と大事なコツです。

---

## java --version で確認する

それでは、Java が正しく入ったかを確認しましょう。

ターミナルに、次の文字を入力して、Enter キーを押します。

```text
java --version
```

`java` と `--version` の間には、半角スペースが1つ入ります。
また、`--version` の先頭は、ハイフン（`-`）が**2つ**です。1つだけにならないよう気をつけてください。
すべて半角の英数字で入力してください。

正しくインストールできていれば、次のような表示が出ます。

```text
openjdk 25 2025-09-16
OpenJDK Runtime Environment Temurin-25+36 (build 25+36)
OpenJDK 64-Bit Server VM Temurin-25+36 (build 25+36, mixed mode, sharing)
```

ここで注目してほしいのは、**1行目**です。

```text
openjdk 25 ...
```

このように、バージョン番号として **`25`** が表示されていれば、Java 25 が正しくインストールできています。
おめでとうございます！

> **補足: 細かい文字は環境によって違う**
>
> 上の表示には、環境によって異なる部分がたくさんあります。
> - 更新版を入れた場合、`25` ではなく **`25.0.1`** や **`25.0.3`** のように表示されます（どれも Java 25 の仲間です）
> - `Temurin-25+36` の `+36`（ビルド番号）や、`2025-09-16` という日付も、入れた時期によって変わります
> - Intel の Mac や Windows では、`64-Bit Server VM` の部分の表記が違うこともあります
>
> 大切なのは、**1行目に `25` と表示されているか**です。
> そのあとに続く細かい数字や日付の違いは、まったく気にしなくて大丈夫です。

> **補足: `-version`（ハイフン1つ）との違い**
>
> インターネットで調べると、ハイフンが**1つ**の `java -version` という書き方や、次のように**引用符付き**で表示された例を見かけることがあります。
>
> ```text
> openjdk version "25" 2025-09-16
> ```
>
> これは古い書き方ではなく、`java -version`（ハイフン1つ）の出力です。
> ハイフン1つの `-version` は、引用符付きで、しかも画面の別の出口（標準エラー出力）に表示される、という違いがあるだけで、確認できる内容は同じです[^java-version-flag]。
> 本書では、より新しく分かりやすい `--version`（ハイフン2つ）で統一しています。
> どちらを使っても、Java 25 が入っていれば `25` という番号を確認できます。

### うまくいかないとき

もし、次のようなメッセージが出た場合は、まだ正しく認識されていません。

```text
（Windows の例）
'java' は、内部コマンドまたは外部コマンド、操作可能なプログラムまたはバッチ ファイルとして認識されていません。
```

```text
（macOS の例）
zsh: command not found: java
```

これは、「`java` という命令が見つかりません」という意味です。
あわてなくて大丈夫です。よくあることです。

このようなときの対処は、この章の最後の第8節「うまくいかないとき」で詳しく説明します。
ここでは、後で出てくる **PATH** という設定が関係していることが多い、とだけ覚えておいてください。

## javac でも確認する

`java` は、プログラムを**動かす**ための命令でした。

JDK には、もう一つ大切な命令があります。
プログラムを**作る**（コンパイルする）ための **`javac`**（ジャバ・シー）です。

これも入っているか、確認しておきましょう。
ターミナルに、次のように入力して Enter を押します。

```text
javac --version
```

次のように表示されれば成功です。

```text
javac 25
```

（更新版を入れている場合は、`javac 25.0.3` のように表示されることもあります。これも問題ありません。）

ここで、`java`（実行）だけでなく `javac`（コンパイル）も使えることを確認できました。
これで、プログラムを「作る」道具と「動かす」道具の両方がそろっていることがわかります。

> **`java` は動いたのに、`javac` が動かない？**
>
> もし `java --version` は成功したのに、`javac --version` で「見つからない」と出る場合、JDK ではなく **JRE**（動かすだけの一式）が入っている可能性があります。
> その場合は、第3節に戻って、**JDK** をインストールし直してください。
> 本書のとおり Temurin の JDK を入れていれば、両方とも使えるはずです。

## PATH とは ― なぜ、どこからでも java と打てるのか

ここで、先ほどから登場している **PATH**（パス）について説明します。

少し技術的な話ですが、トラブルの原因を理解するために、ざっくりとでも知っておくと、この先とても役立ちます。

### コンピュータは、命令の「置き場所」を探している

`java` と打つと、コンピュータは「`java` という命令の本体（プログラム）」を探して、それを実行します。

しかし、その本体は、コンピュータの中のどこか特定の場所（フォルダ）に置かれています。
何も設定していないと、コンピュータは「`java` って何？ どこにあるの？」と、見つけられません。

そこで活躍するのが **PATH** です。

**PATH** とは、コンピュータが命令を探しに行く**フォルダの一覧**のことです。

```text
コンピュータ：「java を実行して」と言われた
   ↓
PATH に登録されたフォルダを、順番に探していく
   ↓
java の本体が見つかった → 実行する
見つからなかった       → 「見つかりません」と言う
```

つまり、JDK の命令（`java` や `javac`）が置かれているフォルダを **PATH に登録（追加）** しておけば、どこからでも `java` と打つだけで実行できるようになります。

![PATH のしくみ。java と入力すると、PATH に登録されたフォルダを上から順に探し、java の本体があるフォルダ（JDK の bin）が見つかると java を実行する。](./images/path-mechanism.svg)

第3節で、Windows のインストーラーの「**Add to PATH**」という項目を有効にしたのは、まさにこの登録を自動で行うためでした。

### PATH に登録されていないと

逆に、PATH に登録されていないと、コンピュータは命令を見つけられず、

```text
command not found: java
```

のようなエラーになります。

先ほど「うまくいかないとき」で見たエラーは、多くの場合、この **PATH の登録ができていない**ことが原因です。
直し方は、第8節でくわしく説明します。

## JAVA_HOME とは

もう一つ、**JAVA_HOME**（ジャバ・ホーム）という言葉も、これからよく見かけることになります。

**JAVA_HOME** とは、「いま使っている JDK が、どのフォルダに入っているか」をコンピュータに教えておくための設定です。

```text
JAVA_HOME ＝ 「JDK 本体は、この場所にあります」という住所
```

Java そのものを動かすだけなら、JAVA_HOME は必ずしも必要ではありません。
しかし、後の章で学ぶ**ビルドツール**（Maven や Gradle など、開発を助ける道具）の多くが、この JAVA_HOME を見て JDK の場所を判断します[^java-home-tools]。

そのため、最初に設定しておくと、後でつまずきにくくなります。
第3節で「**Set JAVA_HOME variable**」を有効にしたのは、このためでした。

いまの段階では、次の2つの違いを押さえておけば十分です。

- **PATH** … `java` などの命令を「どこからでも使えるようにする」設定
- **JAVA_HOME** … JDK が「どこにあるか」を教えておく設定

設定がうまくいっていない場合の直し方は、第8節で説明します。

## まとめ

この節では、JDK が正しくインストールできたかを確認しました。

- インストール後は、必ず**自分の目で確認**することが大切
- 確認には、文字で命令する画面である**ターミナル（コマンドライン）**を使う
- 設定を反映させるため、ターミナルは**新しく開き直して**から確認する
- `java --version` を実行し、1行目に **`25`** と表示されていれば、Java 25 が正しく入っている
- `javac --version` も実行し、`javac 25` と出れば、コンパイラも使える
- ビルド番号・日付・更新版の番号（`25.0.x`）などの細かい違いは、気にしなくてよい
- **PATH** は、コンピュータが命令を探すフォルダの一覧。これに JDK を登録すると、どこからでも `java` が使える
- **JAVA_HOME** は、JDK の場所を教える設定。後のビルドツールで役立つ
- 「見つかりません」と出たときは、多くの場合 PATH の設定が原因（直し方は第8節）

次の節では、プログラムを書きやすくするための **IDE** を準備します。

[^java-version-flag]: Oracle, "java Tool Reference," <https://docs.oracle.com/en/java/javase/25/docs/specs/man/java.html>。`java --version` は GNU 慣習の長オプションで標準出力に出力する。`java -version` は従来からのオプションで、後方互換性のため `"25"` のような引用符付き形式を標準エラー出力に出す。

[^java-home-tools]: Apache Maven, "Installing Apache Maven," <https://maven.apache.org/install.html> および Gradle, "Compatibility Matrix and JVM," <https://docs.gradle.org/current/userguide/compatibility.html>。Maven は `JAVA_HOME` を参照して使用する JDK を決定し、Gradle も `org.gradle.java.home` プロパティの代替として `JAVA_HOME` を利用する。
