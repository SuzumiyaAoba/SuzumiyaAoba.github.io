## うまくいかないとき ― よくあるトラブルと対処

環境構築は、初心者が最もつまずきやすいところです。

手順どおりに進めても、なぜかうまくいかない ―― そんなことは、決して珍しくありません。
プログラミングに慣れた人でも、新しい環境では同じようなことでつまずきます。

大切なのは、**あわてず、原因を一つずつ確認していく**ことです。
このセクションでは、環境構築でよくあるトラブルと、その対処法をまとめます。

うまくいかないときは、目次から自分の症状を探して、該当する部分を読んでください。

- [`java` コマンドが見つからない](#not-found)
- [思っていたのと違うバージョンが表示される](#wrong-version)
- [`javac` だけが見つからない](#javac-not-found)
- [コンパイルでエラーが出る](#compile-error)
- [`java Main` で「クラスが見つからない」と出る](#class-not-found)
- [IDE で JDK が選べない・実行できない](#ide-jdk)
- [日本語が文字化けする](#mojibake)

---

## トラブルへの基本的な向き合い方

具体的な対処に入る前に、すべてに共通する心構えをお伝えします。

第1章でも学んだように、**エラーは失敗ではなく、合図**です。
コンピュータが「ここがうまくいっていません」と教えてくれているのです。

トラブルが起きたら、次の4つを意識してください。

1. **エラーメッセージをよく読む** ― 英語でも、よく見ると原因のヒントが書かれている
2. **直前に何をしたか思い出す** ― 何かを変えた直後にエラーが出ることが多い
3. **一つずつ確認する** ― あれこれ同時に変えず、1か所ずつ確かめる
4. **画面を開き直す** ― 設定を変えたら、ターミナルや IDE を開き直すと反映される

特に4つ目は見落としがちです。
**設定を変えたあとは、ターミナルや IDE をいったん閉じて、開き直す**。これだけで直ることが、本当によくあります。
「うまくいかないな」と思ったら、まず試してみてください。

## `java` コマンドが見つからない {#not-found}

### 症状

`java --version` を実行すると、次のようなメッセージが出る。

```text
（Windows）
'java' は、内部コマンドまたは外部コマンド、操作可能なプログラムまたはバッチ ファイルとして認識されていません。
```

```text
（macOS）
zsh: command not found: java
```

### 原因

これは、第4節で説明した **PATH** が正しく設定されていないことが、ほとんどの原因です。
コンピュータが、`java` という命令の置き場所を見つけられていない状態です。

### 対処（まず試すこと）

いちばん簡単な確認から行います。

- **ターミナルを開き直す** ― インストール直後は、設定がまだ反映されていないことがあります。ターミナルを閉じて、新しく開き直してから、もう一度 `java --version` を試してください。

これで直らない場合は、OS ごとに PATH を設定します。

### 対処（Windows）

1. JDK のインストール先を確認します。多くの場合、次の場所にあります。
   ```text
   C:\Program Files\Eclipse Adoptium\jdk-25...\bin
   ```
   （`jdk-25...` の部分は、バージョンによって少し変わります。）
2. スタートメニューで「環境変数」と検索し、**「システム環境変数の編集」** を開きます。
3. 「環境変数」ボタンを押します。
4. 「ユーザー環境変数」または「システム環境変数」の中から **`Path`** を選び、「編集」を押します。
5. 「新規」を押して、手順1で確認した `bin` までのパスを追加します。
6. すべて「OK」で閉じます。
7. **ターミナルを開き直して**、`java --version` を試します。

> もう一度インストーラーを実行し、「**Add to PATH**」を有効にし直す、という方法もあります。手作業の設定が不安な場合は、こちらのほうが簡単で確実です。

### 対処（macOS）

`.pkg` インストーラーで入れた場合、通常は `java` コマンドがそのまま使えるようになっています。
それでも見つからない場合は、次を確認します。

1. JDK が入っているか確認します。
   ```text
   /usr/libexec/java_home -V
   ```
   ここに Temurin の 25 が表示されれば、JDK 自体は入っています。
2. 使用する JDK を指すように、設定ファイルに次の1行を追加します。
   macOS の標準のシェルである **zsh** の場合、`~/.zshrc` というファイルに書きます。
   ```text
   export JAVA_HOME=$(/usr/libexec/java_home -v 25)
   ```
3. 設定を反映するため、**ターミナルを開き直す**か、次を実行します。
   ```text
   source ~/.zshrc
   ```
4. もう一度 `java --version` を試します。

## 思っていたのと違うバージョンが表示される {#wrong-version}

### 症状

`java --version` を実行すると、`25` ではなく、`17` や `21` など、別のバージョンが表示される。

### 原因

コンピュータに、**以前から別の Java が入っていた**ことが原因です。
複数の Java が入っていると、コンピュータは PATH の中で**先に見つけたほう**を使います。

### 対処

- **Windows**: 「`java` コマンドが見つからない」の対処と同じ手順で `Path` を開き、新しく入れた Java 25 の `bin` を、古い Java の場所より**上（先）**に並べ替えます。
- **macOS**: 上で説明したように、`~/.zshrc` で `JAVA_HOME` を 25 に設定します。
  ```text
  export JAVA_HOME=$(/usr/libexec/java_home -v 25)
  ```

設定を変えたら、必ずターミナルを開き直して、もう一度 `java --version` で確認しましょう。

> **学習だけなら、急いで消さなくてよい**
>
> 古い Java が入っていても、それ自体が悪さをするわけではありません。
> 本書の学習中に Java 25 がきちんと使えていれば、古いほうを無理に削除する必要はありません。

## `javac` だけが見つからない {#javac-not-found}

### 症状

`java --version` は成功するのに、`javac --version` だけ「見つからない」と出る。

### 原因

プログラムを**動かす**ための一式（JRE 相当）は入っているが、**作る**ための **JDK** が正しく入っていない可能性があります。

### 対処

第3節に戻り、**JDK**（Temurin の JDK）をインストールし直してください。
本書のとおり Temurin の JDK を入れていれば、`java` と `javac` の両方が使えるはずです。

あわせて、PATH の設定が JDK の `bin` を指しているかも確認しましょう。

## コンパイルでエラーが出る {#compile-error}

### 症状

`javac Main.java` を実行すると、エラーメッセージが出て、`Main.class` ができない。

### よくある原因と対処

コンパイルエラーは、ほとんどが**入力の間違い**です。
次の点を、見本のコードと1文字ずつ見比べてみてください。

| よくある間違い | 対処 |
|---|---|
| **全角の記号が混ざっている** | `（` `）` `；` `”` などの全角記号を、半角の `(` `)` `;` `"` に直す |
| **`;`（セミコロン）の付け忘れ** | 命令の終わりに、半角の `;` があるか確認する |
| **`{` `}` の数が合っていない** | 開いた `{` の数だけ、閉じる `}` があるか確認する |
| **`"` の閉じ忘れ** | 文字を囲む `"` が、始まりと終わりでそろっているか確認する |
| **大文字・小文字の間違い** | `System` を `system`、`String` を `string` などと書いていないか確認する |
| **つづりの間違い** | `println` や `public` などのつづりを確認する |

> **特に注意: 全角文字**
>
> 初心者が最もつまずきやすいのが、**全角の記号**です。
> 日本語入力をオンにしたまま記号を打つと、見た目はそっくりでも、コンピュータには別の文字として扱われます。
> 全角のスペースが紛れ込んでいることもあり、これは目で見ても気づきにくい厄介な原因です。
> エラーが出たら、まず「全角の記号やスペースが混ざっていないか」を疑いましょう。

IDE（IntelliJ IDEA）を使っている場合は、間違っている場所に**赤い波線**が表示されます。
その部分にマウスのカーソルを合わせると、何が問題かのヒントが表示されるので、ぜひ活用してください。
これも、IDE を使う大きな利点の一つです。

## `java Main` で「クラスが見つからない」と出る {#class-not-found}

### 症状

`java Main` を実行すると、次のようなメッセージが出る。

```text
エラー: メイン・クラスMainを検出およびロードできませんでした
原因: java.lang.ClassNotFoundException: Main
```

> 英語環境では、次のように表示されます（内容は同じです）。
>
> ```text
> Error: Could not find or load main class Main
> ```

（「Main というクラスが見つからない、または読み込めない」という意味です。）

### よくある原因と対処

| 原因 | 対処 |
|---|---|
| **まだコンパイルしていない** | 先に `javac Main.java` を実行し、`Main.class` を作る |
| **違うフォルダにいる** | `Main.class` があるフォルダに `cd` で移動しているか確認する |
| **拡張子を付けてしまった** | `java Main.class` ではなく **`java Main`** と入力する（拡張子なし） |
| **クラス名のつづり違い** | `public class Main` のクラス名と、`java` の後ろの名前が一致しているか確認する |

`ls`（macOS）や `dir`（Windows）で、いまいるフォルダに `Main.class` があることを確認してから `java Main` を実行する。
この順番を意識すると、つまずきにくくなります。

## IDE で JDK が選べない・実行できない {#ide-jdk}

### 症状

IntelliJ IDEA で、プロジェクト作成時に JDK が一覧に出てこない。
あるいは、`Project SDK is not defined`（プロジェクトの JDK が設定されていない）と表示されて実行できない。

### 原因

IntelliJ に、**使う JDK の場所が伝わっていない**ことが原因です。
「IDE は JDK の代わりにはならない」という、第1節・第5節の話を思い出してください。

### 対処

次のどちらかの方法で、JDK を設定します。

**方法1: すでに入れた JDK を指定する**

1. プロジェクト作成画面、または「File」→「Project Structure」→「Project」を開きます。
2. SDK（JDK）を選ぶ欄で、「Add SDK」→「JDK...」を選びます。
3. 第3節で入れた Temurin の場所を指定します。
   - Windows の例: `C:\Program Files\Eclipse Adoptium\jdk-25...`
   - macOS の例: `/Library/Java/JavaVirtualMachines/temurin-25.jdk/Contents/Home`

**方法2: IntelliJ にダウンロードしてもらう**

1. JDK を選ぶ欄で、「**Download JDK**」を選びます。
2. Vendor（提供元）から **Eclipse Temurin（Adoptium）** を、Version から **25** を選びます。
3. 「Download」を押すと、IntelliJ が JDK を入れて、自動で設定してくれます。

どちらの方法でも、JDK が設定されれば、▶ ボタンで実行できるようになります。

## 日本語が文字化けする {#mojibake}

### 症状

`System.out.println("こんにちは")` のように日本語を表示したとき、画面に出る文字が、次のように意味のわからない記号に化けてしまう。

```text
縺薙ｓ縺ｫ縺｡縺ｯ
```

このように、文字が正しく表示されないことを **文字化け**（もじばけ）と呼びます。

### 原因

文字化けは、**文字コード**（文字をコンピュータ内部で表す数値のルール）の食い違いが原因で起こります。
書いたときと読んだときで、別のルールを使ってしまうと、文字が正しく復元できず、化けてしまうのです。

> **「全角文字」とは別の問題**
>
> 第6節やコンパイルエラーで出てきた「全角の記号」の話（`（` を `(` と間違える）とは、別の問題です。
> あちらは**入力ミス**、こちらは**文字コードの食い違い**です。混同しないようにしましょう。

### 対処

現在の Java（Java 18 以降）は、内部の標準の文字コードに、世界中の文字を扱える **UTF-8** を使っています。
そのため、文字化けは以前より起きにくくなっていますが、それでも次のような場面では起こることがあります。

1. **まず IntelliJ IDEA で実行してみる**
   IntelliJ の実行結果ウィンドウは UTF-8 で表示されるため、日本語はそのまま正しく表示されます。
   コマンドラインで文字化けする場合も、IDE で実行すれば問題が出ないことがほとんどです。

2. **Windows のコマンド画面で文字化けする場合**
   Windows の古いコマンドプロンプトは、表示に使う文字コードが UTF-8 以外のことがあり、日本語が化けることがあります。
   - なるべく、新しい **Windows Terminal**（ターミナル）を使う
   - それでも化ける場合は、`chcp 65001` というコマンドを実行して、表示を UTF-8 に切り替える

3. **ソースファイルの保存に使った文字コードを確認する**
   IntelliJ 以外の簡単なエディタでソースを作った場合、保存時の文字コードが UTF-8 になっているか確認してください。
   別の文字コードで保存すると、日本語のコメントや文字列が化ける原因になります。
   IntelliJ で書いている場合は、初めから UTF-8 で保存されるため、心配いりません。

文字コードは奥の深いテーマですが、初心者のうちは「**日本語が化けたら、まず IntelliJ で実行してみる**」「**ファイルは UTF-8 で保存する**」の2つを覚えておけば十分です。

## それでも解決しないとき

ここまでの対処を試しても解決しない場合は、次のことを試してみてください。

- **エラーメッセージをそのまま検索する** ― 同じ問題に遭遇した人の解決方法が見つかることが、とてもよくあります
- **公式サイトの案内を確認する** ― Temurin（[adoptium.net](https://adoptium.net)）や IntelliJ IDEA の公式ドキュメントには、インストールの詳しい説明があります
- **思い切って入れ直す** ― 設定がこんがらがってしまったときは、いったんアンインストールして最初からやり直すのが、結局はいちばん早いこともあります
- **時間をおく** ― 煮詰まったら、少し休んでから見直すと、原因にすぐ気づくこともあります

環境構築でつまずいても、落ち込む必要はまったくありません。
ここを乗り越えれば、あとはプログラムを書くことに集中できます。

## まとめ

このセクションでは、環境構築でよくあるトラブルと対処法を学びました。

- エラーは「失敗」ではなく「合図」。あわてず、一つずつ確認する
- **設定を変えたら、ターミナルや IDE を開き直す**だけで直ることが多い
- `java` が見つからないときは、**PATH** の設定を確認する
- 違うバージョンが出るときは、古い Java より新しい Java 25 を優先するように設定する
- `javac` だけ動かないときは、JRE ではなく **JDK** が入っているか確認する
- コンパイルエラーの多くは、**全角文字・セミコロン・かっこ・大文字小文字**などの入力ミス
- `java Main` でクラスが見つからないときは、フォルダ・コンパイルの有無・拡張子・名前の一致を確認する
- IDE で実行できないときは、使う **JDK の場所を IntelliJ に設定**する
- 日本語が**文字化け**するときは、文字コードの食い違いが原因。まず IntelliJ で実行し、ファイルは **UTF-8** で保存する
- それでも解決しないときは、エラー検索・公式ドキュメント・入れ直しを試す

これで、環境構築でつまずいたときの備えもできました。
次は、この章で学んだ言葉をまとめた「用語集」です。
