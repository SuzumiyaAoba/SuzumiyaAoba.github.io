## 第6部のまとめ ― GoF 23 パターンを学び終えて

第6部では、GoF（Gang of Four）の 23 のデザインパターンを、1 章ずつすべて学びました。

### 生成パターン（Creational, 5）

| 章 | パターン | ひと言で |
|---|---|---|
| 第51章 | Singleton | インスタンスを 1 つに限定する |
| 第52章 | Factory Method | 生成の責任をサブクラスに委ねる |
| 第53章 | Abstract Factory | 製品の家族をまとめて作る |
| 第54章 | Builder | 多くのパラメータを段階的に組み立てる |
| 第55章 | Prototype | 原型からコピーして新しいインスタンスを得る |

### 構造パターン（Structural, 7）

| 章 | パターン | ひと言で |
|---|---|---|
| 第56章 | Adapter | 形を合わせる |
| 第57章 | Facade | 複雑な内部を 1 つの窓口にまとめる |
| 第58章 | Decorator | 機能を着せ重ねる |
| 第59章 | Proxy | 代理人として間に立つ |
| 第60章 | Composite | 葉と節を同じ顔で扱う |
| 第61章 | Bridge | 抽象と実装を分けて持つ |
| 第62章 | Flyweight | 同じ状態を共有してメモリを節約する |

### 振る舞いパターン（Behavioral, 11）

| 章 | パターン | ひと言で |
|---|---|---|
| 第63章 | Iterator | 順に取り出す |
| 第64章 | Strategy | アルゴリズムを差し替える |
| 第65章 | Template Method | 流れを親に、肉付けを子に |
| 第66章 | Observer | 変化を購読する |
| 第67章 | Command | 操作を名詞にする |
| 第68章 | State | 状態によって振る舞いを変える |
| 第69章 | Chain of Responsibility | 順に責任を渡す |
| 第70章 | Mediator | 多対多を 1 点経由にする |
| 第71章 | Memento | 状態のスナップショットを取る |
| 第72章 | Visitor | 構造に操作を訪問させる |
| 第73章 | Interpreter | 小さな専用言語を解釈する |

### 学びの 3 つの心構え

1. **パターンは「型」ではなく「会話の語彙」**
   ―― 「これは Strategy だね」と**指差せること**が、最初のゴールです。
2. **モダン Java を活かす**
   ―― ラムダ・レコード・シールドクラス・パターンマッチングが、GoF の冗長さを大きく解消してくれます。
3. **使うべきでない場面**を知る
   ―― パターンの安易な当てはめ（パターン病）は、しばしばシンプルさを犠牲にします。**SOLID 原則**（第35章）と合わせて、判断力を磨いてください。

## おわりに

ここまで読み進めてくださって、ありがとうございました。
GoF のパターンは、出版から 30 年以上が経ったいまも、ソフトウェア設計の**共通言語**として現役です。
本書での解説をきっかけに、現場のコードを読むときに「あ、これは○○パターンだ」と**気づける目**を、ぜひ持ち帰ってください。

そして、**パターンを当てはめるかではなく、問題を解いているか**を、常に意識し続けてもらえれば、それが本書のいちばんの願いです。
