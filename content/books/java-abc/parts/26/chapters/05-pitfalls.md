## よくあるつまずき

パターンマッチングで、初心者がつまずきやすいポイントを、まとめて確認します。

---

## 1. case の順番を間違える（広いパターンを先に書く）

`case` は**上から順に**チェックされます。
そのため、「広い（ゆるい）パターン」を先に書くと、あとの「狭い（細かい）パターン」に**たどり着けなくなります**。

```java
String f(Object o) {
    return switch (o) {
        case Object obj -> "なんでも";   // ✕ これが全部を拾ってしまう
        case String s   -> "文字列";     // ここには絶対たどり着けない
        default         -> "その他";
    };
}
```

```text
エラー: 先行するcaseラベルが、このcaseラベルよりも優位です
```

`case Object obj` は「あらゆるオブジェクト」にマッチするので、その下の `case String s` には、絶対にたどり着けません。
Java は、この「**到達できない `case`**」を、コンパイルエラーで教えてくれます。
「**狭いパターンを上、広いパターンを下**」に並べる、と覚えておきましょう（`when` 付きの細かい条件も、上に書きます）。

---

## 2. switch の網羅が足りない

`Object` などを `switch` するとき、すべての場合をカバーしないと、コンパイルエラーになります。

```java
String f(Object o) {
    return switch (o) {
        case Integer i -> "整数";
        case String s  -> "文字列";
        // default も、それ以外の case もない → エラー
    };
}
```

```text
エラー: switch式がすべての可能な入力値をカバーしていません
```

`Integer` と `String` 以外の型（`Double` など）が来たら、どうするのかが決まっていません。
**`default`** を追加して、「それ以外」をカバーしましょう。
（取りうる型が限られている場合 ―― 第27章のシールドクラス ―― は、すべて挙げれば `default` は不要です。）

---

## 3. パターン変数の名前がかぶる

異なる `case` で、同じパターン変数名を使うのは問題ありませんが、1つの `case` の中で、外側の変数と名前がぶつかると、混乱のもとです。

```java
case Integer i -> "整数: " + i;
case String i  -> "文字列: " + i;   // 別の case なので、同じ i でもOK（が、分かりにくいことも）
```

別々の `case` なら、同じ名前を使っても動きます。
ですが、読みやすさのために、`Integer i`・`String s` のように、**型に合った分かりやすい名前**を付けるのがよいでしょう。

---

## 4. パターンマッチングを、無理に使う

パターンマッチングは強力ですが、「**ただ1つの型かどうか**」を見るだけなら、ふつうの `instanceof` で十分です。

```java
// 1つの型を見るだけなら、switch にする必要はない
if (obj instanceof String s) {
    IO.println(s.length());
}
```

`switch` のパターンマッチングが活きるのは、「**複数の型（や、複数の条件）で分岐する**」ときです。
分岐が2〜3通り以上あるとき、`if-else if + instanceof` の連打を、`switch` ですっきりさせる ―― これが、ちょうどよい使いどころです。

---

## 5. 従来の switch と混同する

パターンマッチングの `switch`（型でマッチ）と、第6章の従来の `switch`（値でマッチ）は、書き方が似ていますが、別のものです。

```java
// 従来の switch（値で分岐）
switch (signal) {
    case RED -> "止まれ";       // 値（列挙定数）
}

// パターンマッチング（型で分岐）
switch (obj) {
    case Integer i -> "整数";   // 型（とパターン変数）
}
```

`case` のうしろが「**値**」なら従来の switch、「**型 と 変数名**」ならパターンマッチングです。
どちらも便利なので、「値で分けるのか、型で分けるのか」を意識して、使い分けましょう。

---

## まとめ

- `case` は上から順。**狭いパターンを上、広いパターンを下**（到達不能だとエラー）
- `Object` などは**網羅**が必要。**`default`** で「それ以外」をカバーする
- パターン変数は、型に合った**分かりやすい名前**を付ける
- 1つの型を見るだけなら `instanceof`、複数で分岐するなら `switch`
- 「値で分ける従来の switch」と「型で分けるパターンマッチング」を混同しない
