## なぜログが必要か

「動いた」だけでは、業務システムは終わりません。
動かしたあとの**運用**で、ログは不可欠の道具になります。

---

## ログの 4 つの役割

業務システムでログが必要になる場面は、おおむね 4 つに整理できます。

### 1. 障害発生時の調査

「**昨晩 23:30 に注文画面が壊れた**」 ―― そんな問い合わせを受けたとき、頼りになるのはログだけです。

- 何時何分何秒に
- どのリクエストが
- どこで何の例外を投げたのか

これがログから読み取れなければ、原因究明は不可能に近いです。

### 2. 業務イベントの記録

「**ユーザー A が、何時何分に、何の操作をした**」を残すのは、

- 監査要件（金融・医療・行政）
- 顧客対応（「履歴を確認したい」）
- 振り返り（KPI 集計）

など、業務的な要請です。
データベースに残しておくべきものもありますが、軽量に**ログとして**記録するケースも多いです。

### 3. 動作分析と性能改善

「**注文 API が、ふだんより遅い**」ことに気づくのは、ログの**統計**からです。

- レスポンス時間の分布
- どのエンドポイントが重いか
- どの SQL が遅いか

これらを集めて分析することで、**性能改善のヒント**が得られます。

### 4. セキュリティと不正検知

「**1 ユーザーから 1 秒に 100 回の認証失敗**」のような兆候は、ログから検出できます。

- 認証失敗の連続
- 不正な入力パターン
- 想定外の URL アクセス

監視ツール（SIEM、WAF）と組み合わせて、**不正アクセスの早期検知**につながります。

---

## どんなときにログを残すべきか

「ログを残すべきタイミング」は、5 つに整理できます。

| 場面 | 例 |
|---|---|
| **アプリの起動 / 終了** | バージョン、設定の概要 |
| **業務イベントの発生** | 注文受付、ユーザー登録、決済完了 |
| **外部システムとの境界** | DB / HTTP / メッセージキューへのリクエスト・レスポンス |
| **異常な状況** | 業務エラー、想定外の例外、リトライ発生 |
| **重要な分岐** | 設定で動きが変わるとき、フラグで処理を分けるとき |

逆に、**残さないほうがよい**のはこんな場面です。

- 1 トランザクションあたりループの中で**毎回**ログを出す
- 個人情報やパスワードを**そのまま**出す
- フィールドの値全部を**ダンプ**する（巨大な出力になる）

## ログの 3 つの原則

業務システムのログを書くときに、意識したい原則があります。

### 1. 「**人間が読める**」かつ「**機械が処理できる**」

人間が見て状況を把握できる必要がありますが、同時に、ログ集約システム（Splunk、Datadog、Elasticsearch）が**機械的に**処理しやすい形であることも大事です。

このバランスを取るのが、第4節で学ぶ**構造化ログ**です。

### 2. 「**一度だけ書く**」

第39章でも触れたとおり、同じイベントを**複数の場所**でログに残すと、

- どれが本物かわからなくなる
- ノイズで重要な情報が埋もれる
- ストレージとコストを圧迫する

「**捕まえた場所で一度**」が鉄則です。

### 3. 「**何のためにこのログを残すか**」を意識する

意味なくログを増やすと、**ノイズだらけ**になります。

- このログは、**どんな問い合わせ・障害**のときに役立つか?
- このログがなかったら、**何が分からなくなるか?**

迷ったら**書かない**くらいの感覚でちょうどいいです。
本当に必要なログだけを、**慎重に**選んで書きましょう。

## `System.out.println` ではダメな理由

入門段階の `System.out.println` でも、確かに文字は出ます。
ですが、業務システムでは、ほぼ常に**使えません**。

### 1. レベル制御ができない

本番では「**ERROR と WARN だけ**」、開発では「**DEBUG まで**」のように、レベルで出し分けたい場面が多くあります。
`System.out.println` には、その制御ができません。

### 2. 標準出力にしか出せない

ファイル、syslog、JSON、Kafka、CloudWatch、Loki ...
業務システムは、**いろんな出力先**にログを流します。
`System.out.println` は、標準出力に出すことしかできません。

### 3. メタ情報が付かない

```text
[INFO ] 2026-06-18 18:17:44.374 [main] c.e.log.LogDemoApplication - 注文を受け付けました
```

業務システムでは、こんな**タイムスタンプ・レベル・スレッド名・クラス名**を、自動で付けてほしいものです。
`System.out.println("注文を受け付けました")` には、これらが**何も付きません**。

### 4. 非同期化や性能チューニングができない

本番で大量のログを出すと、**ディスク I/O がボトルネック**になります。
バッファ・非同期書き出し・圧縮など、運用に必要な機能が、`System.out.println` にはまったくありません。

### 5. テスト時に困る

`System.out.println` は、テストの出力に**混ざってしまいます**。
ロガーなら、テストでは出力を抑える、特定のロガーだけ ON にするなど、**コントロール**できます。

---

## まとめると

業務システムのロギングは、

- **障害調査・業務記録・性能分析・セキュリティ**のために必要
- **書くべき場面**（起動 / 業務イベント / 境界 / 異常 / 重要分岐）を意識する
- **3 つの原則**（人間と機械が読める / 一度だけ / 意味を意識）を守る
- **`System.out.println` ではダメ**。専用のロガー（SLF4J + Logback）を使う
