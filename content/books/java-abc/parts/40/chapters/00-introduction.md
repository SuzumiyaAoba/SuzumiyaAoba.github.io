## はじめに ― この章で学ぶこと

第39章で、エラーハンドリングの設計を学びました。
そこで何度も登場した、もう一つの主役 ―― それが **ロギング**（logging）です。
本書の最終章では、このロギングを腰を据えて学んでいきます。

業務システムでは、ログは「**目に見えないもの**」を**目に見える**ようにする、唯一の手段です。

- 何時に、誰が、何をしたのか
- どこで、何の例外が起きたのか
- どのリクエストが、何ミリ秒かかったのか
- 外部 API はちゃんと応答したのか

これらすべてが、**ログとして残っていなければ、何もわかりません**。
プログラムが**壊れたとき**、いちばん頼りになるのが、過去のログです。

---

## 「`System.out.println` でいいのでは?」

入門段階では、`System.out.println(...)` でデバッグすることが多かったと思います。
これは学習用には便利ですが、業務システムでは**使ってはいけません**。

| | `System.out.println` | 本物のロガー |
|---|---|---|
| レベル制御 | できない | TRACE/DEBUG/INFO/WARN/ERROR |
| 出力先 | 標準出力のみ | ファイル、syslog、JSON、Kafka… |
| フォーマット | 固定 | 自由に設計 |
| 並行アクセスの安全性 | 不明 | 設計されている |
| パフォーマンス | バッファなし | バッファ・非同期出力可 |
| メタ情報 | 何も付かない | クラス名・スレッド名・タイムスタンプ自動 |

`System.out.println` は、業務システムでは**使い物にならない**のです。
代わりに使うのが、本章で学ぶ **SLF4J + Logback** という組み合わせです。

---

## この章のテーマ

Java の世界で、現代の標準的なロギング構成は次の組み合わせです。

```text
あなたのコード
       ↓
SLF4J           ← ロギングの「インタフェース」
       ↓
Logback         ← 実装。Spring Boot のデフォルト
       ↓
ログ出力先（コンソール / ファイル / 監視基盤 / ...）
```

第36章で見た「Spring の DI と、その実装」と同じ構造です。
**SLF4J が**仕様**、Logback が実装**にあたります。
本書のサンプルは、**Spring Boot 3.5・Java 25** で実機検証しています。

## この章で学ぶこと

第40章は、次の8つの節で構成されています。

| 節 | タイトル | 内容 |
|---|---|---|
| 1 | なぜログが必要か | `System.out.println` を卒業する |
| 2 | SLF4J と Logback | インターフェースと実装の関係 |
| 3 | ログレベル | TRACE / DEBUG / INFO / WARN / ERROR |
| 4 | 構造化ログ | JSON で書く、機械が読めるログ |
| 5 | MDC でリクエスト追跡 | リクエスト ID で 1 リクエストを追う |
| 6 | 設定とローテーション | `logback-spring.xml` とローテーション |
| 7 | よくあるつまずき | パターンの落とし穴と運用視点 |

前半（1〜3節）で、ロギングの**書き方**を習得します。
中盤（4〜5節）で、**機械が読める・追跡できる**ログを作ります。
後半（6〜7節）で、運用フェーズまで含めた**現場の知識**を整理します。

## この章を読み終えると

第40章を読み終えるころには、次のことができるようになっています。

- SLF4J で、適切なログレベルのログを書ける
- ログに**例外オブジェクト**を正しく渡せる
- 必要な情報を**メタデータ付き**で残せる
- **MDC**でリクエスト ID を追跡できる
- 本番運用に耐える**設定ファイル**を作れる
- ログレビューで、現場の不備を**指摘できる**
