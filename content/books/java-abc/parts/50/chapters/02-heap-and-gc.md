## ヒープと GC の調整

第42章で扱った GC とヒープサイズの話を、**実践的な観点**で再整理します。
この節は、第42章の応用編 ―― 「**結局、何をどう調整すれば現場で効くのか**」を中心に据えます。

---

## 「まずは何もしない」

意外と知られていない事実から始めます。
**現代の JVM は、何も設定しなくてもまあまあ動く**ように作られています。

- Java 9 以降のデフォルト GC は **G1**
- ヒープサイズは、物理メモリの 1/4 程度に自動設定
- JIT・GC・スレッドプールも、エンジニアの介入なしで「**それなりに**」動く

つまり、**いきなり JVM フラグを足す前に、まずデフォルトでどうなるかを観察**しましょう。
「**設定しない**」が、たいていの場合の正解です。

---

## 何が起きているか先に知る

調整の前に、まず**何が起きているか**を見ます。手順は:

1. **`-Xlog:gc`** で GC ログを取る（第42章）
2. **JFR** を 5〜10 分取る（第48章）
3. **JMC** で開く、または `jfr summary`

ここで見るのは:

- **マイナー GC の頻度** ―― 1 秒あたり何回?
- **メジャー GC の頻度** ―― 1 分・1 時間あたり何回?
- **GC の合計時間** ―― 全実行時間の何%?
- **GC 後のヒープ使用量** ―― 定常状態は何 MB?

これらが、調整の**出発点**になります。

---

## 「**`-Xmx` を増やす**」がいつ効くか

最も**よく見る調整**が、最大ヒープサイズ `-Xmx` の引き上げです。
これが効くのは、こういうケース:

- **GC 後のヒープ使用量**が `-Xmx` の **70 % 以上**で、頭打ち
- マイナー GC が**毎秒数回**走り、合計の GC 時間が無視できない
- **メジャー GC が頻発**している

逆に、こういうときに `-Xmx` を増やしても**意味がない**:

- GC 後の使用量が `-Xmx` の **30 %**未満
- 「とりあえず多めに」と取った大きすぎる値（実使用量とかけ離れている）

ヒープサイズの目安は、**定常状態の 2〜3 倍**でした。
これが現実的かつ無駄のないサイジングです。

---

## GC アルゴリズムの選び方を実践で

第42章で、GC を 4 種類紹介しました（G1・ZGC・Parallel・Serial）。
実際にどう選ぶか:

### G1（デフォルト）

- ヒープ 数 GB〜数十 GB
- レイテンシ・スループットのバランス
- **大半の Web アプリ**はこれで十分

### ZGC

- ヒープ 数百 MB〜数 TB
- **STW 1 ms 以下**を狙う
- **広告配信・トレーディング・チャット**のような低遅延要件
- Java 21 以降の世代別 ZGC で、ヒープ効率も改善

### Parallel

- バッチ・分析・大量計算
- レイテンシよりスループット最大化
- **CPU を使い切ってよい**前提

### Serial

- 小さなコンテナ、CPU 1 コア
- 軽量、シンプル

「**迷ったら G1、レイテンシで困ったら ZGC を試す**」のが、現実的な指針です。

---

## 「**コードを変える**」vs「**フラグを変える**」

これも、現場でよくあるジレンマです。

| 状況 | 推奨 |
|---|---|
| アロケーション過多が原因 | **コードを変える**（オブジェクト再利用、`StringBuilder`） |
| ヒープサイズ不足 | **`-Xmx` を増やす** |
| GC のレイテンシピーク | **GC アルゴリズム変更**（ZGC など） |
| キャッシュリーク | **コードを変える**（TTL・上限） |
| メタスペース不足 | **`-XX:MaxMetaspaceSize`** か、**動的クラス生成を減らす** |

コード側の問題（アロケーション過多、リーク）を**フラグで誤魔化す**のは、本質的な解にならないことが多いです。
プロファイルを見て、**本当の原因**にアプローチしましょう。

## コンテナ環境での注意

Docker・Kubernetes でアプリを動かすときの、現代的な設定例を挙げます。

```text line-numbers=false
$ java \
  -XX:+UseG1GC \
  -XX:MaxRAMPercentage=75.0 \
  -XX:MaxGCPauseMillis=200 \
  -XX:+ExitOnOutOfMemoryError \
  -Xlog:gc:file=/var/log/gc.log::filecount=5,filesize=10M \
  -jar app.jar
```

それぞれの意味:

- **`-XX:+UseG1GC`** ―― 明示的に G1（コンテナで Serial に勝手に切り替わらないように）
- **`-XX:MaxRAMPercentage=75.0`** ―― コンテナメモリの 75 % をヒープに
- **`-XX:MaxGCPauseMillis=200`** ―― 目標停止時間 200 ms
- **`-XX:+ExitOnOutOfMemoryError`** ―― OOM 時にプロセス終了（K8s が再起動してくれる）
- **`-Xlog:gc:file=...`** ―― GC ログをローテーション付きで

これだけ書いておけば、コンテナの中の Java は安定して動きます。
**「設定 0 個**」よりも、**「適切な数個」**のほうが、運用は楽になります。

---

## 「**最小ヒープ + ヒープ収縮**」 ―― 動的サイズ調整

Java 12 以降の G1 は、**`-XX:G1PeriodicGCInterval`** などで、アイドル時にヒープを縮める機能があります。
これにより、コンテナのメモリ実使用量を**定期的に**コミット解除して、ホストに返せます。

```text line-numbers=false
$ java -XX:G1PeriodicGCInterval=5000 ...
```

サーバーレスや**メモリ削減を厳しく**したいワークロードでは、こうした「**動的調整**」も検討の余地があります。

---

## 「**過剰なチューニング**」のサイン

「JVM フラグが 20 個並んでいる」 ―― これは**チューニングしすぎ**のサインです。

- 何が効いているか不明
- 新しいバージョンの JVM で**意味が変わる**
- 新メンバーが**触りたがらない**

定期的に**フラグの棚卸し**をしましょう。
「これがなくても本当に問題か?」を 1 つずつ確認し、不要なものは削る ―― これも立派なチューニングです。

---

## まとめると

- **まず何もしない**で観察する
- GC ログ・JFR で**現状を把握**する
- `-Xmx` 増は、**実使用量と比較**してから
- GC 選択は **G1 デフォルト、ZGC は低遅延、Parallel はバッチ**
- コードの問題を**フラグで誤魔化さない**
- コンテナでは、**最低限のフラグ**を入れて運用
- フラグは定期的に**棚卸し**
