## はじめに ― この章で学ぶこと

ふつう、Java のコードは「**書いたとおりに**」動きます。
`new User("Alice")` と書けば `User` のインスタンスができ、`user.getName()` と書けば名前が返ります。
コードの中に出てくるクラス名・メソッド名は、**コンパイル時に決まっている**のがふつうです。

これに対し、

- ファイルから読んだ**文字列**「`com.example.User`」をクラス名として使う
- 設定ファイルから読んだメソッド名「`getName`」を**実行時に**呼ぶ
- アノテーション `@Inject` の付いたフィールドを**実行時に**全部見つける

といった、**実行時に型を覗いたり操作したり**する仕組みが**リフレクション**（Reflection）です。
Spring・Jackson・Hibernate・JUnit ―― 世界中の有名なフレームワークが、リフレクションの上に立っています。

第46章では、

- リフレクションで**何ができるか**
- どんな**落とし穴**があるか
- より高速で型安全な代替 **MethodHandle**

を扱います。

---

## 「**コードが、コードを見ている**」

リフレクションを直感的に言うと、

> プログラムが、**自分自身の構造**を覗き、操作する

ということです。
鏡（reflection）を使って、コードがコードを見るイメージから「リフレクション」と呼ばれます。

```java
// 通常のコード ―― 型を知っている
User user = new User("Alice");
String name = user.getName();
```

```java
// リフレクション ―― 型を実行時に決める
Class<?> clazz = Class.forName("com.example.User");
Object user = clazz.getConstructor(String.class).newInstance("Alice");
String name = (String) clazz.getMethod("getName").invoke(user);
```

書く側はちょっと面倒ですが、

- **クラス名やメソッド名**が、ソースに直接書かれていない
- だから、**設定で**動きを変えられる
- **動的なプラグイン**機構を作れる

という決定的な利点があります。

---

## どこで使われているか

リフレクションの目に見える出番は少ないですが、**水面下では**ほとんどすべてのフレームワークが使っています。

| ライブラリ | リフレクションで何をしているか |
|---|---|
| **Spring** | `@Component`・`@Autowired` のクラス／フィールドを探す |
| **Jackson** | JSON のキーから、対応するフィールドを見つけて値を入れる |
| **Hibernate** | エンティティのフィールドに、DB の値を入れる |
| **JUnit** | `@Test` の付いたメソッドを探して呼ぶ |
| **Lombok** | コンパイル時にコードを生成するが、その仕組みはリフレクション API の親戚 |

ふだん意識しない場所で、**リフレクションを使ったコードが動いている** ―― これがフレームワーク時代の Java です。
だからこそ、リフレクションを知ることは、**フレームワークを理解する**こととほぼ同義です。

---

## ただし、危険でもある

リフレクションは強力ですが、**3 つの危険**があります。

1. **遅い**: 通常のメソッド呼び出しより、桁違いに遅い（JIT が最適化しにくい）
2. **型が外れる**: コンパイル時の型チェックが効かない（`Object` に戻る）
3. **モジュールの境界を破る**: `private` も触れるので、設計の意図を壊しやすい

特に 3 は、第45章の JPMS で見たとおり、`opens` 宣言なしでは防がれます。

リフレクションは「**普段のコードでは使わない、フレームワーク作者のための道具**」と位置づけるのが、健全な距離感です。
それでも、**仕組みを知らないとフレームワークの挙動が読めない**ので、本章で道具立てを整理します。

## この章を読み終えると

第46章を読み終えるころには、次のことができるようになっています。

- リフレクションで**できること・できないこと**を区別できる
- `Class<?>` オブジェクトを 3 つの方法で取得できる
- フィールド・メソッドの読み書き・呼び出しを書ける
- カスタムアノテーションを定義し、実行時に読み取れる
- **MethodHandle** がリフレクションより高速な理由を説明できる
- フレームワークの**裏で何が起きているか**を、ある程度想像できる

それでは、最初の節「**リフレクションとは**」から始めましょう。
