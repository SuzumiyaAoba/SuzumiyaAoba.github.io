## はじめに ― この章で学ぶこと

ここまでの章で、

- Java の言語機能（第1〜3部）
- ビルドツール・テスト（第31〜33章）

を、ひととおり身につけました。
これで、**「動く」プログラムは、自由に書ける**ようになっています。

ですが、実務では、もう一段高い問題が待っています。

- 機能追加するたびに、別のクラスが壊れる
- 同じような分岐が、いろんなところに散らばっている
- 「ここだけ変えたい」のに、たくさんのクラスを直すことになる

こうした「**変えにくいコード**」は、書いた瞬間は気づきにくく、半年後に困らされるものです。
この章では、その「**変えやすさ**」のための、先人たちが整理した型 ―― **デザインパターン**を学びます。

---

## デザインパターンって何?

**デザインパターン**（design pattern）は、

> 「**よく出会う設計上の問題**に対する、典型的な解決の型」

のことです。
1990 年代に、**GoF**（Gang of Four、4人組）と呼ばれる4人の著者が、23 個のパターンを整理した本（『デザインパターン』）が出て、それ以降、ソフトウェア設計の**共通言語**として定着しました。

「`Strategy` パターンで書こう」と言えば、Java を書く人同士で**設計の意図がぱっと伝わる**。
そんな**業界の共通語**を、入門書らしい範囲で身につけるのが、この章のゴールです。

---

## 「23 個全部」は学ばない

GoF パターンは 23 個ありますが、本書では、

- 現代の Java で**素直に書けるもの**
- 実務で**よく見るもの**

に絞って、**5 つだけ**を取り上げます。

| パターン | 解決する問題 |
|---|---|
| **Strategy**（ストラテジー） | アルゴリズムを差し替えたい |
| **Factory**（ファクトリー） | オブジェクト生成の責任を切り出したい |
| **Singleton**（シングルトン） | インスタンスを 1 つだけにしたい |
| **Observer**（オブザーバー） | 状態の変化を、複数の関係者に通知したい |
| **Template Method**（テンプレートメソッド） | 大まかな流れは決まっていて、一部だけ差し替えたい |

ほかのパターンは、必要になったときに調べれば十分です。
**「設計パターンの心構え」が身につけば、それで第一歩としては成功**です。

---

## モダン Java では、書き方が変わる

GoF が書かれたのは 1994 年です。
当時の Java には、ラムダ式も、関数型インタフェースも、レコードもありませんでした。
そのため、**いまの Java で素直に書くと、ずっと短くなる**パターンが、たくさんあります。

本書では、

- まず **古典的な書き方** を見せ
- そのあと、**ラムダ式やレコードを使った今風の書き方**を見せます

両方を知っておくと、古い書籍やコードベースを読むときに役立ちます。

---

## この章で学ぶこと

第34章は、次の8つの節で構成されています。

| 節 | タイトル | 内容 |
|---|---|---|
| 1 | デザインパターンとは | 共通言語としての設計パターン |
| 2 | Strategy パターン | 振る舞いを差し替える |
| 3 | Factory パターン | 生成の責任を切り出す |
| 4 | Singleton パターン | インスタンスを1つにする |
| 5 | Observer パターン | イベントを通知する |
| 6 | Template Method パターン | 流れの一部を差し替える |
| 7 | よくあるつまずき | パターン病・過剰設計 |

前半（1節）でパターンの心構えを学び、中盤（2〜6節）で代表的な5つのパターンを書きます。
最後（7節）に、「**パターンを使うべきでないとき**」を整理します。

---

## この章を読み終えると

第34章を読み終えるころには、次のことができるようになっています。

- 「Strategy」「Factory」などの名前を聞いて、何の話かわかる
- それぞれのパターンを、Java 25 で素直に書ける
- ラムダ式やレコードを使った、**今風の書き方**ができる
- 「パターンを使うべきでないとき」を判断できる

---

> **補足: パターンは「使う場所」ではなく「考え方」を学ぶ**
>
> パターン名を覚えて、コードに無理やり当てはめると、かえって設計が悪くなります。
> **「どんな問題を解いているのか」**だけを覚えて、自分の場面で**自然に出てくる**ようになるのが理想です。
> この章は、「読めるようになる」「指差せるようになる」を目標にしましょう。

それでは、最初の節「デザインパターンとは」から始めましょう。
