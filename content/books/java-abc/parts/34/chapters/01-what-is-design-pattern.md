## デザインパターンとは

デザインパターンを、次のように定義しておきましょう。

> **「同じ問題に何度も出会ううちに、自然と落ち着いた解き方」を、名前付きで整理したもの**[^gof-book]

「自然と落ち着いた解き方」というのが、とても大事です。
パターンは、誰かが机上で考案したものではなく、**現場で繰り返し使われた解決法を、あとから名前で呼べるようにした**もの。
ですから、

- 「このパターンを使え」と命令されて使うものではない
- 「気づいたら、同じような形になっていた」を、共有しやすくしたもの

くらいの理解で、ちょうどよいのです。

---

## なぜ「名前」が大事か

パターンを学ぶいちばんの効能は、**会話の語彙が増える**ことです。

たとえば、こんなやり取りを想像してください。

> A: 「決済処理が、クレジットカード・銀行振込・ポイントで分岐していて、ifがネストしてるのを直したい」
> B: 「**Strategy** で切り出したらどう?」
> A: 「あ、そっか、それぞれを別クラスにして、interface で受け取るのね。すぐやってみる」

`Strategy` という 1 語があるおかげで、A と B は、

- どんなコード構造にするか
- どんなクラスを切り出すか
- どこを `interface` 化するか

を、**何分も話さなくても合意できる**わけです。
これが、パターンの最大の価値です。

---

## パターンは「銀の弾丸」ではない

ただし、注意してほしいのが、

> 「パターンに当てはめれば、よい設計になる」**ではない**

ということです。
むしろ、入門段階で気をつけるべきは、

- パターンを覚えた瞬間に、**どこにでも使いたくなる**（パターン病）
- シンプルなコードを、**わざわざパターン化して複雑にする**

という、初学者あるあるです。

> **デザインパターンは「困ったときの選択肢」**

くらいに思っておきましょう。
「いま、困っていないのに使う」と、確実に逆効果になります。

## パターンの 3 つの分類

GoF パターンは、目的別に大きく 3 つに分けられます。
本書では、深入りはしませんが、視野のために紹介しておきます。

### 1. 生成に関するパターン（Creational）

オブジェクトを**どう作るか**に関するパターンです。

- **Factory Method**（第3節）
- **Abstract Factory**
- **Builder**
- **Prototype**
- **Singleton**（第4節）

### 2. 構造に関するパターン（Structural）

クラスやオブジェクトを**どう組み合わせるか**に関するパターンです。

- **Adapter**
- **Decorator**
- **Composite**
- **Facade**
- **Proxy**

### 3. 振る舞いに関するパターン（Behavioral）

オブジェクト同士の**通信や責任分担**に関するパターンです。

- **Strategy**（第2節）
- **Observer**（第5節）
- **Template Method**（第6節）
- **Iterator**
- **Chain of Responsibility**

本書で扱うのは、各分類から「とくに現代の Java で書きやすく、よく見る」5 つです。

## モダン Java とパターン

GoF パターンの中には、いまの Java では**言語機能で十分**になったものがあります。

| GoF パターン | 現代の Java で素直に書く方法 |
|---|---|
| Strategy | **ラムダ式・関数型インターフェース**（第22章） |
| Iterator | **拡張 for 文**（第8章）と `Iterable` |
| Observer | **`Flow.Publisher` / `Flow.Subscriber`**（標準ライブラリ） |
| Command | **`Runnable` / `Consumer`** などのラムダ |

つまり、**パターンを言語が吸収した**わけです。
このため、本書のパターンの章は「**古典 vs 現代**」の対比を意識しながら読み進めるとよいでしょう。

---

## パターンとアンチパターン

「**こうすると、よくない**」ものを、**アンチパターン**（anti-pattern）と呼びます。
パターンを学ぶうえで、アンチパターンも合わせて知っておくと、判断がブレません。

| アンチパターン | こう書くな |
|---|---|
| 神クラス（God Class） | 1 クラスが何百ものことをやる |
| 巨大メソッド（Long Method） | 1 メソッドが数百行になる |
| 過剰な抽象化 | 使わない `interface` だらけにする |
| パターン病 | どのコードにもパターンを当てはめる |

これらの匂いを察知する力 ―― これも、設計の感性のひとつです。

---

## まとめ

- デザインパターンは、**よく出会う問題に対する、典型的な解き方**を名前で整理したもの
- 最大の効能は、**設計の語彙が増えること**
- 「**困ったときの選択肢**」であって、「使うべきもの」ではありません
- 生成・構造・振る舞いの 3 分類
- いまの Java では、**言語機能がパターンを吸収**しているものも多いです
- アンチパターン（神クラス、過剰抽象化など）も合わせて学ぶ

次の節からは、代表的なパターンを 5 つ、書き方とともに見ていきます。
まずは、**Strategy** パターンからです。

[^gof-book]: Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides, *Design Patterns: Elements of Reusable Object-Oriented Software*（Addison-Wesley, 1994, ISBN 0-201-63361-2）。著者4名が "Gang of Four"（GoF）と呼ばれ、Singleton／Factory／Strategy／Observer など23のパターンを生成系・構造系・振る舞い系に分類して整理した。本書「第6部 デザインパターン」もこの分類に従う。
