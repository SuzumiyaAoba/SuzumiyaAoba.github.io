## 用語集 ― この章で学んだ言葉

第34章では、代表的な5つのデザインパターンと、その現代的な書き方を学びました。
節ごとに、言葉を整理しておきます。

---

## デザインパターンの基礎（第1節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| デザインパターン | design pattern | よく出会う設計上の問題に対する典型的な解き方 |
| GoF パターン | Gang of Four | 1994年の書籍『デザインパターン』で整理された23パターン |
| 生成パターン | Creational | オブジェクトの作り方に関するパターン |
| 構造パターン | Structural | クラス・オブジェクトの組み立てに関するパターン |
| 振る舞いパターン | Behavioral | オブジェクト同士の通信に関するパターン |
| アンチパターン | anti-pattern | やってはいけない典型的な設計 |
| 神クラス | God Class | 1クラスが多すぎる責務を持ってしまった状態 |

## Strategy（第2節）

| 用語 | 意味 |
|---|---|
| Strategy パターン | 振る舞いを取り替え可能な部品として切り出す |
| インタフェース | アルゴリズムの共通の窓口 |
| 関数型インタフェース | メソッドが1つだけのインタフェース。ラムダ式で実装できる |
| `IntConsumer` ほか | 標準の関数型インタフェース（第22章） |

## Factory（第3節）

| 用語 | 意味 |
|---|---|
| Factory パターン | オブジェクトの作り方を利用側から隠す |
| Simple Factory | `switch` で生成を切り替える素朴な書き方 |
| Factory Method | サブクラスに「作り方」を任せる古典的なパターン |
| `Map<String, Supplier<T>>` | 登録ベースで Factory を書くモダンな書き方 |

---

## Singleton（第4節）

| 用語 | 意味 |
|---|---|
| Singleton パターン | インスタンスを1つだけにする |
| 古典的シングルトン | `private` コンストラクタ + `getInstance()` |
| マルチスレッドの落とし穴 | 素朴な実装は同時実行で複数インスタンスができる |
| enum シングルトン | Effective Java 推奨。スレッド安全・シリアライズ安全 |
| DI のシングルトンスコープ | Spring などで「1インスタンス」を管理してもらう |

---

## Observer（第5節）

| 用語 | 意味 |
|---|---|
| Observer パターン | 状態変化を複数の関係者に通知する |
| Subject / Publisher | 通知する側（観察される側） |
| Observer / Subscriber | 通知される側（観察する側） |
| メモリリーク | 解除を忘れて参照が残り続けること |
| `Flow.Publisher` / `Flow.Subscriber` | `java.util.concurrent.Flow` の Reactive Streams 実装 |
| バックプレッシャー | 受け取る速度を制御する仕組み |

---

## Template Method（第6節）

| 用語 | 意味 |
|---|---|
| Template Method パターン | 流れは親クラスで決め、一部をサブクラスに任せる |
| テンプレートメソッド | 全体の流れを書く `final` メソッド |
| ハリウッド原則 | 「呼び出すな、こちらから呼び出す」 |
| 関数オブジェクトで合成 | 継承の代わりに関数を引数で渡す現代的な書き方 |

## 落とし穴と原則（第7節）

| 用語 | 意味 |
|---|---|
| パターン病 | どんなコードにもパターンを当てはめたがる症状 |
| 過剰な抽象化 | 使わない interface を切ってしまうアンチパターン |
| パターン名の混乱 | Adapter / Decorator / Facade / Proxy が混同されやすい |
| SOLID 原則 | パターンの背景にある5つの設計原則（第35章） |

---

## おわりに ― 次の章で「なぜ」を学ぶ

第34章では、代表的な5つのパターンと、その背景を見てきました。
ここまで読んで、

- 「なぜ、わざわざクラスを分けるんだろう?」
- 「`interface` 化って、本当に意味があるの?」
- 「Singleton はなぜ嫌われがちなの?」

といった疑問が残っているかもしれません。
それらに答えるのが、次の第35章 ― **SOLID 原則**です。

SOLID は、「**変更に強い設計**を作るための、5つの基本原則」のセットです。
これを身につけると、パターンを**機械的に当てはめる**段階から、**自分で選んで判断できる**段階に進めます。
それでは、次章でお会いしましょう。
