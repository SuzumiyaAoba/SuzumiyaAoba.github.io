## よくあるつまずき

並行処理で、初心者がつまずきやすいポイントを、まとめて確認します。
並行処理は難しいテーマなので、「**これらの落とし穴がある**」と知っておくことが、特に大切です。

---

## 1. 出力の順番が決まると思い込む

並行処理では、**スレッドの実行順は保証されません**。
「`start()` した順に動くはず」と思い込むと、思わぬ結果に戸惑います。

```java
for (int i = 1; i <= 3; i++) {
    int id = i;
    new Thread(() -> IO.println("スレッド " + id)).start();
}
// → 1, 2, 3 の順とは限らない
```

順番が大切なら、`join()` で待つ、`Future` で結果を順に受け取る、といった工夫が必要です。
「**並行処理では、順番は決まらないのが基本**」と、いつも意識しましょう。

---

## 2. run() を呼んでしまう

スレッドを動かすのは **`start()`** です。`run()` を直接呼ぶと、**別スレッドにならず**、その場でふつうに実行されてしまいます。

```java
Thread t = new Thread(() -> IO.println("処理"));
t.run();    // ✕ 別スレッドにならない（ただのメソッド呼び出し）
t.start();  // ◯ 別スレッドで動く
```

見た目は動いているように見えるので、気づきにくい間違いです。
「**新しいスレッドで動かすなら `start()`**」と、しっかり覚えましょう。

---

## 3. 共有データの競合を、見落とす

第3節で学んだ、最大の落とし穴です。
複数のスレッドが同じ可変データをいじると、**競合状態**で、結果がおかしくなります。

```java
count++;   // 複数スレッドから同時に → 増やし損ねる
```

しかも、競合状態は「**運が悪いときだけ**」起きるので、テストでは気づかず、本番で突然おかしくなる、という、たちの悪いバグになりがちです。

- そもそも**可変データを共有しない**（いちばん安全）
- 共有するなら**不変オブジェクト**（第13章）にする
- 数値の集計などは `AtomicInteger`、`synchronized` などの**安全な手段**を使う

「複数スレッドで、同じものをいじっていないか？」を、つねに意識しましょう。

---

## 4. join を忘れて、結果が出る前に終わる

スレッドの処理が終わるのを待たずに、結果を使おうとすると、まだ処理が終わっておらず、おかしな値になることがあります。

```java
Thread t = new Thread(() -> { /* 時間のかかる処理 */ });
t.start();
// t.join() を忘れると…
IO.println(結果);   // まだ処理が終わっていないかも
```

「スレッドの結果を使う前には、`join()`（または `Future.get()`）で、**終了を待つ**」のを忘れないようにしましょう。

---

## 5. スレッドを作りすぎる

従来のスレッド（`new Thread`）を、何千・何万個も作ると、メモリを使い果たして、プログラムが**パンク**します。

```java
for (int i = 0; i < 1000000; i++) {
    new Thread(() -> { ... }).start();   // ✕ 100万個は無理（従来スレッド）
}
```

たくさんのタスクをさばきたいときは、

- **`ExecutorService`**（スレッドプール）で、スレッドを使い回す
- 待ち時間の多い処理なら、**仮想スレッド**（大量に作れる）を使う

と、適切なしくみを使いましょう。「スレッドは、無限に作れるものではない（従来スレッドの場合）」と知っておくことが大切です。

---

## まとめ

- 並行処理では、**実行順は保証されない**（順番が要るなら `join`・`Future`）
- スレッドを動かすのは **`start()`**（`run()` は別スレッドにならない）
- **共有データの競合**に注意（共有しない・不変にする・安全な道具を使う）
- 結果を使う前に、**`join()` で終了を待つ**
- 従来スレッドの**作りすぎ**に注意（`ExecutorService`・仮想スレッドを使う）
