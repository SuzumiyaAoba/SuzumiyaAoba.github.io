## 共有データの問題 ― 競合状態

並行処理の、最大の難所がここです。
複数のスレッドが、**同じデータを同時にいじる**と、思わぬおかしな結果が起きます。
これを **競合状態**（race condition）と呼びます。
この節では、その危険性と、対処の考え方を学びます。

---

## 同じデータを、みんなで増やす

例として、2つのスレッドで、1つのカウンターを、それぞれ10万回ずつ増やしてみます。
合計で20万回増やすので、結果は **200000** になるはずです。

```java
import java.util.concurrent.*;

class Counter {
    static int count = 0;   // みんなで共有するカウンター
}

Runnable task = () -> {
    for (int i = 0; i < 100000; i++) {
        Counter.count++;     // 共有のカウンターを増やす
    }
};

Thread a = new Thread(task);
Thread b = new Thread(task);
a.start();
b.start();
a.join();
b.join();

IO.println("結果: " + Counter.count);
```

```text
（実行の一例 ― 毎回変わる）
結果: 132495
```

結果は **200000 になりません**。しかも、実行するたびに、違う数になります。
2つのスレッドが、同じ `count` を同時に増やそうとして、**増やし損ねている**のです。

---

## なぜ、増やし損ねるのか

`count++` は、一見1つの操作に見えますが、実は**3つのステップ**でできています。

```text
count++ の中身:
  ① count の値を読む（たとえば 100）
  ② 1 を足す（101）
  ③ count に書き戻す（101）
```

ここで、2つのスレッド A・B が、運悪く同時に動くと、こうなることがあります。

```text
スレッドA: ① count を読む（100）
スレッドB: ① count を読む（100）  ← A がまだ書き戻す前に、同じ 100 を読んだ！
スレッドA: ②③ 101 を書き戻す
スレッドB: ②③ 101 を書き戻す    ← 本当は 102 のはずが、101 になった
```

2回増やしたはずなのに、結果は1回分しか増えていません。
このように、「読んで・足して・書き戻す」の**途中に、別のスレッドが割り込む**と、増やした分が消えてしまうのです。
これが、**競合状態**です。「いつ割り込まれるか」は運任せなので、結果も毎回変わります。

---

## 対処1 ― synchronized で、1人ずつにする

この問題を防ぐ古典的な方法が、**`synchronized`**（シンクロナイズド）です。
`synchronized` を付けた処理は、「**同時に1つのスレッドしか実行できない**」ようになります。

```java
class Counter {
    static int count = 0;

    static synchronized void increment() {   // 同時に1スレッドしか入れない
        count++;
    }
}
```

`synchronized` を付けると、あるスレッドがこの処理を実行している間、ほかのスレッドは**順番待ち**になります。
これにより、「読んで・足して・書き戻す」が、割り込まれずに完了するので、結果は正しく 200000 になります。

「みんなで使う処理に、`synchronized` で順番待ちの仕切りを設ける」 ―― これが、基本的な対処です。
（ただし、順番待ちが増えると、並行処理の速さのメリットは減ります。使いどころのバランスが大切です。）

---

## 対処2 ― そもそも共有しない

もっとよい対処は、「**そもそも、可変のデータを共有しない**」ことです。

並行処理のトラブルの原因は、ほぼすべて「**複数のスレッドが、同じ可変データをいじる**」ことにあります。
だったら、最初から共有しなければ、問題は起きません。

- スレッドごとに、別々のデータを持つ
- 共有するなら、**不変オブジェクト**（第13章）にする（変わらないものは、同時に読んでも安全）
- 数値の集計などは、専用の安全な道具を使う（次の補足）

第13章で「不変オブジェクトは安全に共有できる」と学びました。
その安全さが、まさに並行処理で効いてきます。「**変わらないものは、取り合いにならない**」のです。

> **補足: 安全なカウンター ― AtomicInteger**
>
> 「数値を、複数スレッドから安全に増やしたい」という、よくある用途には、**`AtomicInteger`** という専用クラスがあります。
> `incrementAndGet()` を使えば、`synchronized` を書かなくても、安全に増やせます。
>
> ```java
> import java.util.concurrent.atomic.AtomicInteger;
> AtomicInteger count = new AtomicInteger(0);
> // 各スレッドで count.incrementAndGet();
> // → 結果は、正しく 200000 になる
> ```
>
> 並行処理用の安全な道具（`java.util.concurrent` パッケージ）は、ほかにもたくさんあります。「自分で `synchronized` を書くより、用意された安全な道具を使う」のが、現代の定石です。

---

## まとめ

- 複数のスレッドが**同じ可変データを同時にいじる**と、**競合状態**でおかしな結果になる
- 原因は、`count++` のような操作が「読む・足す・書き戻す」の途中で割り込まれること
- 対処1：**`synchronized`** で「同時に1スレッドだけ」にする（順番待ち）
- 対処2（より良い）：**そもそも可変データを共有しない**。共有するなら**不変オブジェクト**に
- 数値の集計などは、`AtomicInteger` のような**安全な道具**を使う
