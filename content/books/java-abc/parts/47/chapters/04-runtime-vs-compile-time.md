## ランタイム処理とコンパイル時処理

アノテーションを使った「**メタプログラミング**」には、大きく **2 つの世界**があります。

- **ランタイム処理**: 実行時にリフレクションでアノテーションを読み、振る舞いを変える
- **コンパイル時処理**: コンパイル時にアノテーションプロセッサが起動し、コードを生成する

この節では、それぞれの**得意・不得意**を整理し、使い分けの判断軸を作ります。

---

## ランタイム処理 ―― リフレクションでつまむ

第46章でやった、Java 入門者にとっても比較的なじみのあるアプローチです。

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@interface Cached {
    int seconds() default 60;
}

// 実行時にこう読む
for (Method m : service.getClass().getDeclaredMethods()) {
    Cached c = m.getAnnotation(Cached.class);
    if (c != null) {
        // キャッシュ処理を差し込む
    }
}
```

利点と欠点を整理します。

| 良い点 | 悪い点 |
|---|---|
| アノテーションを**動的に解釈**できる（設定で振る舞いを変えるなど） | リフレクションの**コスト**がかかる |
| プロセッサや生成コードの仕掛けが**要らない** | 型情報が`Object` になる |
| 設計が**シンプル**で書きやすい | アノテーションが付いた要素を**全件スキャン**する必要 |

Spring のような**動的な DI コンテナ**には欠かせない手法です。
コンテナの起動時に、すべての `@Component` をスキャン、リフレクションで生成 ―― これがランタイム処理の典型です。

---

## コンパイル時処理 ―― プロセッサが代わりにコードを書く

もう一方は、**`javac` の中で動くアノテーションプロセッサ**が、追加のソースを生成するアプローチです。

イメージで言うと、

```text line-numbers=false
あなたが書いたコード:        プロセッサが書いたコード:
                            
@Data                       public class User {
public class User {            ...
    private String name;       public String getName() { return name; }
    private int age;           public void setName(String name) { this.name = name; }
}                              ...
                            }
```

`javac` は両方をまとめてコンパイルし、`.class` には**両方が含まれた**ものができます。
ランタイムでは、生成されたメソッドも**ふつうのメソッド**として、JIT 最適化を含めて全力で動きます。

| 良い点 | 悪い点 |
|---|---|
| **実行時コストはゼロ** ―― 普通のコードと同じ速度 | プロセッサの**実装が難しい** |
| **型安全** ―― 生成コードもふつうのコード | コンパイルが**遅くなる** |
| **デバッグできる** ―― 生成されたコードを直接見られる | IDE との統合に**癖**がある |

Lombok や MapStruct のような**コード生成系**ライブラリは、すべてこちらの世界です。

---

## どちらを選ぶか

両者の使い分けを、表で整理します。

| やりたいこと | 推奨 |
|---|---|
| 起動時に **1 回**だけ走る処理（DI コンテナ） | ランタイム |
| 1 リクエストごとに**たまに**走る処理 | ランタイム |
| **頻繁に走る**処理（ホットパス） | コンパイル時（生成） |
| **コンパイル時にしか分からない**情報を使う | コンパイル時 |
| **ユーザーがアノテーションを動的に増減**したい | ランタイム |
| 生成されるコードを **デバッグできるようにしたい** | コンパイル時 |

判断軸の本質は、

- **柔軟性 vs パフォーマンス・型安全性**

です。
ランタイムは柔軟、コンパイル時は速くて安全。
**フレームワークの「フロント」はランタイム、「ホットパス」はコンパイル時**、と覚えると分かりやすいです。

---

## ハイブリッド ―― 両方を組み合わせる

実は、現実のフレームワークでは**両方**を使うことが多いです。

### Spring 5+ の `@Configuration`

- ランタイム: `@Configuration` クラスを発見し、`@Bean` を呼んでオブジェクトを作る
- コンパイル時: `@ConfigurationProperties` の getter/setter チェック、メタデータ生成

### Quarkus・Micronaut

- コンパイル時に、Spring が「**リフレクションで**」やっていたことの大半を**コード生成**してしまう
- 起動が**圧倒的に速く**なる（Spring 比で 10 倍以上）

Spring Boot も Native Image 対応（第49章）に向けて、`@AOT` 関連のプロセッサを増やしています。
**「ランタイム偏重」から「コンパイル時シフト**」が、ここ数年の Java フレームワークの大きな流れです。

---

## 「**動かない**アノテーション**」 ―― ハマる場面

アノテーションを書いて、`@Retention(RetentionPolicy.RUNTIME)` を指定したのに、**リフレクションで読めない**――そういう事故があります。

原因はだいたいこのいずれかです:

1. **`@Retention` を忘れた** ―― デフォルトは `CLASS` で、実行時には消えている
2. **JPMS の `opens` がない** ―― リフレクションが弾かれている
3. **クラスがロードされていない** ―― そもそも対象クラスが探索範囲に入っていない
4. **コンパイル時処理用のはずだったのに**ランタイム処理で読もうとしている

特に 1 と 4 は、コードを書いた直後によく起きる勘違いです。
「**読めない**」と感じたら、まず `@Retention` を確認。
これだけで、半分は解決します。

---

## まとめると

- アノテーション活用は **ランタイム** と **コンパイル時** の 2 つの世界
- **ランタイム** ―― 柔軟、シンプル、ただし**コスト**がかかる
- **コンパイル時** ―― 高速、型安全、ただし**実装が難しい**
- 現代のフレームワークは**両者をハイブリッド**で使う
- **「ランタイム偏重から、コンパイル時シフトへ」**が現代の流れ
- 動かないと感じたら、まず `@Retention` を確認
