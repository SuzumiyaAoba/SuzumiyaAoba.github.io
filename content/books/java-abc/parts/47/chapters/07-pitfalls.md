## よくあるつまずき

アノテーションとアノテーション処理は、便利ですが、**ハマるポイント**もたくさんあります。
ここでは、現場でよく見る 6 つのつまずきを整理します。

---

## つまずき1: 「`@Retention` を忘れる」

第1節で触れたとおり、`@Retention` を書かないと**デフォルトは `CLASS`** です。
`RUNTIME` のつもりで書いたアノテーションが、**実行時に消えていて読めない**――これは新人エンジニアの定番事故です。

```java
@Target(ElementType.METHOD)
public @interface MyAnno { }    // ← @Retention 抜け

@MyAnno void hello() { }

// 別の場所で
method.getAnnotation(MyAnno.class);   // ← null
```

カスタムアノテーションは、**`@Retention` を常に明示**する習慣を付けましょう。

---

## つまずき2: 「**IDE がプロセッサを認識しない**」

プロセッサを書いて Maven/Gradle で動かしたが、IDE（IntelliJ・Eclipse）が**生成されたクラスを赤線**にする、というのもよくあります。

```java
// 自動生成のはずだが、IDE はこのクラスを「存在しない」と判定
UserMapperImpl impl = new UserMapperImpl();
```

対策:

- **IntelliJ**: Settings → Build → Compiler → Annotation Processors → "Enable annotation processing" にチェック
- **Maven**: `maven-compiler-plugin` に `<annotationProcessorPaths>` を明示
- **Gradle**: `annotationProcessor "..."` 依存を追加

最初の設定の罠です。設定さえ通れば、以後は自動で動きます。

---

## つまずき3: 「**循環参照のプロセッサ**」

「**プロセッサ A が生成したコードに、プロセッサ A 自身がもう一度反応する**」――こんな事態が起きることがあります。

```text line-numbers=false
Round 1: source → プロセッサ A → 新ソース X 生成
Round 2: X を見て、また A が反応 → 新ソース Y 生成
Round 3: ...（無限ループ）
```

`process(...)` の戻り値が `true` のとき、「**このアノテーションは私が処理した、他のプロセッサに渡さなくていい**」という意味です。
無限ループを防ぐには、

- 自分の生成したコードには**自分のアノテーションを付けない**
- または、**生成したクラスに `@Generated`** を付けて、自分の処理対象から除外する

ような工夫が必要です。

---

## つまずき4: 「**`@SuppressWarnings("all")` を貼って警告を全部消す**」

警告がうざい、と全部消すと、

- **未チェックキャスト**で実行時 ClassCastException が出るのを見逃す
- **非推奨 API** を使っていて、次のバージョンで動かなくなる
- **デッドコード**が増殖する

ことになります。
警告は**ノイズ**でなく、**サイン**です。
1 つずつ理由を確かめて、必要なら**ピンポイントで**抑止します。

---

## つまずき5: 「**プロセッサのデバッグが難しい**」

`javac` の中で動くプロセッサは、デバッガからは**普通には**追えません。
コツとして、

- **`Messager.printMessage(Diagnostic.Kind.NOTE, ...)`** で**ログ出力**を仕込む
- **`javac` の VM オプション**でデバッグポートを開けて、IDE からリモートデバッグ
- **小さなテストプロジェクト**でプロセッサだけ動かす

これだけでも、開発が**ぐっと楽**になります。

---

## つまずき6: 「**コンパイルが遅くなる**」

プロセッサが増えると、コンパイル時間が伸びます。
特に Lombok と他のプロセッサ（MapStruct など）を併用すると、**処理順序**で問題が起きることもあります。

対策:

- **使うプロセッサを絞る**
- ビルドを**並列化**する（Gradle の `org.gradle.parallel=true` など）
- **インクリメンタルコンパイル**を活用（Gradle はサポート、Maven は限定的）
- 不要になったプロセッサは**消す**

「**毎日のビルド時間 1 秒**は、年間で大きな差**」 ―― ビルドのチューニングは投資です。

---

## まとめると ―― 5 つの心得

アノテーション処理を扱う心得を 5 つに集約します。

1. **`@Retention` を必ず明示**する
2. **IDE のプロセッサ設定**を初期に整える
3. **生成コードは `@Generated` でマーク**して、自分の処理対象から除外
4. **警告を抑止する前に、理由を確かめる**
5. **プロセッサの数とコンパイル時間**を意識する

アノテーション処理は、Java の**メタプログラミング**を支える地味な、しかし強力な仕組みです。
ふだんは使う側でいい ―― ですが、いざ困ったときに「**裏で何が動いているか**」が分かる ―― それが、本書の目指したゴールです。

次の節「用語集」で本章の言葉を整理し、第48章では、性能調査の主役 ―― **JFR とプロファイリング**を扱います。
