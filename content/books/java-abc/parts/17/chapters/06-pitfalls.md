## よくあるつまずき

レコードで、初心者がつまずきやすいポイントを、まとめて確認します。

---

## 1. レコードの値を変えようとする

レコードは**不変**です。一度作ったら、値は変えられません。
セッターもなく、コンポーネントは `final` なので、書き換えようとしてもできません。

```java
Point p = new Point(3, 4);
p = new Point(10, 4);   // ◯ 新しいレコードを作って、入れ直すならOK
```

「`x` だけ変えたい」ときは、**変えた結果を持つ新しいレコードを作ります**。
レコードを「途中で値が変わるデータ」に使おうとすると、この不変性が壁になります。そういうデータには、ふつうのクラスを使いましょう。

---

## 2. アクセサは getX() ではなく x()

レコードの値を取り出すアクセサは、コンポーネント名そのまま（`x()`）です。
これまでのクラスの感覚で `getX()` と書くと、エラーになります。

```java
Point p = new Point(3, 4);
p.getX();   // ✕ そんなメソッドはない
p.x();      // ◯ アクセサは x()
```

```text
エラー: シンボルを見つけられません
  シンボル:   メソッド getX()
```

「レコードのアクセサは、`get` が付かない」と覚えておきましょう。

---

## 3. レコードはクラスを継承できない／継承もされない

レコードには、継承に関する制約があります。

- レコードは、**ほかのクラスを継承できない**（`record Foo extends Bar` は書けない）
- レコードは、**ほかのクラスから継承されることもない**（レコードは自動的に `final`）

```java
record Point(int x, int y) extends Object {}   // ✕ extends は書けない
```

これは、レコードが「データを束ねる、シンプルで安全な型」であるための制約です。
継承による複雑さを持ち込まないことで、不変性などの保証を、確実なものにしています。
（インターフェースの**実装**（`implements`）は、前節で見たとおり、できます。）

「クラスを継承する設計が必要なら、レコードではなく、ふつうのクラスを使う」と判断しましょう。

---

## 4. コンポーネントが「変わるもの」だと、不変が崩れる

第13章のつまずきでも触れた、参照型の注意点です。
レコードのコンポーネントが、配列やリストのような「**中身が変わりうるもの**」だと、レコード自体は不変でも、**その中身は変えられてしまいます**。

```java
record Team(int[] scores) {}

public class Main {
    public static void main(String[] args) {
        int[] data = {10, 20, 30};
        Team team = new Team(data);

        team.scores()[0] = 999;   // 配列の中身を書き換えられてしまう
        IO.println(team.scores()[0]);
    }
}
```

```text
$ java Main.java
999
```

`team` の `scores` という「配列を指す参照」は変えられませんが、その配列の**中身**は書き換えられてしまいました。
「本当に中身まで変わらない」レコードにしたいなら、コンポーネントには、`int`・`String`・ほかのレコードなど、**それ自体が不変なもの**を使うのが基本です。
（配列やリストを安全に扱う方法は、より進んだトピックなので、いまは「変わりうるものをコンポーネントにすると、不変が崩れることがある」と知っておけば十分です。）

---

## まとめ

- レコードは**不変**。値を変えるには、新しいレコードを作る
- アクセサは **`x()`**（`getX()` ではない）
- レコードは**クラスを継承できず、継承もされない**（インターフェースの実装はできる）
- コンポーネントに**変わりうるもの**（配列など）を使うと、不変性が崩れることがある

次は、この章で学んだ言葉を、用語集としてまとめます。
