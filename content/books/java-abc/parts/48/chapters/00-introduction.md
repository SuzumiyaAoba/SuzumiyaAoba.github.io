## はじめに ― この章で学ぶこと

第41章から第47章まで、私たちは「**Java はどう動いているか**」を学んできました。
JVM、GC、メモリモデル、並行処理、モジュール、リフレクション、アノテーション処理 ―― これだけ知ると、**自分のコードがなぜ遅いのか**、**どこに時間がかかっているのか**を、ある程度推測できるようになります。

ただ、推測は推測です。
本物の解は、**測ってみる**ことの中にあります。

第48章では、**測る**ためのツール ―― **プロファイリング**を扱います。
主役は、JDK に**標準で**入っている強力なツール、**JFR**（Java Flight Recorder）と **JMC**（Java Mission Control）です。

---

## 「測らずに最適化する」のは罠

ソフトウェア工学には、**ドナルド・クヌース**の有名な格言があります。

> 早すぎる最適化は、諸悪の根源である。

意味するところは、

- 「なんとなく遅そう」で書いたチューニングは、**実はボトルネックでない**場所を直していることが多い
- それでいて、**コードが複雑**になり、保守性が下がる
- 直すべき場所は、**測って初めて見える**

ということです。
プロファイリングは、**この罠を避ける**唯一の手段です。

---

## プロファイリングで分かること

プロファイラがあると、次のような問いに**実測で**答えられるようになります。

| 問い | プロファイラで分かること |
|---|---|
| どのメソッドに**時間がかかっている**か? | CPU プロファイル（メソッド別の累計時間） |
| なぜ**たまに遅い**のか? | スレッドの状態（待ち・ロック・I/O） |
| GC が**どれくらい止めている**か? | GC イベントの累計時間 |
| どこで**メモリを大量割り当て**しているか? | アロケーションプロファイル |
| **どのスレッド**が動いているか? | スレッドダンプ・タイムライン |
| 例外を**何回投げているか** | 例外イベントの数 |

ふつうのログでは見えない情報が、**プロファイラを通すと見えるようになります**。

---

## JFR ―― 「常に取れる」プロファイラ

JFR の最大の特徴は、**本番運用中でも取り続けられる**ほど軽量なことです。
一般的なプロファイラは「**JVM のフックを大量に貼って遅くする**」ものですが、JFR は**JVM 内部のイベントを**直接記録するため、

- **オーバーヘッドが極めて小さい**（CPU 数 % 程度）
- **本番でも**有効化できる
- **常時取得**して、問題発生時に過去 N 分を保存する運用も可能

第41章の「JIT・GC・クラスローディング」の各種イベントが、JFR の中で**最初から**記録対象になっています。
これを使わずに「**詳しい挙動が分からない**」と言うのはもったいない、というレベルの強力さです。

---

## この章で扱う範囲

JFR は奥が深いですが、本書では「**何ができるか・どう取るか**」を中心に扱います。
具体的な分析手法（フレームグラフの読み方、GC ログとの相関分析など）は、本格的な性能チューニングの本に譲り、本書はその**入り口**を提供します。

## この章を読み終えると

第48章を読み終えるころには、次のことができるようになっています。

- プロファイリングの**目的と種類**を区別できる
- JFR を**起動時・実行中**の両方で取れる
- JFR 出力（`.jfr`）の中身を、`jfr summary` で覗ける
- JMC や `jfr` CLI で、**CPU・メモリ・GC・スレッド**の傾向を読める
- **async-profiler** を選ぶべき場面を判断できる
- **早すぎる最適化**を避ける姿勢を、技術として身につける

それでは、最初の節「**プロファイリングとは**」から始めましょう。
