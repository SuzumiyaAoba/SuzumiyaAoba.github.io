## プロファイリングとは

「**プロファイリング**」（profiling）は、プログラムを実際に動かして、**何にどれくらいの時間を使ったか**を計測することです。
この節では、プロファイリングの**種類**と、ベンチマークとの違いを整理します。

---

## プロファイリングとベンチマークの違い

似たような言葉に「**ベンチマーク**」（benchmark）があります。
両者は目的が違います。

| | プロファイリング | ベンチマーク |
|---|---|---|
| 目的 | **どこに**時間がかかっているか | **どれだけ**時間がかかっているか |
| 結果 | メソッド別の累計時間、スレッド状態など | スループット (req/s)、レイテンシ (ms) |
| 出力 | プロファイラの分析画面 | 数値（平均・分布） |
| 使う場面 | **ボトルネック**を見つける | 改善前後の**比較** |

ベンチマークが「**速度メーター**」なら、プロファイリングは「**地図**」です。
両者は補完的で、どちらも欠かせません。

---

## プロファイリングの主要な種類

何を測るか、で大きく分けると 4 種類あります。

### 1. CPU プロファイリング

「**どのメソッドが**多く CPU を使っているか?」 ―― 最も基本的で、最も使われる種類です。

実装には大きく 2 つあります。

| 方式 | 内容 |
|---|---|
| **サンプリング** | 一定間隔（10 ms ごとなど）でスレッドの**スタックを記録** |
| **インストルメンテーション** | メソッドの**入口・出口**にコードを差し込んで時間計測 |

サンプリングは軽量で本番でも使えますが、**統計的**な精度になります。
インストルメンテーションは正確ですが、**重い**ので開発時のみに向きます。

JFR は**サンプリング**を使うため、本番でも回せます。

### 2. メモリプロファイリング（アロケーションプロファイリング）

「**どこで**多くオブジェクトを作っているか?」 ―― GC の原因を追うときに使います。

たとえば、ループの中で `String + String` を繰り返している箇所が、毎秒数 GB のオブジェクトを作っている、というのが見えます。
このタイプの問題は、**ヒープダンプ**で見るより、**割り当てプロファイル**で見るほうが解決が早いことが多いです。

### 3. ロック・スレッドプロファイリング

「**どのロックで**スレッドが待っているか?」 ―― 並行処理のボトルネックを探すときに使います。

第43章で見た `synchronized` ロックや `ReentrantLock` の待ち時間が見えると、

- ロックの**取りすぎ**
- ロック内で**I/O を待ってしまっている**箇所

などが見つかります。

### 4. I/O プロファイリング

「**どのファイル・ソケット**が、どれだけ時間を食っているか?」 ―― DB クエリやネットワーク呼び出しの遅さを追跡します。

特に Web アプリでは、CPU でもメモリでもなく**外部 I/O**がボトルネック、というケースがほとんどです。
I/O プロファイルがないと、何時間眺めても原因にたどり着けません。

## プロファイリングの「**いつ・どこで・どうやって**」

実際にプロファイリングを始めるとき、3 つの判断軸があります。

### いつ ―― 開発時 / ステージング / 本番

- **開発時**: コードを書きながら、軽く取って見る。重いプロファイラもアリ
- **ステージング**: 本番に近い負荷で計測。これが**いちばん効く**
- **本番**: 軽量なプロファイラ（JFR）を常時 ON にして、問題発生時にデータを保存

ステージング環境でしっかり負荷をかけて、本番と同じ JVM オプションで動かす ―― これが、**現実的に最も効率の良い**プロファイリングです。

### どこで ―― ローカル / リモート

- **ローカル**: ノート PC で動かして、ローカルプロセスに繋ぐ
- **リモート**: 本番サーバーから JFR を取って、ローカルで解析

リモートで `jstack`、`jcmd`、JFR を使う方法を覚えておくと、**本番トラブルの初動**が圧倒的に速くなります。

### どうやって ―― サンプリング頻度・対象

- **頻度**: 高いほうが精度は上がるが、オーバーヘッドも増える
- **対象**: 全スレッド / 特定スレッドのみ
- **時間**: 短時間 / 数時間

たとえば「**この 1 リクエストが遅い**」を追うなら、リクエスト処理時間だけ高頻度サンプリング。
「**1 日のうち深夜帯が遅い**」を追うなら、低頻度で 24 時間取り続ける。
目的に応じて、設定を調整します。

## 性能評価の鉄則

プロファイリングを習慣にするなら、**3 つの鉄則**を覚えておきましょう。

### 鉄則1: 「**まず動かして測る**」

「ここが遅いはず」という推測は、**当たらない**ことが多いです。
「**測ってから判断する**」を徹底すれば、的外れな最適化を避けられます。

### 鉄則2: 「**ウォームアップを忘れない**」

第41章で見たとおり、JIT のウォームアップで、起動直後と定常状態は速度が違います。
プロファイルを取るときは、**ウォームアップを終えてから**測ります。

ベンチマークも同じです。
JFR で「**`-XX:StartFlightRecording=delay=30s,duration=2m`**」のように、**30 秒待ってから 2 分間**取る、という指定が一般的です。

### 鉄則3: 「**1 回の計測で結論を出さない**」

同じプログラムを 2 回測ると、**5〜10 % 違う**ことはよくあります。
GC のタイミング、JIT の最適化、OS の負荷状態 ―― 揺らぐ要因がたくさんあります。

最低**3 回**は測って、ばらつきを確認しましょう。
それでも判断が割れるときは、ウォームアップ時間・サンプル数を増やします。

## プロファイリングと**観測コスト**

「プロファイラを動かすこと自体が、計測対象を変える」というジレンマがあります。

| プロファイラ | オーバーヘッド | 用途 |
|---|---|---|
| **JFR**（既定設定） | < 1 % | **本番でも常時 OK** |
| **JFR**（profile.jfc） | 1〜3 % | 本番でも OK |
| **async-profiler** | 1〜2 % | 本番でも OK |
| 古典的なサンプリング（JVisualVM） | 数 % | 開発時・ステージング |
| インストルメンテーション系 | 10〜100 % | 開発時のみ |

JFR と async-profiler は、いずれも本番で使える設計です。
迷ったら、まず**JFR**から ―― これが現代の Java の答えです。

---

## まとめると

- プロファイリングは「**どこに**時間がかかっているか」を測る
- 種類は **CPU・メモリ・ロック・I/O** の 4 つ
- 開発・ステージング・本番、それぞれに合った頻度で計測
- 鉄則: **測ってから判断・ウォームアップ・複数回計測**
- JFR と async-profiler は、いずれも**本番投入できる軽量**プロファイラ
