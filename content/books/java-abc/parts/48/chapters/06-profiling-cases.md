## プロファイリングの実例

ツールの使い方を覚えたら、次は「**実際の問題に、どうやって使うか**」です。
この節では、よく遭遇する 5 つの典型ケースを、プロファイリングの**流れ**として整理します。

---

## ケース1: CPU が常時 100% に張り付く

### 症状

- `top` で CPU 使用率が 100 % 近く
- レスポンスが遅い
- アクセスが少ないときも CPU が落ちない

### 流れ

1. **JFR を 60 秒取る**
2. JMC で **Method Profiling** を開く
3. 上位のメソッドを見る

よく見つかる原因:

- **無限ループ** ―― `while(true)` 内で `Thread.sleep` 忘れ
- **`HashMap` の衝突** ―― 高速計算ループの中で hash 衝突を引いている
- **ロギングのフォーマット** ―― 大量の文字列連結でCPUを食う
- **正規表現** ―― `Pattern.matches(...)` を毎回コンパイル

### 解決

- ループに `sleep` を入れる
- `String + String + ...` を `StringBuilder` に
- `Pattern` をフィールドで使い回す
- ロギングを `if (log.isDebugEnabled())` でガード

---

## ケース2: GC が頻発する

### 症状

- ユーザーから「**たまに止まる**」と報告
- レスポンスの 99 パーセンタイルが**飛び抜けて**遅い
- 平均は速いのに、一部リクエストが極端に遅い

### 流れ

1. **GC ログを取る**（`-Xlog:gc`）
2. JFR の **Garbage Collection** イベントを確認
3. JMC の **Memory → TLAB Allocations** で割り当て元を特定

よく見つかる原因:

- **無駄な中間オブジェクト**: ストリームを多用しすぎ、ループ中のデータコピー
- **JSON シリアライズ・デシリアライズ**: 1 リクエストごとに大量の中間オブジェクト
- **`String.format` の濫用**: 毎回新しい `String` を作る
- **オートボクシング**: `Integer` の `List` を扱うコードなど

### 解決

- ホットパスでオートボクシングを避ける（`int[]` を使うなど）
- 文字列フォーマットを `StringBuilder` に
- JSON マッパーで**ストリーム API** を使う
- ヒープサイズを上げる（最後の手段）

---

## ケース3: 一部のリクエストだけ遅い

### 症状

- 99 % のリクエストは 10 ms
- ですが、たまに 1 秒を超える
- 共通する条件が見つからない

### 流れ

1. **連続記録 JFR** を ON
2. 問題のリクエストが来たら、その時刻の `JFR.dump`
3. JMC で**スレッドタイムライン**を開いて、その時刻のスレッド状態を見る

よく見つかる原因:

- **GC stop-the-world**: ちょうどそのリクエストの最中に GC
- **ロック競合**: 別の重い処理が、共有ロックを長時間握っていた
- **DB クエリ**: そのリクエストだけ、N+1 問題で何千クエリ
- **キャッシュミス**: 偶然そのリクエストのキャッシュが失効
- **外部 API**: 下流サービスがそのときたまたま遅延

### 解決

- GC のチューニング、または**ZGC** への切り替え
- 重い処理を専用スレッドに分離
- N+1 を JPA の `@EntityGraph` などで解消
- キャッシュの**TTL**と**再生成**戦略を見直す
- 外部 API に**タイムアウト**と**サーキットブレーカ**

---

## ケース4: 起動が遅い

### 症状

- アプリの起動に**数十秒**かかる
- Spring Boot で「Started in 30s」と出る

### 流れ

1. **起動から JFR を取る**（`-XX:StartFlightRecording=...`）
2. JMC で **Code → Class Loading** を確認
3. **Initialization Time** を見て、何の初期化が長いか

よく見つかる原因:

- **クラスパスが膨大**: スキャン対象が多すぎる
- **コンポーネントスキャン**: Spring が大量のクラスをリフレクションで初期化
- **大量の `static` 初期化**
- **接続プールの初期化**: DB やキャッシュへの接続を直列に作っている

### 解決

- **`@Lazy`** で遅延初期化（Spring）
- 接続プールの初期化を**並行**に
- 不要な`@Component` を絞る
- **GraalVM ネイティブイメージ**（第49章）を検討

---

## ケース5: メモリリーク

### 症状

- ヒープ使用量が**右肩上がり**で増え続ける
- 何時間か後に `OutOfMemoryError`
- アプリを再起動すれば直る

### 流れ

1. **ヒープダンプを取る** （`jcmd <pid> GC.heap_dump filename=heap.hprof`）
2. **Eclipse MAT**（Memory Analyzer Tool）で開く
3. **Dominator Tree** で、ヒープを占有する**根のオブジェクト**を特定
4. その**保持パス**（GC ルートからの参照経路）を見る

よく見つかる原因:

- **`HashMap` キャッシュに入れっぱなし**
- **リスナの remove 忘れ**
- **`ThreadLocal` の `remove` 忘れ**
- **接続プールの leak**: コネクションを返却し忘れ

### 解決

- キャッシュに**TTL**と**上限**を入れる（Caffeine など）
- リスナは `removeListener` で解除
- `ThreadLocal` は `try-finally` で `remove`
- コネクションは **`try-with-resources`**

---

## ケース全般の進め方

5 つのケースに共通する進め方は、こうです。

1. **症状を観察**: いつ、どの程度遅いか、再現性は?
2. **計測**: ログ、JFR、async-profiler でデータを集める
3. **分析**: ボトルネックを特定（CPU? メモリ? I/O? ロック?）
4. **仮説**: 「**これが原因では?**」を立てる
5. **修正**: 仮説に基づいてコードを変える
6. **再計測**: 改善されたか、別のところで悪化していないか

このループを、**勘ではなくデータで**回す。
それが「**根拠のあるパフォーマンスチューニング**」です。

---

## まとめると

- **5 つの典型ケース**: CPU 高負荷・GC 頻発・レスポンスばらつき・起動遅延・メモリリーク
- それぞれ、**特定のツールと特定の確認手順**がある
- 共通する流れは「**症状 → 計測 → 分析 → 仮説 → 修正 → 再計測**」
- **勘ではなく、データで判断**するのが鉄則
