## よくあるつまずき

プロファイリングは「測れば分かる」と思いがちですが、**測り方と解釈**で間違えると、的外れな結論に至ります。
この節では、よくある 6 つのつまずきを整理します。

---

## つまずき1: 「ウォームアップなしで計測」

第41章・第48章の繰り返しになりますが、これは何度言っても足りません。

```text
# NG
$ java -XX:StartFlightRecording=duration=10s MyApp
```

起動直後の 10 秒は、**JIT がまだ仕事しきっていない**状態です。
ここで取ったプロファイルは、本来知りたい性能を**反映していません**。

```text
# OK
$ java -XX:StartFlightRecording=delay=30s,duration=2m MyApp
```

`delay=30s` でウォームアップを待つ ―― これを忘れない。

---

## つまずき2: 「**1 回の計測で結論**」

```text
$ java -XX:StartFlightRecording=duration=60s MyApp
# 「`Foo.process` が 30 % だった！直そう」
```

1 回の計測には、

- たまたまその時間に GC が走った
- 偶然そのリクエストパターンだった
- システム全体の負荷が高かった

など、再現性のない要因が混ざります。
**最低 3 回**、できれば**5 回**取って、傾向を確認しましょう。

---

## つまずき3: 「**プロファイラを動かして遅くなった**」

特に古いプロファイラ（インストルメンテーション系）では、**プロファイラ自身が遅さの原因**になります。

```text
# プロファイラ ON: 100 ms
# プロファイラ OFF: 10 ms
```

これでは、何を測っているのか分かりません。
JFR や async-profiler ならオーバーヘッドが小さいので、まず起こりませんが、**測定対象のリクエストが本番より遅すぎる**と感じたら、プロファイラを疑います。

---

## つまずき4: 「**CPU でも I/O でも、全部 CPU プロファイル**」

第5節で触れたとおり、

- CPU バウンドなら CPU プロファイル
- I/O バウンドなら**Socket I/O や File I/O のイベント**

を取るべきです。
**I/O 待ちが原因**なのに CPU プロファイルだけ取っても、**スレッドは何もしていない**ように見えるだけ。
原因にたどり着けません。

CPU プロファイルを取って「**何も CPU を食っていないのに遅い**」と感じたら、それが I/O バウンドの兆候です。
JFR で `Socket I/O` イベントを見るか、`jstack` でスレッドダンプを取って、**何待ち**かを確認します。

---

## つまずき5: 「**フレームグラフの**大きいブロック**を必ず直す**」

フレームグラフで上位に出るメソッドが、必ずしも**直すべき**メソッドとは限りません。

- **ライブラリの内部メソッド**: 直しようがない（or 別の使い方をすべき）
- **`Object.hashCode`** や `equals`: ホット = 呼ばれている、だけ
- **`System.arraycopy`**: 何かをコピーしているという**結果**であって原因ではない

「**上位のメソッドが何をしているか**」を理解し、その**呼び出し元**を辿る ―― これがフレームグラフの正しい読み方です。

---

## つまずき6: 「**実行環境が違う**」

ローカルの Mac で取ったプロファイルを、本番の Linux サーバーの根拠にすると、

- CPU アーキテクチャが違う（x86 vs ARM）
- メモリ量が違う
- I/O 性能が違う
- JVM バージョンが微妙に違う

など、**前提条件が大きく違う**ので、結論は当てになりません。

ステージング環境を本番に近く揃え、**そこで計測**するのが定石です。
ローカルで取れるのは「**こういう傾向がありそう**」までです。

---

## まとめると ―― プロファイリングの 5 つの心得

第48章の総まとめとして、プロファイリングを扱う 5 つの心得を:

1. **ウォームアップ後**に測る
2. **複数回**測って、傾向を確認する
3. **CPU か I/O か**を最初に判別する
4. **フレームグラフは原因ではなく結果**: 呼び出し元を辿る
5. **本番に近い環境**で測る

プロファイリングは、Java 開発者の**最強の武器**です。
ふだんから「**測ってから判断する**」習慣を付ければ、的外れな最適化を減らし、本物の改善に集中できます。

次の節「用語集」で本章の言葉を整理し、第49章では、Java を**起動からネイティブ並みに速く**する仕組み ―― **GraalVM ネイティブイメージ**を扱います。
