## カプセル化とは ― なぜデータを隠すのか

この節では、**カプセル化**（Encapsulation）が、なぜ必要なのかを、具体的な例で考えます。
仕組み（書き方）に入る前に、まず「動機」を理解しておきましょう。

---

## 問題: フィールドが丸見えだと、何でもできてしまう

第11章のやり方で、銀行口座を表す `BankAccount` クラスを作ってみます。
口座には「残高（balance）」があり、入金・出金ができる、と考えます。

```java
class BankAccount {
    int balance;   // 残高
}
```

フィールドが外から見えるので、残高を自由に読み書きできます。

```java
BankAccount acc = new BankAccount();
acc.balance = 1000;   // 1000円 入金（のつもり）
acc.balance = -5000;  // ← マイナスの残高にできてしまう！
```

ここに、大きな問題があります。
**残高がマイナス**という、現実にはありえない状態を、誰でも簡単に作れてしまうのです。

しかも、`acc.balance = -5000;` のような誤った操作は、コードのどこからでも書けます。
プログラムが大きくなると、「いったい、どこで残高がおかしくされたのか」を探すのが、とても大変になります。

---

## カプセル化の発想 ― 「直接さわらせない」

そこで、発想を変えます。

> 残高（`balance`）には、**外から直接さわれないようにする**。
> そのかわり、「入金する」「出金する」という**決められたメソッドを通してのみ**、残高を変えられるようにする。

こうすれば、残高を変える「入口」を、メソッドに限定できます。
そして、そのメソッドの中で「マイナスにはしない」「出金は残高の範囲内だけ」といった**ルールをチェック**すれば、口座はつねに正しい状態に保たれます。

```java
class BankAccount {
    private int balance;   // 外から直接さわれない（private）

    void deposit(int amount) {     // 入金は、このメソッド経由でだけ
        if (amount > 0) {
            balance += amount;
        }
    }
}
```

`private` という印を付けると、`balance` は**そのクラスの中からしか**さわれなくなります（くわしくは次の節）。
外からは `acc.balance = -5000;` と書けなくなり、入金は `acc.deposit(1000)` のように、メソッドを通すしかなくなります。
入口を1か所にしぼることで、そこに**見張り（チェック）**を置けるようになるのです。

---

## カプセル化の3つの利点

データを隠し、メソッドを入口にすると、次のような利点があります。

### 1. データを正しい状態に保てる（不正な値を防ぐ）

メソッドの中でチェックすれば、「マイナスの残高」「マイナスの速度」のような、ありえない値を防げます。
オブジェクト自身が、自分のルールを守れるようになります。

### 2. 変更に強くなる（内部を自由に変えられる）

外からは「メソッド（入口）」しか見えないので、**内部のフィールドの持ち方は、あとから自由に変えられます**。
たとえば「残高を `int` から、より大きな数を扱える型に変える」としても、メソッドの使い方さえ同じなら、そのクラスを使う側のコードは1行も直さずに済みます。
内部の都合と、外から見た使い方を、切り離せるのです。

### 3. 使い方が分かりやすくなる

使う人は、「`deposit`（入金）」「`withdraw`（出金）」といった**メソッドの名前**を見れば、何ができるかが分かります。
フィールドの細かい中身を知らなくても、決められた入口を使うだけで、正しく扱えます。

---

## 「公開するもの」と「隠すもの」を分ける

カプセル化の本質は、クラスを2つの部分に分けて考えることです。

| | 役割 | ふつうの扱い |
|---|---|---|
| **隠す部分**（内部の実装） | データの持ち方など、内部の都合 | フィールドは `private` に |
| **見せる部分**（公開する窓口） | 外から使ってよい操作 | メソッドは `public` に |

「外に見せる窓口（メソッド）」を **インターフェース**（Interface、接点）と呼ぶこともあります。
（同じ言葉が、第16章で学ぶ Java の `interface` という機能の名前にもなっていますが、ここでは「外から使うための窓口」という一般的な意味です。）

家電にたとえると、わかりやすいかもしれません。
電子レンジの内部の配線（実装）は、ふたの中に隠されています。私たちが使うのは、外に出ているボタン（公開された窓口）だけです。
内部を直接いじらなくても、ボタンを押すだけで安全に使える ―― これがカプセル化のイメージです。

---

## まとめ

- フィールドが外から丸見えだと、**ありえない値**を誰でも入れられてしまう
- **カプセル化**は、データを**隠し**、決められた**メソッドを入口**にして操作させる考え方
- 利点は、(1) データを正しく保てる、(2) 内部を自由に変えられる（変更に強い）、(3) 使い方が分かりやすい
- クラスを「隠す部分（内部の実装）」と「見せる部分（公開する窓口）」に分けて考える

次の節では、この「隠す・見せる」を実現する **アクセス修飾子**（`public`・`private` など）を学びます。
