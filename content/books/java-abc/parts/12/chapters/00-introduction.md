## はじめに ― この章で学ぶこと

第11章では、クラスとオブジェクトの基本を学びました。
フィールドにデータを持たせ、メソッドでふるまいを与え、`new` でオブジェクトを作って操作する ―― オブジェクト指向の土台ができました。

ですが、第11章のやり方には、1つ大きな弱点がありました。
それは、**フィールドを、外から誰でも自由に書き換えられてしまう**ことです。

```java
Car c = new Car();
c.speed = -100;   // マイナスの速度！？ でも、止められない
```

`speed` に `-100` という、ありえない値を入れても、Java は何も文句を言いません。
これでは、せっかくオブジェクトにまとめても、データが**おかしな状態**になるのを防げません。

この第12章では、この問題を解決する **カプセル化**（Encapsulation）という考え方を学びます。
オブジェクトを「**安全に、正しく使える部品**」に育てるための、とても大切な章です。

---

## カプセル化 ― 中身を守り、入口を決める

**カプセル化**とは、ひとことで言えば、

> オブジェクトの**データ（フィールド）を外から直接いじれないように隠し**、決められた**メソッド（入口）を通してのみ**操作できるようにすること

です。

薬のカプセルを思い浮かべてください。
中身（薬の成分）はカプセルに包まれていて、直接は触れません。私たちは「カプセルを飲む」という決められた使い方をするだけです。

オブジェクトも同じように、

- 中身（フィールド）は**隠す**
- 使い方（メソッド）は**公開する**

とすることで、「**正しい使い方しかできない**」安全な部品になります。
たとえば「速度はマイナスにできない」「残高は直接書き換えられず、入金・出金でしか変わらない」といったルールを、オブジェクト自身に守らせられるのです。

## この章で学ぶこと

第12章は、次の7つの節で構成されています。

| 節 | タイトル | 内容 |
|---|---|---|
| 1 | カプセル化とは | なぜデータを隠すのか。その利点 |
| 2 | アクセス修飾子 | `public`・`private` など、公開範囲を決める印 |
| 3 | フィールドを private にする | データを隠して、直接アクセスを禁止する |
| 4 | ゲッターとセッター | 隠したデータを、安全に読み書きする入口 |
| 5 | パッケージと import | クラスを整理し、名前の衝突を防ぐしくみ |
| 6 | よくあるつまずき | public のしすぎ・セッターの誤解 など |

前半（1〜4節）が、この章の中心です。`private` でデータを隠し、`getter`／`setter` で安全な入口を作る、という王道の流れを学びます。
後半（5節）で、クラスを整理する「パッケージ」を学びます。
（一度作ったら変わらない「不変オブジェクト」という、さらに安全な設計もありますが、これはコンストラクタを学んでからのほうが分かりやすいので、第13章・第17章で扱います。）

---

## この章を読み終えると

第12章を読み終えるころには、次のことができるようになっています。

- カプセル化が「なぜ」必要なのかを説明できる
- `public` と `private` の違いがわかり、使い分けられる
- フィールドを `private` にして、ゲッター・セッターで安全に公開できる
- セッターで「ありえない値」をはじく、検証つきのオブジェクトを作れる
- パッケージと `import` の役割がわかる

それでは、最初の節「カプセル化とは」から始めましょう。
