## パッケージと import ― クラスを整理する

プログラムが大きくなると、クラスの数はどんどん増えていきます。
何十、何百というクラスを、1つの場所にごちゃ混ぜにしておくと、すぐに管理できなくなります。

そこで Java には、クラスを**グループに分けて整理する**しくみがあります。
それが **パッケージ**（Package）です。

---

## パッケージ ― クラスを入れる「フォルダ」

**パッケージ**は、関連するクラスをまとめる入れ物です。
コンピュータで、関連するファイルをフォルダにまとめて整理するのと、よく似ています。

たとえば、次のように分けます。

```text
shop.product   ← 商品に関するクラス（Product, Category …）
shop.order     ← 注文に関するクラス（Order, Cart …）
shop.user      ← 利用者に関するクラス（User, Address …）
```

このように分けておくと、「商品のことは `shop.product` を見ればよい」と、目的のクラスを探しやすくなります。
パッケージは `.`（ドット）で区切って、階層を表します（`shop.product` は「`shop` の中の `product`」）。

---

## パッケージを宣言する

クラスを特定のパッケージに入れるには、ファイルの**いちばん先頭**に、`package` 文を書きます。

```java
package shop.product;   // このクラスは shop.product パッケージに属する

public class Product {
    // ...
}
```

`package 〜;` は、必ずファイルの先頭（コメントを除いて1行目）に書く、というルールがあります。
そして、パッケージの名前は、ファイルを置く**フォルダの構成**と一致させます（`shop.product` なら `shop/product/` フォルダに置く）。

> **補足: パッケージ名の慣習 ― ドメインを逆さにする**
>
> 世界中の人が作ったクラスと名前がぶつからないように、パッケージ名には**自分のドメイン名を逆にしたもの**を使う慣習があります。
> たとえば `example.com` を持つ会社なら、`com.example.shop.product` のようにします。
> 個人の学習では、ここまで厳密にする必要はありませんが、「パッケージ名でぶつかりを防ぐ」という考え方は知っておきましょう。

---

## import ― 別のパッケージのクラスを使う

別のパッケージにあるクラスを使うには、本来、**パッケージ名を含めた完全な名前**で書く必要があります。

```java
java.util.Arrays.sort(data);   // java.util パッケージの Arrays クラス
```

この `java.util.Arrays` のような、パッケージ名を含む正式な名前を、**完全修飾名**（Fully Qualified Name）と呼びます。
しかし、毎回この長い名前を書くのは大変です。

そこで、ファイルの先頭で **`import`** を使って、「このクラスを使います」と宣言しておくと、以降は短い名前（`Arrays`）で書けるようになります。

```java
import java.util.Arrays;   // 最初に宣言しておく

// ...
Arrays.sort(data);          // 以降は短い名前で書ける
```

第8章で `import java.util.Arrays;` と書いたのは、まさにこれです。
`import` は「このクラスを、短い名前で使えるようにする」ための宣言だったのです。

---

## import しなくても使えるもの

ここで、いくつか疑問が浮かぶかもしれません。

「`String` や `IO`、`System` は、`import` しなくても使えていたのはなぜ？」

理由は2つあります。

1. **`java.lang` パッケージは、自動で import される**
   `String`・`System`・`Integer`・`Math`・`IO` などは、すべて **`java.lang`** という特別なパッケージにあります。
   このパッケージは、Java が**自動的に import** してくれるので、自分で書かなくても短い名前で使えます。
2. **同じパッケージのクラスは、import がいらない**
   自分が書いたクラスどうしが同じパッケージにあれば、`import` なしで、おたがいに使えます。

`import` が必要になるのは、「**`java.lang` 以外の、別のパッケージにあるクラス**」を使うときです。
`java.util.Arrays` がその例でした。

---

## パッケージとアクセス修飾子の関係

第2節で、アクセス修飾子の「何も付けない（パッケージプライベート）」は「**同じパッケージの中だけ**」アクセスできる、と説明しました。
これで、その意味がはっきりしたと思います。

- **`public`** … どのパッケージからでもアクセスできる
- **何も付けない（パッケージプライベート）** … 同じパッケージの中だけアクセスできる
- **`private`** … 同じクラスの中だけアクセスできる

つまり、パッケージは「クラスのグループ」であると同時に、「**アクセス制御の境界**」でもあるのです。
「このパッケージの中だけで使う、内部的なクラス」には `public` を付けず、「広く公開するクラス」には `public` を付ける、という使い分けができます。

> **補足: 本書のサンプルと、IDE での扱い**
>
> 本書のサンプルは、説明をシンプルにするため、`package` 宣言を省いています（この場合、「名前のないパッケージ」に属します）。
> 実際の開発では、IntelliJ IDEA などの IDE が、パッケージのフォルダ作りや `import` の追加を**自動で**やってくれます。
> 必要なクラスを使うと、IDE が「`import` を追加しますか？」と提案してくれるので、手で `import` を書くことはほとんどありません。
> ここでは「パッケージはクラスを整理するしくみ」「`import` は別パッケージのクラスを短い名前で使う宣言」という考え方を、つかんでおけば十分です。

---

## まとめ

- **パッケージ**は、関連するクラスをまとめて整理するしくみ（フォルダのようなもの）
- ファイル先頭の `package 名前;` で、所属パッケージを宣言する（名前は `.` 区切りの階層）
- 別パッケージのクラスは、本来**完全修飾名**（`java.util.Arrays`）で書くが、`import` で短い名前にできる
- **`java.lang`**（`String`・`System`・`IO` など）は自動 import。**同じパッケージ**内も import 不要
- パッケージは「アクセス制御の境界」でもある（パッケージプライベート＝同じパッケージ内だけ公開）
- 実際の開発では、IDE がパッケージや `import` を自動で管理してくれる
