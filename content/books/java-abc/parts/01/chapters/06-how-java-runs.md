## Java の動く仕組み ― コンパイルとバイトコード

ここまで何度か、「Java は一度書けば、どこでも動く」という特徴を紹介してきました。

では、なぜそんなことが可能なのでしょうか。

この節では、私たちが書いた Java のプログラムが、どのような道のりを経て実際に動くのかを見ていきます。
ここは少し技術的な内容ですが、Java の仕組みを理解するうえでとても大切な部分です。
ひとつずつ、ていねいに見ていきましょう。

---

## コンピュータは Java の言葉を直接は理解できない

前の節で、コンピュータの内部では、すべてが 0 と 1 で表されていると学びました。

CPU が本当に理解できるのも、0 と 1 で書かれた命令だけです。
この、CPU が直接理解できる言葉を、**機械語**（マシン語、Machine Code）と呼びます。

一方、私たちが書く Java のプログラムは、人間が読み書きしやすいように作られています。

```java
System.out.println("こんにちは");
```

このコードは人間には読めますが、CPU はこのままでは理解できません。
人間の言葉（Java）と、コンピュータの言葉（機械語）の間には、大きな隔たりがあるのです。

そこで、両者をつなぐための「変換」が必要になります。

---

## ソースコードからバイトコードへ ― コンパイル

私たちが書いた Java のプログラム（ソースコード）は、まず**コンパイル**（Compile）という処理を通ります。

コンパイルとは、人間が書いたソースコードを、コンピュータが扱いやすい別の形に変換することです。
この変換を行う道具を、**コンパイラ**（Compiler）と呼びます。

Java の場合、コンパイルによって作られるのは、機械語そのものではありません。
**バイトコード**（Bytecode）と呼ばれる、特別な形式に変換されます[^jvms-class-file]。

ファイルの形で見ると、次のように変わります。

```text
Hello.java   （人間が書いたソースコード）
     │
     │  コンパイル
     ▼
Hello.class  （バイトコード）
```

- `.java` ファイル ： 私たちが書く、人間向けのソースコード
- `.class` ファイル ： コンパイルによって作られる、バイトコード

---

## バイトコードとは何か

ここで重要な疑問が浮かびます。

なぜ、CPU が直接理解できる機械語にせず、バイトコードという中間的な形にするのでしょうか。

その理由こそが、「一度書けば、どこでも動く」という特徴の秘密です。

機械語は、CPU やコンピュータの種類によって少しずつ異なります。
もしソースコードを機械語に直接変換してしまうと、その機械語は特定の種類のコンピュータでしか動きません。
別の種類のコンピュータで動かすには、作り直す必要が出てきてしまいます。

そこで Java は、特定のコンピュータに依存しない、**共通の中間的な言葉**としてバイトコードを用意しました。

バイトコードは、たとえるなら**世界共通の中間言語**のようなものです。

- 人間が書いたソースコード（日本語の指示書のようなもの）を、
- いったん、世界中どこでも通じる共通語（バイトコード）に翻訳しておく

こうしておけば、あとは「その共通語を、それぞれの土地の言葉に通訳する人」さえいれば、どこでも同じ指示を実行できます。

その「通訳する人」の役割を果たすのが、次に説明する **JVM** です。

---

## JVM がバイトコードを実行する

バイトコードを受け取って、実際に動かす仕組みが、**JVM**（Java Virtual Machine、Java 仮想マシン）です[^jvms-introduction]。

「仮想マシン」という言葉は少し難しく聞こえますが、ここでは**バイトコードを実行するための、ソフトウェアでできた専用の機械**だと考えてください。

JVM は、コンピュータの種類ごとに、それぞれ専用のものが用意されています。

- Windows 用の JVM
- Mac 用の JVM
- Linux 用の JVM

それぞれの JVM は、共通のバイトコードを受け取り、それを自分が動いているコンピュータの機械語に変換しながら実行します。

ここがポイントです。

![同じ Hello.class（バイトコード）を、Windows・Mac・Linux それぞれの JVM が受け取り、その環境の機械語に変換して実行する。](./images/write-once-run-anywhere.svg)

つまり、私たちは1つのバイトコードを用意するだけで、それぞれの環境に合わせた変換は JVM が引き受けてくれます。
これが、「一度書けば、どこでも動く」を実現している仕組みです。

先ほどの通訳のたとえで言えば、各環境の JVM が、共通語（バイトコード）をその土地の言葉（機械語）に通訳してくれる通訳者なのです。

## 全体の流れを整理する

ここまでの流れを、ひとつの図にまとめてみましょう。

![コンパイルから実行までの流れ。Hello.java を①コンパイルして Hello.class（バイトコード）にし、②JVM がその環境の機械語に変換しながら実行する。](./images/compile-and-run.svg)

大きく分けると、次の2つの段階があります。

1. **コンパイル** ： ソースコード（`.java`）をバイトコード（`.class`）に変換する
2. **実行** ： JVM がバイトコードを読み込み、その環境に合わせて動かす

この2段構えこそが、Java の大きな特徴です。

---

## JVM は賢く実行する ― インタプリタと JIT

JVM がバイトコードを実行する方法には、もう少し工夫があります。

JVM は、最初はバイトコードを1つずつ読み取りながら実行していきます。
このように、命令を1つずつ解釈しながら動かす方式を、**インタプリタ**（Interpreter、解釈実行）方式と呼びます。

インタプリタ方式はすぐに実行を始められる手軽さがある一方で、毎回バイトコードを解釈しながら動くため、速度の面では不利になりがちです。

そこで JVM は、プログラムの動きを観察し、**何度も繰り返し使われる部分**を見つけ出します。
そして、そうした「よく使われる部分」を、あらかじめ機械語に変換してから実行するようにします。
こうすることで、2回目以降は高速に動かせるようになります。

この、実行しながら必要な部分を機械語に変換する仕組みを、**JIT コンパイラ**（Just-In-Time Compiler、実行時コンパイラ）と呼びます[^hotspot-jit]。

```text
インタプリタ ： バイトコードを1つずつ解釈して実行する（手軽だが遅め）
JITコンパイラ ： よく使う部分を機械語に変換して、高速に実行する
```

JVM は、この2つをうまく組み合わせています。
プログラムを開始してすぐはインタプリタで実行し、よく使われる部分を見つけたら JIT コンパイラで高速化していくのです。

このしくみのおかげで、Java は「どこでも動く」という柔軟さと、「実用的な速さ」の両方を実現しています。

> **補足: いまは深く覚えなくて大丈夫**
>
> インタプリタや JIT コンパイラの細かい仕組みは、いまの段階では覚えなくてかまいません。
> 「JVM は、ただ実行するだけでなく、速くなるように工夫してくれている」とだけ知っておけば十分です。

## メモリの後片付けも JVM の仕事

前の節で学んだ**ガベージコレクション**（使い終わったメモリの自動的な後片付け）も、この JVM が行ってくれる仕事のひとつです。

JVM は、単にバイトコードを実行するだけではありません。
プログラムが使うメモリを管理し、不要になったものを片付けながら、プログラムが安定して動くように裏側で支えています。

私たちプログラマーが、メモリ管理や環境ごとの違いをあまり気にせずにすむのは、この JVM がさまざまな面倒を引き受けてくれているおかげなのです。

---

## まとめ

この節では、Java が動く仕組みを学びました。

- CPU が直接理解できるのは、0 と 1 で書かれた**機械語**だけである
- 私たちが書いた Java のソースコード（`.java`）は、**コンパイル**によって**バイトコード**（`.class`）に変換される
- バイトコードは、特定の環境に依存しない、共通の中間的な言葉である
- **JVM**（Java 仮想マシン）が、バイトコードを読み込み、動いている環境の機械語に変換しながら実行する
- 環境ごとに専用の JVM が用意されているため、同じバイトコードがどこでも動く ―― これが「一度書けば、どこでも動く」の仕組み
- JVM は、**インタプリタ**と **JIT コンパイラ**を組み合わせ、柔軟さと速さを両立している
- ガベージコレクションなどのメモリ管理も、JVM が引き受けてくれている

「コンパイルしてバイトコードを作り、JVM が実行する」という2段構えが、Java を支える中心的な仕組みです。

ここまで、JVM という言葉が何度も登場しました。
次の節では、この JVM と、よく似た言葉である **JRE**・**JDK** の違いを整理し、Java で開発を始めるために何が必要なのかをはっきりさせます。
