## はじめに ― この章で学ぶこと

第20章で、例外の**書き方**を学びました。
そして、これまでの章で、

- ファイル入出力での `IOException`（第30章）
- データベースアクセスでの `Optional<T>` と `BookNotFoundException`（第37章）
- バリデーション失敗時の `MethodArgumentNotValidException`（第38章）
- REST API での `@RestControllerAdvice`（第38章）

など、**いろんな場面の例外**に出会ってきました。

第39章では、こうした個別の経験を**ひとつの設計の話**としてまとめます。
業務システムを書くうえで、

> 「**どんな例外を、どこで投げ、どこで捕まえるべきか**」

という、エラーハンドリングの**戦略**を学んでいきます。

---

## なぜ「例外設計」が大事か

「動く」コードと「**運用できる**」コードの差は、エラー処理に表れます。

- 想定外のエラーで**プロセスが止まる**
- 利用者に**意味のない 500 エラー**が見えてしまう
- 例外のスタックトレースから**機密情報**が漏れる
- 「何が起きたか」が**ログから追えない**
- リトライしても安全か**分からない**

これらは、設計の段階で「**どう例外を扱うか**」を決めておかないと、後から直すのが大変です。
本章では、現代の Java で実用的な、**例外設計の指針**を整理します。

---

## この章で扱うこと・扱わないこと

扱うこと：

- チェック例外 / 非チェック例外の**使い分け**
- **ドメイン例外**の設計
- 早期失敗（Fail-Fast）と防御的プログラミング
- 例外と **`Optional`**・**Result 型**の選び方
- REST API での**エラーレスポンス設計**

扱わないこと：

- 細かい例外クラスの全列挙
- ライブラリごとの専用例外の解説

「ふつうの業務システムで、よくぶつかるパターン」だけを、現代的な視点で整理します。

---

## この章で学ぶこと

第39章は、次の8つの節で構成されています。

| 節 | タイトル | 内容 |
|---|---|---|
| 1 | チェック例外と非チェック例外 | いつ・どちらを使うか |
| 2 | ドメイン例外を設計する | ビジネスエラーをクラスで表現 |
| 3 | 早期失敗（Fail-Fast） | 「いまここ」で気づく |
| 4 | 境界での集約とログ | どこで捕まえ、どう記録するか |
| 5 | Result 型と Optional | 例外を使わない選択肢 |
| 6 | エラーレスポンスの設計 | API として何を返すか |
| 7 | よくあるつまずき | 握りつぶし・ログ二重出力など |

前半（1〜2節）で、**例外の使い分け**の指針を整理します。
中盤（3〜5節）で、**いつ例外を投げ、いつ別の手段を使うか**を学びます。
後半（6〜7節）で、**API としての見え方**と、現場の落とし穴を集めます。

---

## この章を読み終えると

第39章を読み終えるころには、次のことができるようになっています。

- チェック例外と非チェック例外を、現代の Java の感覚で使い分けられる
- ビジネスエラーを、**ドメイン例外クラス**として設計できる
- 早期失敗の発想で、コードを**短く・安全に**書ける
- 例外と `Optional` / Result 型を、目的に応じて選べる
- API のエラーレスポンスを、**業務システムとして筋のいい形**で設計できる

> 文法が怪しいときは、第20章にいつでも戻ってください。

それでは、最初の節「チェック例外と非チェック例外」から始めましょう。
