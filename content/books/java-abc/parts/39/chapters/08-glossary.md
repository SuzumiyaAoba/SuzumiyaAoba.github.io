## 用語集 ― この章で学んだ言葉

第39章では、エラーハンドリングと例外設計を学びました。
節ごとに、言葉を整理しておきます。

---

## 例外の使い分け（第1節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| チェック例外 | checked exception | `Exception` を継承（`RuntimeException` 以外） |
| 非チェック例外 | unchecked exception | `RuntimeException` を継承 |
| 例外チェーン | exception chaining | `cause` で元の例外を保持する仕組み |
| ラッピング | wrapping | 元の例外を別の例外でくるんで投げ直す |
| `DataAccessException` | ― | Spring が SQLException 等をラップした非チェック例外 |

---

## ドメイン例外（第2節）

| 用語 | 意味 |
|---|---|
| ドメイン例外 | 業務的な失敗を表す自作の例外 |
| `DomainException` | すべての業務例外の共通親（慣習） |
| 例外の階層化 | `NotFoundException` などの中間親で分類 |
| エラーコード | 機械可読な業務エラー識別子（`STOCK_INSUFFICIENT` など） |
| 形式エラー / 業務エラー | 入力の形式が NG / 業務ルール違反 |

---

## Fail-Fast（第3節）

| 用語 | 意味 |
|---|---|
| Fail-Fast | 問題が起きた時点ですぐ例外を投げて止める |
| ガード節 | メソッド冒頭の引数チェック |
| `Objects.requireNonNull` | 引数が null なら NPE を投げる標準ユーティリティ |
| `IllegalStateException` | オブジェクトの状態が不正 |
| `IllegalArgumentException` | 引数の値が不正 |
| 防御的プログラミング | 想定外の入力でも動き続けるアプローチ |

---

## 境界とログ（第4節）

| 用語 | 意味 |
|---|---|
| 境界 | サービスの最外周（API・バッチの main） |
| 例外の握りつぶし | `catch` しただけで何もしないアンチパターン |
| ログレベル | TRACE / DEBUG / INFO / WARN / ERROR |
| 構造化ログ | 機械処理しやすい形式（key=value, JSON） |
| traceId | リクエストを追跡するための ID |

---

## Result 型と Optional（第5節）

| 用語 | 意味 |
|---|---|
| `Optional<T>` | 値が「あるか・ないか」を表す |
| Result 型 | 「成功 or 失敗（理由つき）」を表す（自作 or vavr） |
| `sealed interface` + `record` | Result 型の自前実装の基本パターン |
| `Either[E, A]` | Scala / Haskell の同等概念 |
| 関数型エラーハンドリング | 例外ではなく値でエラーを扱う設計 |

---

## エラーレスポンス（第6節）

| 用語 | 意味 |
|---|---|
| ステータスコード | 400 / 401 / 403 / 404 / 409 / 422 / 500 など |
| 共通エラーフォーマット | API 全体で一貫したボディ構造 |
| ProblemDetail | RFC 7807 標準エラーフォーマット |
| 情報漏洩 | スタックトレース・SQL・パスワードを返さないこと |
| ログとレスポンスの分離 | 内部詳細はログ、利用者にはサマリ |

---

## つまずき（第7節）

| 用語 | 意味 |
|---|---|
| 握りつぶし | `catch` しただけで何もしないアンチパターン |
| 二重ログ | 同じ例外が複数の場所でログに出る |
| `printStackTrace` | 標準エラーに直接出す古い書き方（業務では使わない） |
| リソースリーク | クローズし忘れによる枯渇 |
| 結果整合性 | トランザクションを跨ぐ非同期処理での整合の取り方 |
| Resilience4j | リトライ・サーキットブレーカーの宣言的ライブラリ |

---

## おわりに ― 「壊れにくく、追跡できる」コードへ

第39章で、業務システムを**安全に運用する**ためのエラーハンドリングの基本がそろいました。
ここまでで、

- 例外と Result 型を、目的に応じて使い分けられる
- ドメイン例外で、業務エラーを型として表現できる
- 境界での集約とログで、運用しやすい構造を作れる
- API のエラーレスポンスを、業務システムらしく設計できる

ようになりました。
