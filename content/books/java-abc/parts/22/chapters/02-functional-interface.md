## 関数型インターフェース ― ラムダ式の型

前の節で、ラムダ式を `Runnable` 型の変数に代入しました。
では、ラムダ式は、どんな型にでも代入できるのでしょうか。
答えは「いいえ」です。ラムダ式が代入できるのは、**関数型インターフェース**と呼ばれる、特別なインターフェースだけです。
この節で、その正体を学びます。

---

## 関数型インターフェース ―「メソッドが1つだけ」のインターフェース

**関数型インターフェース**（Functional Interface）とは、

> **中身のないメソッド（抽象メソッド）が、ちょうど1つだけ**のインターフェース

のことです。

第16章で、インターフェースは「メソッドの約束」だと学びました。
そのうち、**約束しているメソッドが1つだけ**のものが、関数型インターフェースです。

前の節で使った `Runnable` も、その1つです。`Runnable` の中身は、実質これだけです。

```java
interface Runnable {
    void run();   // メソッドは、これ1つだけ
}
```

メソッドが `run()` 1つだけなので、`Runnable` は関数型インターフェースです。

---

## なぜ「1つだけ」が大事なのか

ラムダ式 `() -> IO.println("...")` は、「引数なしで、ある処理をする」というだけの、シンプルな式です。
これを、`Runnable` 型の変数に代入できたのは、**`Runnable` のメソッドが1つだけ**だからです。

考えてみてください。もしインターフェースにメソッドが2つあったら、ラムダ式 `() -> ...` は「**どちらのメソッドの中身なのか**」が分かりません。
メソッドが1つだけなら、迷う余地がありません。「このラムダ式は、その唯一のメソッドの中身だ」と、はっきり決まります。

```java
Runnable task = () -> IO.println("実行");
//      ↑                  ↑
//  関数型インターフェース   その唯一のメソッド run() の中身
```

つまり、**ラムダ式は「関数型インターフェースの、唯一のメソッドを実装したもの」**なのです。
だから Java は、ラムダ式を見ると、「ああ、これは、あの1つのメソッドの中身だな」と理解できます。

---

## 自分で関数型インターフェースを作る

関数型インターフェースは、自分でも作れます。
試しに、「2つの整数を受け取って、計算する」インターフェース `Calculator` を作ってみましょう。

```java
interface Calculator {
    int calculate(int a, int b);   // メソッドは1つだけ
}
```

メソッドが `calculate` 1つだけなので、これは関数型インターフェースです。
すると、この型の変数に、ラムダ式を代入できます。

```java
interface Calculator {
    int calculate(int a, int b);
}

Calculator add = (a, b) -> a + b;        // 足し算
Calculator multiply = (a, b) -> a * b;   // かけ算

IO.println(add.calculate(3, 5));
IO.println(multiply.calculate(3, 5));
```

```text
8
15
```

同じ `Calculator` 型なのに、`add` には「足す」処理、`multiply` には「かける」処理が入っています。
ラムダ式で、**処理の中身を、後から自由に決められる**のです。
そして `add.calculate(3, 5)` のように、その唯一のメソッド名で呼び出します。

---

## @FunctionalInterface ― 目印をつける

「このインターフェースは、関数型インターフェースとして使う」とはっきりさせたいときは、**`@FunctionalInterface`** というアノテーション（第14章）を付けます。

```java
@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);
}
```

`@Override`（第14章）と同じように、これは「**目印**」であり、間違いを防いでくれます。
もし、うっかりメソッドを2つ書いてしまったら、「関数型インターフェースなのに、メソッドが1つではない」とコンパイルエラーで教えてくれるのです。

```java
@FunctionalInterface
interface Bad {
    int calculate(int a, int b);
    int another();              // ✕ 2つ目のメソッド → エラー
}
```

```text
エラー: 予期しない@FunctionalInterface注釈
  Badは機能インタフェースではありません
    インタフェース Badで複数のオーバーライドしない抽象メソッドが見つかりました
```

付けるかどうかは任意ですが、「関数型インターフェースのつもり」なら、付けておくと安全です。

> **補足: デフォルトメソッドは数えない**
>
> 第16章で学んだ**デフォルトメソッド**（`default` 付き、中身のあるメソッド）は、いくつあっても構いません。
> 「**中身のない抽象メソッドが1つ**」であれば、関数型インターフェースです。
> 実際、標準ライブラリの関数型インターフェースの多くは、抽象メソッド1つに加えて、便利なデフォルトメソッドをいくつか持っています。

---

## まとめ

- **関数型インターフェース**は、**中身のない抽象メソッドが、ちょうど1つだけ**のインターフェース
- ラムダ式は、その「**唯一のメソッドの中身**」を表す（だからメソッドが1つでないと困る）
- ラムダ式を代入できるのは、関数型インターフェース型の変数だけ
- `Runnable` をはじめ、標準ライブラリに多くの関数型インターフェースがある（次の第4節）
- **`@FunctionalInterface`** を付けると、「メソッドが1つ」をコンパイラがチェックしてくれる

次の節では、ラムダ式の**書き方**（引数や本体の省略ルール）を、くわしく学びます。
