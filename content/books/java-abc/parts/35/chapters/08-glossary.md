## 用語集 ― この章で学んだ言葉

第35章では、SOLID の 5 原則と、その実践方法を学びました。
節ごとに、言葉を整理しておきます。

---

## SOLID の全体像（第0節）

| 略 | 原則 | ひとことで |
|---|---|---|
| **S** | 単一責任原則（SRP） | 1 クラス 1 責務 |
| **O** | 開放閉鎖原則（OCP） | 拡張に開き、修正に閉じる |
| **L** | リスコフの置換原則（LSP） | サブクラスは置換可能 |
| **I** | インターフェース分離原則（ISP） | 使わないメソッドに依存しない |
| **D** | 依存性逆転原則（DIP） | 具象ではなく抽象に依存 |

---

## SRP（第1節）

| 用語 | 意味 |
|---|---|
| 単一責任原則 | 1 つのクラスは、1 つの理由でしか変更されない |
| 神クラス | 何でもやる巨大クラス（アンチパターン） |
| 変更の理由 | クラスを変える主体・きっかけ |
| 過剰分割 | やみくもに細かいクラスに分けすぎること |

---

## OCP（第2節）

| 用語 | 意味 |
|---|---|
| 開放閉鎖原則 | 拡張に開き、修正に閉じる |
| 多態（ポリモーフィズム） | 同じ呼び出しが相手によって違う動きをする |
| 変わるところ／変わらないところ | 設計の境界を見抜く視点 |
| YAGNI | You Aren't Gonna Need It（必要にならない先回り設計） |

---

## LSP（第3節）

| 用語 | 意味 |
|---|---|
| リスコフの置換原則 | サブクラスは親の代わりに使える |
| 契約による設計 | 親クラスの「振る舞い」を契約として尊重する |
| 「ペンギン問題」 | 「ペンギンは鳥」だが `fly()` ができない |
| コンポジション | 「is-a」より「持つ関係（has-a）」で表現する設計 |

---

## ISP（第4節）

| 用語 | 意味 |
|---|---|
| インターフェース分離原則 | 使わないメソッドに依存させない |
| 巨大インターフェース | あらゆるメソッドを持つ大きな interface |
| マイクロインターフェース化 | 過剰に小さい interface を作るアンチパターン |
| 利用側ごとに分ける | クライアントの視点で interface を切る |

---

## DIP（第5節）

| 用語 | 意味 |
|---|---|
| 依存性逆転原則 | 上位も下位も抽象に依存する |
| 上位モジュール | ビジネスロジックの中心 |
| 下位モジュール | DB・HTTP・メールなどの実装 |
| 抽象 | 上位のニーズで定義された interface |
| 依存性の注入（DI） | DIP を実現するテクニック（コンストラクタ注入） |

---

## 実践と落とし穴（第6〜7節）

| 用語 | 意味 |
|---|---|
| コンストラクタ注入 | 依存先をコンストラクタの引数で受け取る形 |
| マントラ化 | 中身を考えず原則を振り回す態度 |
| 衝突する原則 | SRP + ISP で過剰分割、DIP で interface 爆発 |
| 読みやすさ最優先 | すべての判定の最終基準 |

---

## おわりに ― 次は「DI と Spring」

第35章で、設計の判断軸となる SOLID 原則がそろいました。
ここまでで、

- **テストの書き方**（第32〜33章）
- **パターンとその背景**（第34〜35章）

を学び終え、「**変更に強く、テストできる設計**」を組む力が、ひととおり身につきました。

次の第36章では、ここで学んだ「**抽象に依存し、外から注入する**」設計を、**フレームワークが自動でやってくれる**しくみ ―― **依存性の注入（DI）と Spring** を学びます。
これまで自分の手で行ってきた `new UserService(repo, mailer, ...)` のような組み立てを、Spring が代わりにしてくれるようになります。

ここまで学んだ SOLID 原則は、Spring を使うときの**強力な羅針盤**になります。
それでは、次の章でお会いしましょう。
