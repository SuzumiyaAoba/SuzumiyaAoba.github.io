## よくあるつまずき

SOLID 原則を学んだ直後の落とし穴を、まとめて確認します。

---

## 1. 原則を「マントラ」にする

「SRP に従え!」「DIP に従え!」と、唱えるように原則を振りかざすのは、いちばん危険な落とし穴です。
コードの中身を見ずに、形だけ原則に当てはめても、よい設計にはなりません。

### 対処

- 原則は「**指針**」であって、「**義務**」ではない
- **何のために**原則に従うのか（変更しやすさ・テストしやすさ）を意識する
- 「**いまの問題は、どの原則で説明できるか?**」と、後から振り返るくらいでよい

---

## 2. interface を切り過ぎる

DIP と ISP を一度に学ぶと、こんなコードを書きがちです。

```java
interface UserGetter { User getById(long id); }
interface UserSaver { void save(User user); }
interface UserDeleter { void delete(long id); }
interface UserListFetcher { List<User> all(); }
```

実装も使う側も**1 つしかない**のに、interface だけが量産される。
これは「**抽象化のための抽象化**」で、コードの理解を逆に妨げます。

### 対処

- **実装が 1 つ**で、**モック化の予定もない**なら interface はいらない
- 「**いま、抽象を入れて何がうれしいか**」を必ず問う

---

## 3. SRP に従って細かく分け過ぎる

```java
// △ やりすぎ
class UserNameFormatter { ... }
class UserEmailFormatter { ... }
class UserPhoneFormatter { ... }
class UserAddressFormatter { ... }
```

責務を細かく分け過ぎると、

- **クラス数だけ**増えて、コード全体が追いにくくなる
- **どこに何があるか**、毎回ファイルを開いて確認することになる

「Formatter ファミリー」のような細分化は、ふつう過剰です。

### 対処

- 「**変更の理由**」で測る（4 つとも同じ「フォーマット担当」かも）
- 1 つのクラスに**まとまっていても自然**なら、そのままでよい

---

## 4. OCP に従って「将来の拡張」を想像し過ぎる

```java
// △ 1つしかない実装に、将来のために interface を切る
interface AbstractTax { int calculate(int amount); }
class JapanTax implements AbstractTax { ... }
// （他に実装する予定はない）
```

「いつか海外対応するかも」と先回りして抽象化するのは、典型的な**YAGNI**（You Aren't Gonna Need It、必要にならない）違反です。

### 対処

- **必要になってからリファクタリング**する
- 「変わるかも」より「**変わったから直す**」
- 想像上の拡張のために、いまの設計を複雑にしない

---

## 5. LSP を「is-a だから OK」で済ます

「**ペンギンは鳥である**」「**正方形は長方形である**」のように、**日本語の常識**で継承を判断するのは危険です。

### 対処

- 「**親の振る舞いを、サブクラスがすべて満たせるか?**」を必ず確認する
- 「同じ`〜である`なのに、振る舞いが矛盾する」なら、継承はやめる
- 継承の代わりに、**コンポジション**（持つ関係）や、**インターフェース**で表現する

---

## 6. DIP のために「具象を直接呼んだら違反」と機械的に判断する

たとえば、`LocalDate.now()` や `Math.max(a, b)` を直接呼ぶことは、ふつう DIP 違反とは言いません。

```java
// これは DIP 違反ではない（標準ライブラリの純粋関数）
int max = Math.max(a, b);
LocalDate today = LocalDate.now();
```

### 対処

- DIP が問題になるのは、**「テストで差し替えたい」「実装を切り替えたい」**境界だけ
- 標準ライブラリの**純粋関数**や、**ふつうの値オブジェクト**まで抽象化しない
- 「**外の世界**との接点」を中心に、DIP を適用する

---

## 7. SOLID を「学んだから使う」ではなく「困ったから使う」

「学んだから、新しいコードに全部適用してみよう」とすると、シンプルなコードまで複雑にしてしまいます。

### 対処

- **困っていないコードに、原則をねじ込まない**
- 「**何度も書き換えるなあ**」と感じてから、SRP や OCP を思い出す
- リファクタリングする**理由**があってから、原則を**当てる**

---

## 8. 原則どうしの衝突を「どっちも従う」と無理する

実は、SOLID 原則は **互いに衝突する**ことがあります。

- SRP で細かく分けると、**ISP で interface も細分化**したくなる → クラスと interface だらけ
- OCP で抽象に依存させると、**DIP の境界が増える** → ファイル数が爆発

### 対処

- 衝突したら、**「いま、どの原則が解決したい問題に近いか」**で優先順位を決める
- 全部に満点で従う必要はない。**バランスのよい妥協点**を選ぶ
- 「**コードが読みやすいか**」が、最終的な判定基準

---

## 9. SOLID をクラスにしか適用しないと思い込む

SOLID は、もともと**クラス設計**の原則ですが、

- **メソッド**（1 メソッド 1 責務 → SRP）
- **モジュール**（パッケージごとに責務 → SRP）
- **サービス**（マイクロサービス間の依存 → DIP）

など、**いろんな粒度**で応用できます。
「クラスにしか使えない」と限定する必要はありません。

---

## まとめ

- 原則は**指針**であって、**義務ではない**
- **interface・クラスを切り過ぎない**。1 実装なら抽象は不要
- **将来の拡張**を先回りしない（YAGNI）
- LSP は「is-a」ではなく**振る舞いの一致**で判断する
- DIP は**境界**にだけ使う。標準ライブラリ全部に適用しない
- **困ってから**原則を使う。学んだから使う、ではない
- 原則の**衝突**は、優先順位を決めて妥協する
- **読みやすさ**が、すべての判定の最終基準

次は、この章で学んだ言葉を、用語集としてまとめます。
