## JIT コンパイラ ― 走らせながら最適化する

第1節で「JVM はバイトコードを**まず解釈実行**し、ホットな部分は**ネイティブコードに翻訳**する」と書きました。
その翻訳を担うのが、**JIT**（Just-In-Time、実行時に）**コンパイラ**です。
この節では、その動きを実測で確かめながら整理します。

---

## まず実測 ―― 「走らせるうちに速くなる」

同じ計算を 5 回繰り返す、シンプルなベンチマークです。

```java
public class Bench {
    public static long sum(int n) {
        long s = 0;
        for (int i = 0; i < n; i++) {
            s += i;
        }
        return s;
    }
    public static void main(String[] args) {
        for (int trial = 0; trial < 5; trial++) {
            long t = System.nanoTime();
            long r = sum(50_000_000);
            long elapsed = System.nanoTime() - t;
            System.out.printf("trial %d: %d  (%.2f ms)%n", trial, r, elapsed / 1_000_000.0);
        }
    }
}
```

実機で走らせると、こうなります（実機・実行ごとに数字は変わります）。

```text
$ java Bench
trial 0: 1249999975000000  (30.46 ms)
trial 1: 1249999975000000  (29.70 ms)
trial 2: 1249999975000000  (15.44 ms)
trial 3: 1249999975000000  (15.32 ms)
trial 4: 1249999975000000  (15.46 ms)
```

同じコードなのに、**最初の 2 回は 30 ms、3 回目以降は 15 ms** ―― ほぼ半分に縮みました。
これが JIT の効果です。
途中で `sum` が「ホットな（よく呼ばれる）メソッド」と判明し、ネイティブコードに翻訳されたのです。

このような「**走らせるうちに速くなる**」現象は、**ウォームアップ**（warmup、暖機運転）と呼ばれます。

---

## なぜ「走らせながら」翻訳するのか

「最初から全部ネイティブコードにコンパイルすればいいのでは?」と思うのが自然です。
ですが、JIT 方式には、それを上回るメリットがあります。

| 静的コンパイル（C などの方式） | JIT |
|---|---|
| ビルド時点での情報だけで最適化 | **実行時の情報**（実際の呼び出し頻度・引数の傾向）で最適化 |
| すべてのコードを翻訳しておく必要 | **使われない部分は翻訳しない**（時間と空間の節約） |
| バイナリは特定の CPU 用 | バイトコードは**どこでも動く** |

実行時情報を使うので、JIT は静的コンパイラよりも**強い最適化**をしばしば行います。
代表的な最適化を紹介します。

- **インライン化**（Inlining）: 呼び出しコストを消すため、メソッド呼び出しを呼び出し先の本体で置き換える
- **エスケープ解析**（Escape Analysis）: そのオブジェクトがメソッドの外に漏れないと分かれば、ヒープではなく**スタックに置く**（スカラ置換）
- **ループアンロール**（Loop Unrolling）: ループの中身を 2 回・4 回ぶん展開して、判定回数を減らす
- **デッドコード除去**: 結果が使われない計算を削る
- **分岐予測ヒント**: 実際の頻度から、可能性の高い分岐を CPU パイプラインに優しい順に並べる

これらは**実行時の挙動**を見てから判断します。
たとえばインライン化は、「**インタフェースの実装が実は 1 つしかない**」と分かれば、仮想呼び出しを直接呼び出しに置き換える、といった具合です。

## 階層的コンパイル（Tiered Compilation）

HotSpot JVM の JIT は、実は**1 つではなく 4 段階**で動いています。

| Tier | 何をするか |
|---|---|
| **Tier 0** | インタプリタで実行（最も遅いが、立ち上がりは即時） |
| **Tier 1** | C1（クライアントコンパイラ）で「とりあえず軽く」翻訳 |
| **Tier 2-3** | C1 で**プロファイル収集付き**に翻訳 |
| **Tier 4** | C2（サーバーコンパイラ）で**徹底最適化**して翻訳 |

C1 は**速く出来上がるが最適化は弱い**、C2 は**最適化は強いがコンパイルに時間がかかる** ―― という棲み分けです。
最初は C1 で軽く翻訳して動かし、本当によく呼ばれると分かれば C2 で本気の最適化をかける、というスタンスです。

実機の閾値は、`-XX:+PrintFlagsFinal` で見られます。

```text
$ java -XX:+PrintFlagsFinal -version | grep -E "Tier|CompileThreshold"
intx Tier3CompileThreshold     = 2000
intx Tier4CompileThreshold     = 15000
bool TieredCompilation         = true
intx CompileThreshold          = 10000
...
```

これらは「およそ何回呼ばれたら（あるいはループが何回回ったら）コンパイル候補にするか」の目安です。
覚える必要はありません。「**閾値はチューニングできる**」とだけ知っておけば十分です。

## バイトコードからネイティブまでの道筋

ここまでをまとめて、第1節の図に**もう一段**追加します。

```text
.class
  ↓  クラスローダ
JVM のメソッド領域に登録
  ↓  まずインタプリタで実行（Tier 0）
プロファイル収集（呼び出し回数・ループ回数・型の分布）
  ↓  閾値超え
C1 で翻訳（Tier 1〜3、軽い最適化）
  ↓  さらに閾値超え
C2 で翻訳（Tier 4、徹底最適化）
  ↓
ネイティブコードキャッシュ（CodeCache）に置かれる
  ↓
以後の呼び出しは、ネイティブコード版を実行
```

JIT が翻訳したネイティブコードは、**コードキャッシュ**（CodeCache）というネイティブメモリの領域に置かれます。
ここが溢れると `CodeCache is full` という警告とともに、JIT が止まり、**インタプリタ走行に戻る**ことがあります。
本番で「**徐々に遅くなった**」原因がこれだった、という事例もあるので、頭の片隅に置いておきましょう。

## 「JIT が効かない」コード

最適化は万能ではありません。次のような書き方は、JIT に**情報を渡せず**、最適化が効きにくくなります。

- **超巨大なメソッド**（数千行クラスのメソッドなど）: そもそも JIT 対象から外れる
- **動的に毎回違う型を返す**: 型のプロファイルが定まらず、インライン化が決まらない
- **過剰なリフレクション**（第46章）: 動的呼び出しでインライン化が効きにくい
- **`Thread.sleep` や I/O が中心**: 計算が走らないので、最適化しても効果が出ない

ここで重要なのは、「**JIT に好かれるコードを書く**」ことが目的化してはいけない、という点です。
**読みやすく、まず正しく**書く。それで遅かったときに、初めて**測って**（第48章）チューニングを考えます。

---

## ベンチマークを書くときの注意 ― 「ウォームアップを忘れない」

JIT があるからこそ、ベンチマークを書くときは**ウォームアップ**が必須です。
本節の冒頭の `Bench` でも、

- **trial 0, 1** は JIT がまだ仕事しきっていない（**捨て計測**）
- **trial 2 以降** が、本来知りたい性能

このように、最初の数回を**捨てる**のが鉄則です。
正しいベンチマークの取り方は、**JMH**（Java Microbenchmark Harness）を使うのが定石で、第50章で扱います。

---

## まとめると

- JIT は、バイトコードを**実行時にネイティブコード**に翻訳するコンパイラ
- HotSpot JVM では、**インタプリタ → C1 → C2** の **階層的コンパイル**で動く
- 翻訳ずみコードは **CodeCache** に置かれる
- 「**走らせるうちに速くなる**」のはウォームアップ。ベンチマークでは最初を捨てる
- JIT に好かれる書き方を目的化せず、**まず正しく書く**
