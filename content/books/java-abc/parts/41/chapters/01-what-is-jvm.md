## JVM とは何か ― コードが実行されるまで

第1章で、Java は「**書いたコードをコンパイルしてバイトコードにし、JVM の上で動かす**」と説明しました。
この節では、その「**JVM の上で動く**」の中身を、もう一段ほどいて見ていきます。

---

## JVM の役割 ― 仮想的なコンピュータ

**JVM**（Java Virtual Machine、Java仮想マシン）は、その名のとおり「**仮想的なコンピュータ**」です[^jvms-25]。

| | 物理的な CPU | JVM |
|---|---|---|
| 実行する命令 | x86 / ARM など、CPU 固有の機械語 | バイトコード（どの OS でも同じ） |
| メモリ管理 | OS が管理 | JVM が管理（ヒープなど） |
| 実体 | シリコンの回路 | OS 上で動くプログラム（`java` コマンド） |

ハードウェアの CPU は、x86 と ARM では機械語の命令体系が違います。
JVM が間にはさまることで、書いた Java コードは「**x86 でも ARM でも同じバイトコードのまま**」動きます。
これが、第1章で学んだ「**Write Once, Run Anywhere**（一度書けば、どこでも動く）」の正体です。

---

## ソースから実行までの 5 段階

`.java` を書いてから、CPU で命令が走るまでには、おおまかに 5 つの段階があります。

```text line-numbers=false
[1] .java ファイル
       ↓  javac（コンパイラ）
[2] .class ファイル（バイトコード）
       ↓  クラスローダ
[3] JVM のメモリ（メソッド領域・ヒープ など）
       ↓  実行エンジン（インタプリタ）
[4] バイトコードを 1 命令ずつ解釈して実行
       ↓  JIT コンパイラ（ホットな部分だけ）
[5] ネイティブコード（CPU が直接実行する機械語）
```

注目してほしいのは、**JVM の中で 2 段階の翻訳**が走っていることです。

- **コンパイル時**（`javac`）: `.java` → バイトコード
- **実行時**（JVM 内の JIT）: バイトコード → ネイティブコード

「Java はコンパイル言語」と言われますが、実はもう一段、**実行中にもコンパイル**しています。
このおかげで、JVM は**動かしながら最適化**できます。

---

## まずは触って確かめる ― `javac` と `javap`

「コードが `.class` になる」感覚を、手で動かしてつかみましょう。
次の `Hello.java` を作って、`javac` でコンパイルします。

```java
public class Hello {
    public static void main(String[] args) {
        int x = 1 + 2;
        System.out.println("x = " + x);
    }
}
```

```text line-numbers=false
$ javac Hello.java
$ ls
Hello.class  Hello.java
```

`Hello.class` ができました。これがバイトコードです。
中身は人間が読めるテキストではないので、`javap` という標準ツールで読み解きます。

```text line-numbers=false
$ javap -c -p Hello.class
Compiled from "Hello.java"
public class Hello {
  public Hello();
    Code:
         0: aload_0
         1: invokespecial #1     // Method java/lang/Object."<init>":()V
         4: return

  public static void main(java.lang.String[]);
    Code:
         0: iconst_3
         1: istore_1
         2: getstatic     #7     // Field java/lang/System.out:Ljava/io/PrintStream;
         5: iload_1
         6: invokedynamic #13,  0  // InvokeDynamic #0:makeConcatWithConstants:(I)Ljava/lang/String;
        11: invokevirtual #17    // Method java/io/PrintStream.println:(Ljava/lang/String;)V
        14: return
}
```

`iconst_3` `istore_1` ... これがバイトコードの**ニーモニック**（命令の人間向け表記）です。
読み解きは第3節でやります。今ここで気づいてほしいのは、次の3点です。

- `int x = 1 + 2;` と書いたのに、バイトコードは **`iconst_3`**（3 をスタックに積む）になっている
- ソースには書いていない `Hello()` という**デフォルトコンストラクタ**が生えている
- `"x = " + x` は **`invokedynamic`** という命令を使って文字列連結している

1 つめは、`javac` が「**コンパイル時に分かる計算**は事前に済ませる」**定数畳み込み**（constant folding）を行ったためです。
このように、`.java` と `.class` は、**1 対 1 の対応ではありません**。
javac の段階で**ある程度の最適化**が入っています。

---

## 実行時にも翻訳が走る ― インタプリタと JIT

`java Hello` を実行すると、JVM はおおまかに次の流れで動きます。

1. クラスローダが `Hello.class` を読み込む
2. メソッド領域に、クラス情報を配置する
3. **実行エンジン**が、`main` のバイトコードを 1 命令ずつ**解釈**して実行する（**インタプリタ**）
4. 同じメソッドが「**よく呼ばれる**」と判明したら、JIT コンパイラがそれを**ネイティブコード**に翻訳する
5. 次からはネイティブコード版を呼ぶ（速い）

つまり、**最初は遅く、走らせるうちに速くなる**のが Java の標準的なふるまいです。
ベンチマークで「**1 回目だけ遅い**」のは、これが理由です。詳しくは第5節で扱います。

---

## どこに何があるか ― 主要な部品

JVM の内部は、本書の範囲では次のように分けて理解すれば十分です。

```text line-numbers=false
           ┌─────────────────────────────────────────┐
           │              JVM プロセス               │
           │                                         │
ロード     │  ┌─────────┐                            │
─────────► │  │クラスローダ│ ── .class を読み込む       │
.class     │  └─────────┘                            │
           │                                         │
           │  ┌─────────────────────────────────┐    │
           │  │     ランタイムデータ領域         │    │
           │  │ ┌──────┐ ┌──────┐ ┌──────────┐ │    │
           │  │ │ヒープ│ │スタック│ │メソッド領域│ │    │
           │  │ └──────┘ └──────┘ └──────────┘ │    │
           │  └─────────────────────────────────┘    │
           │                                         │
           │  ┌─────────────────────────────────┐    │
           │  │       実行エンジン              │    │
           │  │  インタプリタ + JIT + GC        │    │
           │  └─────────────────────────────────┘    │
           └─────────────────────────────────────────┘
```

- **クラスローダ**: 必要なクラスを `.class` ファイル（または jar、jmod）から読んでくる ―― 第2節
- **ランタイムデータ領域**: 実行中のデータが置かれるメモリ ―― 第4節
- **実行エンジン**: バイトコードを解釈／コンパイルして実行する ―― 第5節
- **GC**: 不要になったオブジェクトを回収する ―― 第42章

第41章では、上の図の中身を、節ごとに少しずつ展開していきます。

---

## まとめると

- JVM は、バイトコードを実行する**仮想的なコンピュータ**である
- ソースから実行までは、**5 段階**（`.java` → バイトコード → ロード → 解釈実行 → JIT）
- `javac` の時点で**定数畳み込み**などの最適化が入る
- 実行中もインタプリタと JIT の **2 段構え**で動く
- 主要部品は「**クラスローダ・ランタイムデータ領域・実行エンジン・GC**」の4つ

次の節では、この図の左上「**クラスローダ**」を、3 階層の構造とともに見ていきます。
