## ランタイムデータ領域 ― ヒープ・スタック・メソッド領域

JVM が動いているあいだ、データはひとつのメモリにまとめて置かれているわけではありません。
役割ごとに**いくつかの領域**に分かれています。
それぞれの違いを知っておくと、`OutOfMemoryError` が出たときの原因当てが**確かに**なります。

---

## 全体像

JVM のランタイムデータ領域は、**5 つ**に分けて理解しておけば十分です。

```text
   ┌─────────────────────────────────────────────────────────┐
   │                       JVM プロセス                     │
   │                                                         │
   │  ┌─────────────┐   ┌─────────────────────────────────┐  │
   │  │ ヒープ      │   │  メソッド領域（Metaspace）       │  │
   │  │ （オブジェクト） │   │ クラス情報・static フィールド     │  │
   │  └─────────────┘   └─────────────────────────────────┘  │
   │                                                         │
   │  ┌─── スレッドごと ──────────────────────────────┐      │
   │  │ ┌────────┐ ┌────────────────┐ ┌────────────┐ │      │
   │  │ │JVM スタック│ │ PC レジスタ     │ │ ネイティブ    │ │      │
   │  │ │（フレーム） │ │ （実行位置）    │ │ メソッド     │ │      │
   │  │ └────────┘ └────────────────┘ │ スタック    │ │      │
   │  │                              └────────────┘ │      │
   │  └────────────────────────────────────────────┘      │
   │                                                         │
   └─────────────────────────────────────────────────────────┘
```

「**スレッドごと**」と書いた 3 つは、スレッドが 1000 本あれば 1000 個ずつあります。
これに対し、**ヒープ**と**メソッド領域**は、JVM プロセス全体で **1 つだけ**共有されます。

---

## 1. ヒープ（Heap） ― オブジェクトの住処

`new` で作ったオブジェクトは、全部ここに置かれます。
String・配列・あなたが書いたクラスのインスタンス ―― すべて**ヒープ**です。

| 特徴 | |
|---|---|
| **誰が触れるか** | すべてのスレッドが共有 |
| **誰が掃除するか** | GC（第42章） |
| **サイズ** | 起動オプションで設定（`-Xms` 初期・`-Xmx` 最大） |
| **`OutOfMemoryError` の表記** | `Java heap space` |

ヒープサイズは、jshell から `Runtime.getRuntime().maxMemory()` で確認できます。

```text
jshell> Runtime.getRuntime().maxMemory() / 1024 / 1024 + " MB"
$1 ==> "6144 MB"
```

明示しないと、JVM は**物理メモリの 1/4 程度**を最大値の目安にします（実機依存）。
意図して制御したいときは、`java -Xmx512m -Xms256m MyApp` のように指定します。

---

## 2. メソッド領域（Method Area） ― クラスのカタログ

クラスローダが読み込んだ**クラス情報**は、ここに置かれます。

| 何が置かれるか | |
|---|---|
| クラスの構造（フィールド・メソッドの定義） | |
| 定数プール（第6節） | |
| `static` フィールドの実体 | |
| メソッドのバイトコード | |

> Java 7 までは、メソッド領域は「**Permanent Generation**（永続世代、PermGen）」というヒープの一部として実装されていました。
> Java 8 で **Metaspace** に置き換わり、ヒープから独立して**ネイティブメモリ**上に置かれるようになりました。
> `-XX:MaxMetaspaceSize` で上限を設定できます。
> 設定がないと**事実上無制限**で、リーク時はネイティブメモリを食い潰します。

`OutOfMemoryError` の表記としては `Metaspace` と出ます。
動的にクラスを生成し続けるアプリ（旧来の JSP・OSGi・一部の DI コンテナ）で詰まりやすい領域です。

---

## 3. JVM スタック ― メソッド呼び出しのレール

`main` から `a()` を呼び、`a()` から `b()` を呼ぶと、スタックには**フレーム**（Frame）が積み上がります。

```text
[ b のフレーム ]   ← 一番上が今実行中
[ a のフレーム ]
[ main のフレーム ]
```

1 つのフレームには、その**メソッド呼び出し 1 回ぶん**の情報が入っています。

- **ローカル変数領域**: 引数とローカル変数の表（`this`、`String[] args`、`int x` …）
- **オペランドスタック**: 計算の作業場（`iadd` で 2 値を取り、結果を 1 値積む）
- **戻り先**: `return` で戻る位置

メソッドの呼び出しごとに**1 つフレームが生まれ**、`return` で**1 つ捨てられる**。
それだけのシンプルな仕組みです。

| 特徴 | |
|---|---|
| **誰のものか** | スレッドごとに 1 本 |
| **サイズ** | `-Xss512k` などで指定 |
| **`StackOverflowError`** | スタックが深くなりすぎた（無限再帰など） |

---

## 4. PC レジスタ ― 次に実行する位置

物理 CPU の PC（プログラムカウンタ）と同じ役割で、**次に実行するバイトコードのアドレス**を持ちます。
スレッドごとに 1 つあります。
ふだん意識する必要はありませんが、「**スレッドはこの値を切り替えながら走る**」という事実だけ覚えておくと、第43章の並行処理が腑に落ちます。

## 5. ネイティブメソッドスタック

`native` 修飾子のついたメソッド（JNI/Panama 経由で呼ぶ C/C++ コードなど）の呼び出し用に、別のスタックがあります。
本書のサンプルでは触れません。

---

## 「変数はどこに置かれるか」

入門書で必ず出てくる疑問に答えを整理しておきます。

| 値の種類 | 置かれる場所 |
|---|---|
| `int x = 1;`（メソッドの中の int） | **そのメソッドのフレーム**（スタック）の中 |
| `new int[100]`（配列の中身） | **ヒープ**。`x` から指している参照は**フレーム**にある |
| `String s = "abc";`（リテラル） | "abc" は**文字列プール（ヒープ内）**。`s` の参照は**フレーム** |
| `static int counter;` | **メソッド領域** |
| `class Foo` の構造定義 | **メソッド領域** |
| `new Foo()` のインスタンス | **ヒープ** |

「**プリミティブはスタック、参照型はヒープ**」というのは、おおむね正しいですが、**メソッドのローカル変数の話**として読むと正確です。
たとえば**インスタンスフィールド**としての `int` は、そのオブジェクトが**ヒープ**に置かれている以上、**ヒープ**にあります。

## `OutOfMemoryError` から原因領域を当てる

実務で見る `OutOfMemoryError` には、**メッセージで領域が書いてあります**。
慣れると、これだけで対処の見当がつきます。

| メッセージ | 領域 | 主な原因 |
|---|---|---|
| `Java heap space` | ヒープ | オブジェクトを溜め込みすぎ（コレクションへの追加忘れ、キャッシュリーク） |
| `Metaspace` | メソッド領域 | クラスを動的に作り続けている |
| `GC overhead limit exceeded` | ヒープ | GC は走るが回収できない（リーク寸前） |
| `unable to create native thread` | OS のスレッド数 | スレッド作りすぎ。仮想スレッドの出番（第44章） |
| `Direct buffer memory` | DirectByteBuffer | NIO の直接バッファ。`-XX:MaxDirectMemorySize` を見直す |
| `Requested array size exceeds VM limit` | 単発の配列 | 配列を `Integer.MAX_VALUE` 近くまで取ろうとした |

そして、これとは別物の `StackOverflowError` が出たら、**スタックの問題**（フレームの積み過ぎ）です。
無限再帰、または深すぎる再帰を疑います。

---

## まとめると

- JVM の領域は、**ヒープ・メソッド領域・スタック・PC・ネイティブスタック** の 5 つ
- ヒープとメソッド領域は**全スレッド共有**、スタック類は**スレッドごと**
- メソッド領域は Java 8 以降は **Metaspace**（ネイティブメモリ）
- 「**変数はどこに**」は、メソッド内の変数か、フィールドかで答えが変わる
- `OutOfMemoryError` は**メッセージで領域がわかる**
