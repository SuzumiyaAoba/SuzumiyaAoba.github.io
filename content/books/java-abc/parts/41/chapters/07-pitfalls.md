## よくあるつまずき

第41章で扱った内容は、本書のなかでも特に**現場で混乱しやすい**領域です。
ここでは、運用の場面で「**あれ?**」となりやすい 7 つを整理しておきます。

---

## つまずき1: 「`OutOfMemoryError` = ヒープ不足」と決めつける

`OutOfMemoryError` と聞くと反射的に「ヒープを増やせばいい」と思いがちですが、第4節で見たように、**領域はいくつもあります**。

| メッセージ | 増やすべきもの |
|---|---|
| `Java heap space` | `-Xmx`（ヒープ） |
| `Metaspace` | `-XX:MaxMetaspaceSize`（Metaspace） |
| `Direct buffer memory` | `-XX:MaxDirectMemorySize` |
| `unable to create native thread` | OS のスレッド上限（ulimit）。または仮想スレッド（第44章）に切り替える |
| `GC overhead limit exceeded` | 増やしてもダメ。**リークを直す** |

ヒープを増やしても直らない `OOM` は世の中に山ほどあります。
**メッセージを最後まで読む**ことから始めましょう。

---

## つまずき2: 「ローカル変数の `int` はヒープ?」

第4節で見たとおり、**メソッド内の `int x = 1;` の `x` は、スタック（フレーム）**にあります。
ヒープではありません。

ただし、

- **インスタンスフィールド**としての `int` は、そのインスタンスが**ヒープ**にあるので、ヒープに含まれる
- ローカル変数の **`Integer i = 1;`**（参照型）は、**`i` 自体はスタック**にあるが、**指している `Integer` オブジェクトはヒープ**

このように、「**変数の住所**」と「**指し示すオブジェクトの住所**」を分けて考えることが、Java の型を理解する核です。

---

## つまずき3: 「ベンチマークの 1 回目が一番速い／遅いと結論する」

第5節で触れたウォームアップの逆を踏むケースです。

- `java MyApp` を起動して、即計測した結果を「これが性能」と判断してしまう
- 1 回しか測らず、たまたまの遅さを「遅い」と決めつける

JIT がある以上、**最初の数回は安定しない**前提で測るのが鉄則です。
本格的な計測は **JMH**（第50章で紹介）を使い、ウォームアップ回数と計測回数を明示的に分けます。

## つまずき4: 「`ClassNotFoundException` を catch しても何も起きない」

```java
try {
    Class.forName("com.foo.Bar");
} catch (ClassNotFoundException e) {
    // 握りつぶし
}
```

これだと、後で `Bar` を使ったところで `NoClassDefFoundError` が出て、原因が**もう一段下**に潜ります。
**`ClassNotFoundException`** は、対処の余地があるか **ログを出す**かのどちらかで扱いましょう。
握りつぶすと、本当の問題（クラスパスの設定ミスなど）が見えなくなります。

## つまずき5: 「`static` の初期化順を式の左辺の登場順だと思う」

第2節の「**初めて使われたときに初期化が走る**」ルールは、デバッグで効いてきます。

```java
public class A {
    static int x = B.value() + 1;
}
public class B {
    static int value() { return A.x; }
}
```

`A` を最初に使うと、`A` の `<clinit>` が走り、`B.value()` を呼ぶと **A.x はまだ初期化中**で `0`（int のデフォルト値）が返ります。
結果、`A.x = 1`。
一方、`B` を先に使うと、`B.value()` 呼び出し時点で `A` の `<clinit>` が走り、こちらが**再帰**して `B.value()` を呼ぶので、また `A.x = 0` を見て `A.x = 1` 。

このような**循環的な依存**は、デフォルト値で**静かに** 落ち着いてしまうので、気づきにくいです。
**static 初期化に他のクラスの static 呼び出しを混ぜない**のが、いちばんの予防策です。

## つまずき6: 「`UnsupportedClassVersionError` を見ても原因が分からない」

メッセージ例:

```text line-numbers=false
java.lang.UnsupportedClassVersionError: com/example/Foo has been compiled by
a more recent version of the Java Runtime (class file version 65.0),
this version of the Java Runtime only recognizes class file versions up to 61.0
```

第6節の表で、

- `65.0` → Java **21** でビルドされた
- `61.0` → 実行中は Java **17**

と読み取れます。
**ビルドより古い JVM で動かそうとした**のが原因です。
CI と本番で JVM のバージョンが揃っていないとよく起きます。`java --version` で**実行する JVM**を確認しましょう。

## つまずき7: 「`javap` の出力は読まなくていい」

最後はマインドセットの話です。
バイトコードを読めるようになる必要はありません。
でも、`javap -c` でちらっと覗いてみる、`javap -v` で定数プールを覗いてみる ―― この**少しの慣れ**が、

- ライブラリの「**本当に呼ばれているメソッド**」を知る
- ラムダ・switch パターンが「**どんな命令に展開された**」かを知る
- バージョン更新で「**生成バイトコードが変わった**」ことに気づく

といった、地に足のついたデバッグを可能にします。
怖がらず、`javap` を**触ってみる**こと。これが第41章を読んだ最大の収穫です。

## まとめると ―― 4 つの心得

最後に、第41章全体から得るべき心得を 4 つにまとめます。

1. **「動くしくみ」を知ると、トラブルの原因が早く絞れる**
2. **`OutOfMemoryError` はメッセージを最後まで読む**
3. **ベンチマークはウォームアップを忘れない**
4. **`javap` で覗くクセをつける**
