## はじめに ― この章で学ぶこと

前の第26章で、何度か「**取りうる型が限られていれば、`switch` に `default` がいらない**」と予告しました。
その「取りうる型を限定する」しくみが、この第27章で学ぶ **シールドクラス**（Sealed Classes）です。

第14章・第16章で学んだ継承では、「親クラスやインターフェースは、**誰でも自由に**継承（実装）できる」のが基本でした。
ですが、ときには「**この型を継承（実装）できるのは、決まったクラスだけにしたい**」ことがあります。

たとえば、「図形（`Shape`）は、円・長方形・三角形の**3種類だけ**」と決めたいとします。
シールドクラスを使うと、「`Shape` を実装してよいのは、この3つだけ」と、**はっきり宣言**できます。
それ以外のクラスが勝手に `Shape` を実装しようとすると、コンパイルエラーになります。

```java
// 「Shape を実装してよいのは、Circle・Rectangle・Triangle だけ」と宣言
sealed interface Shape permits Circle, Rectangle, Triangle {}
```

`sealed`（シールド＝封をする）という言葉どおり、「**継承できる相手を、封じて限定する**」のです。

---

## なぜシールドクラスが便利なのか

「種類を限定する」と、何がうれしいのでしょうか。
最大の利点は、第26章のパターンマッチングと組み合わさったときに現れます。

```java
double area = switch (shape) {
    case Circle(double r)              -> r * r * 3.14;
    case Rectangle(double w, double h) -> w * h;
    case Triangle(double b, double h)  -> b * h / 2;
    // 3種類しかないので、これで「網羅」できている → default 不要！
};
```

`Shape` が「円・長方形・三角形の3つだけ」と分かっているので、その3つを `case` で挙げ尽くせば、コンパイラは「**すべてのケースをカバーできている**」と判断してくれます。
だから、`default`（「それ以外」）が不要になるのです。

さらに大きな利点があります。
もし将来、`Shape` に4つ目の図形（`Hexagon` など）を追加したら、この `switch` は「**新しいケースが処理されていない**」とコンパイルエラーになります。
つまり、「**新しい種類を追加したのに、対応を忘れる**」というバグを、コンパイラが防いでくれるのです（第18章の enum の switch と、同じ利点です）。

第27章は、次の5つの節で構成されています。

## この章を読み終えると

第27章を読み終えるころには、次のことができるようになっています。

- シールドクラスが「継承を限定するしくみ」だと理解できる
- `sealed` と `permits` で、継承できるクラスを限定できる
- 子クラスの3つの選択肢（`final`・`sealed`・`non-sealed`）がわかる
- シールドクラス＋パターンマッチングで、`default` なしの網羅的な分岐が書ける
- 種類の追加もれを、コンパイラに防いでもらえる

> **補足: シールドクラスは Java 17 から**
>
> シールドクラスは **Java 17** で正式機能になりました。
> 第17章のレコード、第26章のパターンマッチングと組み合わせることで、その真価を発揮します。
> この3つ（レコード・パターンマッチング・シールドクラス）は、まとめて「**データ指向プログラミング**」を支える機能と呼ばれます。

それでは、最初の節「シールドクラスとは」から始めましょう。
