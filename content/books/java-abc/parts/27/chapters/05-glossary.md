## 用語集 ― この章で学んだ言葉

第27章では、継承を限定する「シールドクラス」を学びました。
節ごとに、言葉を整理しておきます。

---

## シールドクラスの基本（第1〜2節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| シールドクラス | Sealed Class | 継承（実装）できるクラスを限定するしくみ（Java 17+） |
| `sealed` | シールド | 「継承を限定する」キーワード |
| `permits` | パーミッツ | 継承を許可するクラスを並べる |
| `final` | ― | 子クラスを「これ以上継承させない」 |
| `non-sealed` | ノンシールド | 子クラスの継承制限を解除する |

子クラス（ふつうのクラス）は `final`・`sealed`・`non-sealed` のどれかが必須。レコードは自動的に `final`。同じファイルなら `permits` 省略可（jshell では省略不可）。

---

## パターンマッチングとの組み合わせ（第3節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| 網羅性 | Exhaustiveness | すべての種類をカバーしているか |
| データ指向プログラミング | Data-Oriented Programming | レコード・シールド・パターンマッチングを組み合わせた設計 |

シールドクラスの switch は、全種類を挙げれば `default` 不要。追加もれはコンパイルエラーで分かる。

---

## おわりに

この章では、継承を限定する「シールドクラス」を学びました。

- **`sealed` / `permits`** で、継承できる種類を限定する
- 子クラスは **`final`**（レコードが手軽）
- **パターンマッチングと組み合わせる**と、`default` なしの網羅的な分岐ができ、追加もれも防げる
- **レコード・シールドクラス・パターンマッチング**の三位一体が、「データ指向プログラミング」を支える

第17章（レコード）・第26章（パターンマッチング）・第27章（シールドクラス）と、近年の Java を特徴づける機能が、ここでひとつにつながりました。
