## なぜ java.time か ― 古い API の問題

`java.time` の良さを理解するために、まず「古い日付 API（`Date`・`Calendar`）の、何が問題だったのか」を知っておきましょう。
歴史を知ると、`java.time` がなぜこう設計されているかが、腑に落ちます。

---

## 古い API の問題点

Java には、古くから **`java.util.Date`** と **`java.util.Calendar`** という日付クラスがありました。
ですが、これらには、初心者を何度も泣かせてきた、悪名高い問題がいくつもありました。

### 問題1 ― 月が「0から」始まる

`Calendar` では、月が **0から**数えられていました。1月が `0`、12月が `11` です。

```text
// 古い Calendar（混乱のもと）
カレンダーで「6月」を表すには … 5 を使う（6 ではない！）
```

「6月」を表すのに `5` と書く ―― これは、数えきれないほどのバグを生みました。
「12月（`11`）のつもりが `12` と書いて、勝手に翌年1月になっていた」といった事故が、後を絶ちませんでした。

### 問題2 ― 値を変更できてしまう（可変）

古い `Date` は、**あとから中身を変更できる**（可変、第13章）オブジェクトでした。
そのため、「ある日付を、別の場所で、知らないうちに書き換えられていた」という、追いにくいバグが起きがちでした。

### 問題3 ― 日付と時刻が、ごちゃまぜ

「日付だけ」「時刻だけ」を扱いたいのに、`Date` は、つねに日付と時刻の両方を持っていました。
「誕生日（日付だけでよい）」のような場面でも、不要な時刻が、ついて回りました。

---

## java.time の設計 ― 問題を解決する

**Java 8** で導入された `java.time` は、これらの問題を、すべて解決するように設計されています[^jsr310]。

| 古い API の問題 | java.time の解決 |
|---|---|
| 月が 0 始まり | **1月は 1**（直感どおり） |
| 値を変更できる（可変） | **不変**（変更できない。安全） |
| 日付と時刻がごちゃまぜ | **目的別のクラス**（日付だけ・時刻だけ・両方） |

それぞれ、見ていきましょう。

### 解決1 ― 月は「1から」

`java.time` では、月は**直感どおり**、1月が `1`、12月が `12` です。

```java
LocalDate d = LocalDate.of(2026, 6, 9);   // 2026年6月9日。6 はちゃんと「6月」
IO.println(d);
```

```text
2026-06-09
```

`6` と書けば、ちゃんと「6月」です。古い `Calendar` の `5` のような混乱は、もうありません。

### 解決2 ― すべて不変（イミュータブル）

`java.time` のクラスは、すべて**不変オブジェクト**（第13章）です。
一度作った日付は、絶対に変わりません。「3日後」を求めても、**元の日付は変わらず、新しい日付が返ります**（第3節でくわしく見ます）。
これにより、「知らないうちに書き換えられていた」という事故が、起こりえません。

### 解決3 ― 目的別のクラス

`java.time` は、目的に応じて、クラスが分かれています。

| クラス | 表すもの | 例 |
|---|---|---|
| `LocalDate` | **日付だけ** | 2026-06-09（誕生日など） |
| `LocalTime` | **時刻だけ** | 14:30（開店時刻など） |
| `LocalDateTime` | **日付＋時刻** | 2026-06-09T14:30（予約日時など） |

「誕生日」なら `LocalDate`、「開店時刻」なら `LocalTime`、「予約日時」なら `LocalDateTime`、というように、**ぴったりのクラス**を選べます。
不要な情報を持たずにすむので、間違いが減ります。

---

## まとめ

- 古い **`Date`・`Calendar`** には、「月が 0 始まり」「可変」「日付と時刻がごちゃまぜ」という問題があった
- **Java 8** の **`java.time`** は、これらをすべて解決した、モダンで安全な API
- `java.time` では、**月は 1 から**（1月＝`1`）、直感どおり
- すべて**不変オブジェクト**なので、知らないうちに書き換わる事故がない
- 目的別に、**`LocalDate`**（日付）・**`LocalTime`**（時刻）・**`LocalDateTime`**（両方）が分かれている

次の節では、これらの基本クラスを、実際に作って使ってみます。

[^jsr310]: JSR 310: Date and Time API, <https://jcp.org/en/jsr/detail?id=310>。Java SE 8（2014年）で導入。Joda-Time の設計思想を取り入れた新しい日時 API で、ISO 8601 ベースの不変クラス群（`LocalDate`、`LocalTime`、`LocalDateTime`、`ZonedDateTime`、`Instant`、`Duration`、`Period` など）を提供する。詳細は Java SE 25 API（<https://docs.oracle.com/en/java/javase/25/docs/api/java.base/java/time/package-summary.html>）参照。
