## 用語集 ― この章で学んだ言葉

第32章では、JUnit 5 を使ったユニットテストの書き方を学びました。
節ごとに、言葉を整理しておきます。

---

## ユニットテストの基礎（第1〜2節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| ユニットテスト | unit test | メソッドやクラスの単位を、自動で検証するテスト |
| AAA | Arrange / Act / Assert | テストの3ステップ（準備・実行・検証） |
| アサーション | assertion | 「こうあるはず」を断定する道具 |
| リグレッションテスト | regression test | 過去の動きを壊していないかを確かめるテスト |
| JUnit | ― | Java のテストフレームワーク。本書は JUnit 5（Jupiter） |
| JUnit Jupiter | ― | JUnit 5 の正式名称。ArtifactId は `junit-jupiter` |
| `@Test` | ― | テストメソッドを示すアノテーション |
| `import static` | ― | アサーションを短く書くための静的インポート |

---

## アサーション（第3節）

| アサーション | 用途 |
|---|---|
| `assertEquals(期待, 実際)` | 等しい |
| `assertEquals(期待, 実際, 誤差)` | 浮動小数点の許容誤差つき比較 |
| `assertNotEquals` | 等しくない |
| `assertTrue` / `assertFalse` | 真偽の検証 |
| `assertNull` / `assertNotNull` | null かどうか |
| `assertSame` / `assertNotSame` | 同一インスタンスかどうか（`==`） |
| `assertArrayEquals` | 配列の要素が等しい |
| `assertIterableEquals` | コレクションの要素が等しい |
| `assertLinesMatch` | 行単位で比較（正規表現可） |
| `assertAll(ラムダ群)` | 複数の検証をまとめて実行 |
| メッセージ引数 | 失敗時に表示する説明文 |

---

## ライフサイクル（第4節）

| 用語 | 意味 |
|---|---|
| `@BeforeEach` | 各テストの直前に毎回呼ばれる |
| `@AfterEach` | 各テストの直後に毎回呼ばれる |
| `@BeforeAll` | テストクラスで1回だけ最初に呼ばれる（`static` 必須） |
| `@AfterAll` | テストクラスで1回だけ最後に呼ばれる（`static` 必須） |
| `@DisplayName` | テスト結果に表示される名前を自由文字列で指定 |
| テスト独立性 | テスト同士が干渉せず、順序に依存しないこと |

---

## パラメータ化テスト（第5節）

| 用語 | 意味 |
|---|---|
| `@ParameterizedTest` | パラメータ化テストを示すアノテーション |
| `@ValueSource` | シンプルな1値を複数渡す |
| `@CsvSource` | 複数値をCSV風にまとめて渡す |
| `textBlock` | `@CsvSource` で表のように書くための属性 |
| `@MethodSource` | データ生成メソッドを参照する |
| `@CsvFileSource` | 外部 CSV ファイルからデータを読む |
| `Arguments.of(...)` | `@MethodSource` で複数引数を組み立てるヘルパー |

---

## 例外・タイムアウトと落とし穴（第6〜7節）

| 用語 | 意味 |
|---|---|
| `assertThrows(型, ラムダ)` | 例外が投げられるはず |
| `assertDoesNotThrow(ラムダ)` | 例外が投げられないはず |
| `assertTimeout(時間, ラムダ)` | 指定時間内に終わるはず（最後まで実行） |
| `assertTimeoutPreemptively(時間, ラムダ)` | 超えたら別スレッドで即中断 |
| `@TempDir` | テストごとに新品の一時ディレクトリを割り当てる |
| `@Disabled("理由")` | テストを一時的に無効化する |

---

## おわりに ― テストを「書ける人」になりました

第32章で、Java プロジェクトに**ユニットテスト**を書く力が、ひととおり身につきました。
これで、

- `mvn test` / `./gradlew test` で、書いたテストが自動実行される
- 期待値と実際値を、アサーションで明示的に比べられる
- 同じロジックを複数入力で検証できる
- 例外や性能の上限も、テストで担保できる

という状態になりました。

ところが、実務のテストには、もうひとつ大きな壁があります。
それは、「**他のクラスや外部システムに依存する処理を、どうテストするか**」です。

たとえば、DB から取ってきたユーザー情報を加工するメソッドをテストするとき、本物の DB に繋ぎたくはありません。
そんなときに使うのが、**モック**（mock）と呼ばれる**偽物**を作って差し込む技術です。

次の第33章では、**Mockito** というモック作成ライブラリを使って、依存を切り離す方法を学んでいきます。
