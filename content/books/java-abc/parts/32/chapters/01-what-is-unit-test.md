## ユニットテストとは

「**ユニット**」とは、英語で「**単位**」のことです。
プログラムでは、ふつう **1 つのメソッド**、または小さな**クラス**が、その「単位」にあたります。

**ユニットテスト**（unit test）は、そのいちばん小さな単位ひとつひとつが、**期待どおりに動いているか**を確かめるテストです。

---

## ユニットテストの 3 ステップ

ユニットテストの中身は、いつも次の 3 ステップでできています。

1. **準備**（Arrange） … テスト対象や入力を用意する
2. **実行**（Act） … 対象のメソッドを呼ぶ
3. **検証**（Assert） … 戻り値や状態が、期待どおりかを比べる

これは英語で **AAA**（Arrange / Act / Assert）の三文字で語呂よく覚えられます。
よいテストは、見ただけでこの 3 つに分けて読めるように書かれています。

たとえば、足し算をするメソッド `add(2, 3)` をテストするなら、

```java
@Test
void addsTwoNumbers() {
    Calculator calc = new Calculator();         // 準備
    int result = calc.add(2, 3);                // 実行
    assertEquals(5, result);                    // 検証
}
```

このように、3 行に明確に分かれます。
慣れないうちは、コメントで `// Arrange` `// Act` `// Assert` と書いておくのも、よい練習になります。

---

## 「期待どおり」の意味 ― 期待値と実際値を比べる

検証ステップで使う `assertEquals(5, result)` は、ひとことで言えば

> 「**5（期待値）**と **`result`（実際値）**は、等しいか？」

を確認する命令です。

- **等しければ** … 何も起きません。テスト成功
- **等しくなければ** … その場でテストを失敗扱いにし、エラーを表示します

これを、**アサーション**（assertion、断定）と呼びます。
ユニットテストとは、**「こうあるはず」を、コードで断定すること**だと言ってもいいでしょう。

---

## 自動テストの威力 ― 同じことを何百回でも

自分で書いたコードを「動かして確認」するだけでも、最初は十分に思えるかもしれません。
ですが、自動テストには、手作業では絶対にできないことが**3 つ**あります。

### 1. すぐ・何度でも実行できる

`mvn test` ・ `./gradlew test` で、何百ものテストが**数秒〜数分**で終わります。
朝のコード修正の後でも、お昼休みの前でも、退勤前でも、**いつでもボタンひとつ**で、すべての動きを再確認できます。

### 2. 修正しても、これまでの動きが壊れていないか確かめられる

たとえば、`add(int, int)` を、性能のために少し書き換えたとします。
新しい実装でも、`add(2, 3)` が `5` を返してくれるか、`add(-1, 1)` が `0` を返してくれるか ―― これを自動で確かめてくれます。
壊れていたら、テストが**失敗を教えてくれる**ので、すぐに気づけます。
このように、過去の動きを壊していないかを確認することを、**リグレッションテスト**（回帰テスト）と呼びます。

### 3. テスト自体が「仕様の説明書」になる

`add(2, 3) == 5`、`divide(10, 0)` は例外を投げる、`isPositive(-1) == false` ……。
こうしたテストが大量に並んでいると、それを読むだけで「**このクラスが何をするのか**」が伝わります。
ドキュメントは古くなりますが、**テストが通っているかぎり、テストは正確**です。

---

## いつテストを書くのか

実務でテストを書くタイミングは、いくつかあります。

| タイミング | やり方の名前 | コメント |
|---|---|---|
| 実装が終わってから | 後付け（あとからテスト） | 入門段階はまずこれでOK |
| テストを先に書いて、実装する | テスト駆動開発（TDD） | 上級者向け。実装の指針に使える |
| 不具合が見つかったとき | 再現テスト | 不具合を**先に**テスト化してから修正する |

本章では、いちばん基本的な「実装してから、テストを書く」流れで学んでいきます。

---

## 「ユニットテストでできないこと」もある

最後に、ユニットテストが**苦手なこと**にも触れておきます。

| 種類 | 何をテストするか |
|---|---|
| **ユニットテスト** | 1 つのメソッド / クラスの中身 |
| **結合テスト**（integration） | 複数のクラスやモジュールを連携させた動き |
| **E2E テスト**（end-to-end） | 画面操作からデータベースまで、システム全体の動き |
| **負荷テスト**（load） | 大量アクセス時の性能・安定性 |

ユニットテストでできるのは、いちばん小さな単位のことだけです。
全部のテストを「ユニットテストにしよう」とすると無理が出ます。
それぞれを、組み合わせて使うのが実務の基本です。
ですが、いちばん下にユニットテストの厚い土台があってこそ、上の層のテストも意味を持ちます。

---

## まとめ

- ユニットテストは、**1 つのメソッド / クラス**を確かめる、いちばん小さなテストです
- 中身は **準備 → 実行 → 検証（AAA）** の 3 ステップ
- **アサーション**で、「こうあるはず」を断定して比べます
- 自動テストは、何度でも・速く・正確に、過去の動きを守ってくれます
- それでもユニットテストにできないこと（結合・E2E・負荷）もあります
