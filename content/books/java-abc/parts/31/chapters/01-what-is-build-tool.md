## ビルドツールとは

まずは、「ビルドツールが、いったい何をしてくれるのか」を、はっきりさせるところから始めます。

ひとことで言うと、

> **ビルドツール**は、Java プロジェクトの**お世話をぜんぶ自動でやってくれる道具**

です。
お世話とは、つぎのようなことです。

- 依存ライブラリを、インターネットから取ってくる
- ソースコードを、`javac` でコンパイルする
- テストを、まとめて実行する
- 配布用の `.jar` ファイルを、作る
- 必要なら、サーバーとして起動する

これらを、**ファイル 1 つ**（Maven なら `pom.xml`、Gradle なら `build.gradle.kts`）に書いておけば、**1 つのコマンド**で全部やってくれる ―― それが、ビルドツールです。

---

## まずは、手作業の `javac` を思い出す

第3章で、はじめてのプログラムを書いたときのことを思い出してください。
`Hello.java` を書き、コマンドラインで `javac` と `java` を叩いて、`Hello, World!` を表示しました。

```text line-numbers=false
$ javac Hello.java
$ java Hello
Hello, World!
```

クラスが 1 つだけなら、これで十分です。
ですが、現実のプログラムは、もっと複雑です。

たとえば、こんなプロジェクトを想像してみましょう。

- `src/` ディレクトリの下に、Java ファイルが **120 個**ある
- そのうちのいくつかは、**JSON ライブラリ**（Jackson）を使っている
- いくつかは、**ログ出力ライブラリ**（SLF4J）を使っている
- `test/` ディレクトリには、**70 個のテスト**がある
- 配布するときは、これらを **1 つの JAR ファイル**にまとめたい

これを `javac` だけでやろうとすると、どうなるでしょうか。

```text line-numbers=false
$ javac -cp jackson-databind.jar:jackson-core.jar:jackson-annotations.jar:slf4j-api.jar \
        -d build/classes \
        src/com/example/foo/Foo.java \
        src/com/example/foo/Bar.java \
        src/com/example/foo/Baz.java \
        ... （これが120行つづく）
```

しかも、**Jackson や SLF4J の `.jar` ファイルは、自分で探してダウンロードしないといけません**。
さらに、Jackson はバージョンごとに別ファイル ―― 新しいバージョンを使いたければ、また探し直し。

これを毎日、毎回、手でやる気にはなれませんよね。
**この煩雑さから解放してくれるのが、ビルドツール**です。

## ビルドツールがやってくれること ― 4つの仕事

ビルドツールの仕事を、4つに分けて見ていきましょう。

### 1. 依存ライブラリの管理

「Jackson の 2.18.0 を使いたい」と**設定ファイルに書く**だけで、ビルドツールがインターネットから自動でダウンロードし、コンパイル時に勝手に使ってくれます。
`.jar` ファイルを探したり、コピーしたり、`-cp`（クラスパス）に並べる手間は、もういりません。

### 2. コンパイル

ソースコードのフォルダ（`src/main/java`）を、自動で読み取り、すべての `.java` ファイルを `javac` でコンパイルします。
順序を考えるのも、すべてビルドツールの仕事です。

### 3. テストの実行

`src/test/java` の下にあるテストを、**まとめて自動で実行**します。
テストが失敗したら、ビルドを失敗扱いにします。
これにより、「テストが通らないコードを配布してしまう」事故を防げます。

### 4. パッケージング

コンパイル済みのクラスや、リソースファイル（設定ファイル・画像など）を、**1 つの JAR ファイル**にまとめます。
Web アプリならサーバーへ配布できる WAR / JAR、ライブラリなら他のプロジェクトから使える JAR と、用途に応じた形式で出力できます。

## なぜ「設定」だけで動くのか ― 規約と慣習

ビルドツールが少ない設定で動く理由は、「**規約による設定**」（Convention over Configuration）という考え方を採用しているからです。

たとえば、

- ソースコードは **`src/main/java/`** に置く
- テストは **`src/test/java/`** に置く
- 設定ファイルは **`src/main/resources/`** に置く

という**お約束（規約）**を守っていれば、「ここがソースです」「ここがテストです」と毎回設定する必要がなくなります。
このお約束は、Maven が先に決めて世の中に広まり、Gradle もそのまま採用しているため、**Java プロジェクトのほぼ標準**になっています。

```text line-numbers=false
my-project/
├── pom.xml                    ← Maven の設定ファイル
└── src/
    ├── main/
    │   ├── java/              ← 本番コード
    │   └── resources/         ← 設定ファイル
    └── test/
        ├── java/              ← テストコード
        └── resources/
```

このお約束さえ守れば、`mvn package` の 1 コマンドで、コンパイルからテスト実行、JAR の作成まで、全部やってくれます。

## Maven と Gradle ― 2 つの選択肢

Java の世界で、ビルドツールといえば**ほぼこの 2 つ**です。

| ビルドツール | 登場年 | 設定ファイル | 書き方の特徴 |
|---|---|---|---|
| **Maven** | 2004 年 | `pom.xml` | XML で書く。お約束が厳しいぶん、迷いにくい[^maven] |
| **Gradle** | 2007 年 | `build.gradle.kts`（または `build.gradle`） | Kotlin（または Groovy）で書く。柔軟で速い[^gradle] |

どちらも、同じ Maven Central というライブラリ倉庫（次節）からライブラリを取ってきます。
**「やってくれること」は、ほぼ同じ**です。

ですので、両方さわってみて、現場で使っているほう・自分が好きなほうを選べばよい、というのが基本姿勢です。
本章では、まず**両方の感触をつかむこと**を目指します。

## まとめ

- **ビルドツール**は、依存解決・コンパイル・テスト・パッケージングを**自動化**する道具です
- 設定ファイル 1 つ（`pom.xml` / `build.gradle.kts`）に書くだけで、1 コマンドで動きます
- 「規約による設定」のおかげで、最小限の設定でプロジェクトが動きます
- Java 界では **Maven** と **Gradle** が二大ビルドツールです

次の節では、ビルドツールの心臓部 ―― **依存関係**と**リポジトリ**のしくみを見ていきます。

[^maven]: Apache Software Foundation, "Apache Maven Project," [https://maven.apache.org/](<https://maven.apache.org/>)。Maven は2004年に Apache プロジェクトとして公開された Java 向けビルドツールで、`pom.xml` による宣言的構成と「規約より設定」（Convention over Configuration）の方針を採用している。

[^gradle]: Gradle Inc., "Gradle Build Tool," [https://gradle.org/](<https://gradle.org/>)。Gradle は2007年に登場した Groovy／Kotlin DSL によるビルドツールで、漸進的ビルド（incremental build）とビルドキャッシュにより Maven より高速な実行を実現する。Android 標準のビルドツールとしても採用されている。
