## 依存関係とリポジトリ

前の節で、ビルドツールが**依存ライブラリを自動で取ってきてくれる**ことを学びました。
この節では、その**しくみ**を、もう少し詳しく見ていきます。

ここを理解しておくと、Maven でも Gradle でも、**設定ファイルが読めて書ける**ようになります。

---

## 「ライブラリを使う」とは何か

ライブラリ（library）とは、**他の人が書いてくれた、便利な道具のかたまり**です。
たとえば、

- **Jackson** … JSON データを読み書きする
- **SLF4J / Logback** … ログを出力する
- **JUnit** … テストを書いて実行する
- **Spring Boot** … Web アプリの土台

これらはすべて、**JAR ファイル**として配布されています。
JAR は、コンパイル済みのクラスファイル（`.class`）をひとまとめにした、Java の世界の**圧縮パッケージ**です。

「ライブラリを使う」とは、

1. その JAR ファイルを手に入れて
2. コンパイル時にクラスパス（`-cp`）に含めて
3. 自分のコードから `import` する

という、3ステップのことです。
ビルドツールは、**1 と 2 を完全に自動化**してくれます。あなたは **3 だけ**書けばよい、ということです。

---

## 依存関係 ― 依存ライブラリの「住所」

ビルドツールに「このライブラリを使いたい」と伝えるには、**3 つの情報**が必要です。
これを **GAV 座標**（GroupId・ArtifactId・Version）と呼びます。

| 要素 | 読み方 | 役割 | 例 |
|---|---|---|---|
| **GroupId** | グループ ID | 提供している組織（逆ドメイン名で書く） | `com.fasterxml.jackson.core` |
| **ArtifactId** | アーティファクト ID | ライブラリの名前 | `jackson-databind` |
| **Version** | バージョン | バージョン番号 | `2.18.0` |

この 3 つで、世界中のどのライブラリも**一意に特定できる住所**になります。
たとえば、Jackson の JSON データバインダーを使いたければ、

> 「`com.fasterxml.jackson.core` の `jackson-databind` のバージョン `2.18.0`」

と指定するわけです。

Maven の `pom.xml` だとこう書きます。

```xml
<dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
    <version>2.18.0</version>
</dependency>
```

Gradle の `build.gradle.kts` だとこう書きます。

```kotlin
dependencies {
    implementation("com.fasterxml.jackson.core:jackson-databind:2.18.0")
}
```

書き方は違いますが、**指している情報は同じ**です。
`groupId:artifactId:version` という形は、Gradle・Maven 以外でもよく見かけます。

## リポジトリ ― ライブラリの「倉庫」

依存関係を書いただけでは、まだライブラリは手元にきません。
ビルドツールは、**リポジトリ**（repository）と呼ばれる「ライブラリの倉庫」から JAR をダウンロードします。

リポジトリには 3 種類あります。

### 1. リモートリポジトリ ― **Maven Central**

Java の世界で、**事実上の標準**となっている公開リポジトリが、**Maven Central**（メイヴン・セントラル）です。
URL は [https://repo.maven.apache.org/maven2/](https://repo.maven.apache.org/maven2/) です。

世界中のオープンソースライブラリが、ここに集約されています。
Jackson も、JUnit も、Spring Boot も、すべて Maven Central から配信されています。

Maven と Gradle は、**何も設定しなくても**この Maven Central を見にいくように初期設定されています。
だから、`pom.xml` に `jackson-databind` と書いただけで、勝手にダウンロードされるのです。

> **補足: [https://central.sonatype.com/](https://central.sonatype.com/) で検索できる**
>
> Maven Central のライブラリは、[https://central.sonatype.com/](https://central.sonatype.com/) で検索できます。
> 「使いたいライブラリ名」を入れて、最新版の GAV をコピーする、というのが現場の定番ワークフローです。

### 2. ローカルリポジトリ ― 自分のマシンにあるキャッシュ

1度ダウンロードしたライブラリは、自分のマシンにキャッシュされます。
Maven の場合は **`~/.m2/repository/`**、Gradle の場合は **`~/.gradle/caches/`** に保存されます。

2回目以降のビルドは、ここから読み込むので、すぐに終わります。
ネットがない状態でも、すでにキャッシュにあるライブラリなら使えます。

たとえば Maven なら、`~/.m2/repository/com/fasterxml/jackson/core/jackson-databind/2.18.0/` の下に、JAR が置かれます。
GroupId をドットで分けた階層になっているのがポイントです。

### 3. プライベートリポジトリ ― 社内ライブラリのための倉庫

企業の中では、Maven Central に公開できない**社内専用ライブラリ**を、自社のリポジトリで管理することがよくあります。
代表的なソフトウェアは、**Nexus** や **Artifactory** です。

入門段階では「そういうものがある」と知っておく程度で十分です。

## 推移的依存関係 ― 依存の依存も、自動で連れてくる

ここからは、ビルドツールの**便利だけれど、ちょっとトリッキー**なしくみの話です。

たとえば、`jackson-databind` を依存に追加すると、実は **`jackson-core` と `jackson-annotations`** も、いっしょに自動でダウンロードされます。
なぜなら、`jackson-databind` 自身が、これら 2 つを内部で使っているからです。

このような **「依存の依存」を、自動でたどって取ってきてくれる**しくみを、**推移的依存関係**（transitive dependencies）と呼びます。

```text
あなたのプロジェクト
  └── jackson-databind:2.18.0
        ├── jackson-core:2.18.0           ← 推移的に追加
        └── jackson-annotations:2.18.0    ← 推移的に追加
```

これがあるおかげで、あなたは `jackson-databind` 1 行を書くだけで、必要な JAR がすべてそろいます。
昔の `javac -cp` ではこれを手で並べないといけませんでした。
**「ビルドツールが大変ありがたい」のは、まさにここなのです。**

## バージョンの指定 ― 固定するか、新しいものを使うか

依存関係の **`<version>`** には、いくつかの書き方があります。

| 書き方 | 意味 | 評価 |
|---|---|---|
| `2.18.0` | このバージョンに固定 | **基本これにする**。再現性が保てる |
| `[2.18.0,)` | 2.18.0 以上、上限なし | 危険。勝手にバージョンが上がる |
| `LATEST` / `RELEASE` | 最新版を使う | **使わない**。Maven 3 では非推奨 |

実務では、**`2.18.0` のように固定して書く**のが基本です。
理由は、

- 「昨日まで動いていたのに、今日ビルドしたら壊れた」を防ぐため
- チーム全員が、**同じバージョン**でビルドできるようにするため

「バージョンを上げる」ことは、本来、明示的な判断であるべきです。
勝手に上がる設定は避けましょう。

---

## まとめ

- ライブラリは **GroupId・ArtifactId・Version** の 3 つで特定します
- ビルドツールは、**Maven Central**（公開リポジトリ）からライブラリを取ってきます
- 1 度ダウンロードしたライブラリは、**ローカルリポジトリ**（`~/.m2`・`~/.gradle/caches`）にキャッシュされます
- **推移的依存関係**のおかげで、依存の依存も自動でついてきます
- バージョンは**固定指定**するのが基本です

次の節からは、いよいよ**実際に Maven を動かして**プロジェクトを作っていきます。
