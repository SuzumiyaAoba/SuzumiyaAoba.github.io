## はじめに ― この章で学ぶこと

第29章で、並行処理を学びました。
そのとき、「**複数のスレッドが同じデータを触ると、思わぬことが起きる**」という現象を、`counter++` の例で見ました。

第43章では、その「思わぬこと」が **なぜ・どのように**起きるのか ―― そして、それを**どう防ぐか**を、根本から扱います。
キーワードは 4 つです。

- **メモリモデル**（Java Memory Model、JMM）
- **happens-before** 関係
- **`volatile`** と **`synchronized`**
- **`java.util.concurrent.atomic`**（CAS）

これらは、Java の中でも特に**抽象度が高く**、初学者がつまずきやすい領域です。
ですが、これを理解せずに並行処理のコードを書くと、**動いているように見えて、本番で稀に壊れる**コードを生んでしまいます。

---

## 「タイミングで壊れる」とは

第29章のおさらいです。次のコードを 2 つのスレッドで同時に動かすと、どうなるでしょう?

```java
int counter = 0;
// Thread A: counter++ を 100000 回
// Thread B: counter++ を 100000 回
```

期待値は 200000、ですが実際は **200000 未満になることがある**。
理由は、`counter++` が、CPU レベルでは「**読む → 1 足す → 書く**」の 3 段階に分かれていて、2 つのスレッドの操作が**かみ合ってしまう**からです。

この章で扱うのは、これより**もっと深い**話です。
たとえば、次のような疑問。

- **書いた値が、他のスレッドからすぐ見えるのか?**
- **JIT は、コードの実行順を勝手に入れ替えていないか?**
- **CPU は、書き込みをキャッシュに溜めて、後から書くことがあるのではないか?**

答えはすべて「**そういうことが起きる**」です。
そして、それを**どこまで気にすべきか**を決めているのが、**Java メモリモデル**（JMM）です。

---

## 「動いた／動かない」のあいだに

並行処理のバグは、**多くの場合、テストでは見えません**。

| 単スレッドのバグ | 並行処理のバグ |
|---|---|
| 同じ入力で**同じ結果** | 同じ入力で**毎回違う結果**になりうる |
| `System.out.println` で原因が辿れる | デバッガが**バグを隠す**（タイミングが変わる） |
| ユニットテストで再現 | ユニットテストでは**1 回も再現しない**こともある |
| 動けば動く | 「動いているように見える」が、たまに壊れる |

「**たまに壊れる**」コードは、ふつうの単スレッドの感覚で書くと、簡単に量産できてしまいます。
これを防ぐには、「**スレッド間で値はどう見えるか**」という**ルール**を、まず頭に入れることが必要です。
そのルールこそが、**メモリモデル**です。

---

## この章で学ぶこと

第43章は、次の8つの節で構成されています。

| 節 | タイトル | 内容 |
|---|---|---|
| 1 | なぜメモリモデルが必要か | 並行処理の落とし穴と「順序」の問題 |
| 2 | happens-before 関係 | 「先に起きた」を保証する関係 |
| 3 | volatile | 可視性を保証する最小コスト |
| 4 | synchronized と内部ロック | 排他制御と happens-before |
| 5 | atomic と CAS | ロックなしで安全に値を変える |
| 6 | 不変性とスレッド安全性 | 「変えない」が最強の戦略 |
| 7 | よくあるつまずき | 二重チェックロック、`Thread.stop` など |

前半（1〜2節）で、JMM の核となる**順序の保証**を理解します。
中盤（3〜5節）で、`volatile`・`synchronized`・atomic を**使い分け**られるようにします。
後半（6〜7節）で、設計レベルの**スレッド安全性**を考えます。

---

## この章を読み終えると

第43章を読み終えるころには、次のことができるようになっています。

- 「**書いた値が見えない**」が、Java でなぜ起きうるかを説明できる
- **happens-before** 関係を使って、コードの安全性を判断できる
- `volatile` と `synchronized` の**使い分け**ができる
- `AtomicInteger` などの **CAS ベースの操作**を、適切な場面で選べる
- **不変オブジェクト**を、スレッド安全な設計の柱に据えられる
- 「**二重チェックロック**」のような有名な落とし穴を避けられる

並行処理の話題に不安があれば、第29章を先に読み返してから戻ってきてください。
それでは、最初の節「**なぜメモリモデルが必要か**」から始めましょう。
