## `volatile` ― 可視性を保証する

`volatile` は、フィールドに 1 つキーワードを付けるだけの、最も**軽量な同期手段**です。
ですが、その効果は侮れません。
この節では、`volatile` で**何が保証され**、**何は保証されない**かを、はっきりさせます。

---

## 構文と意味

`volatile` は、フィールド宣言に付けます。

```java
class Flag {
    volatile boolean stopped = false;
}
```

これだけで、`stopped` フィールドに対して、JMM は次の 2 つを保証します。

1. **可視性**: あるスレッドの書き込みは、**他のスレッドから即座に見える**
2. **順序**: `volatile` 書き込み**より前**のすべての書き込みは、`volatile` 読み込み**より後**から見えたとき、**完了している**

2 つめは、第2節の「**volatile による happens-before**」のことです。
`volatile` の書き込みと読み込みのあいだに、**happens-before の橋**がかかります。

---

## 「止まらないループ」の修正

第1節の冒頭で見た「永久に止まらないループ」を、`volatile` で直してみましょう。

```java
class Worker implements Runnable {
    volatile boolean stopped = false;   // ← volatile を付けた

    public void run() {
        while (!stopped) {
            // 何か仕事
        }
    }
    public void stop() {
        stopped = true;
    }
}
```

これで、別のスレッドが `worker.stop()` を呼べば、`worker.stopped = true` は**ループスレッドから即座に見え**、ループは確実に止まります。
`volatile` のおかげで、

- キャッシュに値が**置きっぱなし**にならない（メインメモリに書く）
- JIT が値を**ホイスト**（ループの外に出す）できない
- 並べ替えが**制限**される

ようになりました。

たった 1 つのキーワードで、不気味な「**動いてるけど止まらない**」バグが直る、というのは小気味よい話です。

---

## `volatile` で**できない**こと ―― アトミシティ

ですが、`volatile` には**重要な制限**があります。それは、

> **複合操作はアトミックにならない**

ということです。
たとえば、次のコードを `volatile` だけで安全にすることは、**できません**。

```java
volatile int counter = 0;
counter++;   // ← これは依然としてレース条件
```

`counter++` は内部で「**読む → 加算 → 書く**」の 3 ステップです。
`volatile` は「**個々の読み・書き**」をアトミックにしますが、**3 ステップを 1 つにまとめる効果はありません**。

「**読んでから次に書くまでのあいだ**」に、別スレッドが割り込んで書いてしまう可能性は、`volatile` では防げないのです。
これを直すには、**`synchronized`**（第4節）か、**`AtomicInteger`**（第5節）を使います。

---

## `volatile` の効く場面・効かない場面

具体的に整理してみます。

| 操作 | `volatile` で安全? |
|---|---|
| `bool flag = true;`（書き込み） | **OK** |
| `if (flag) ...`（読み込み） | **OK** |
| `state = new State(...);`（参照の差し替え） | **OK** ―― 新しい状態オブジェクトに**差し替える**だけ |
| `counter++`（読み→計算→書き） | **NG** ―― レース条件 |
| `if (x == null) x = ...;`（テスト→セット） | **NG** ―― レース条件 |
| `long timestamp = ...;`（64bit 代入） | `volatile` **必須**（さもないと半分書きされる可能性） |
| `volatile int[] arr;`（配列全体） | **NG** ―― 配列の参照は保護されるが、**要素**は別の話 |

最後の「配列」は、よく混乱するポイントです。
`volatile int[] arr;` は「**`arr` がどの配列を指しているか**」を保証しますが、`arr[i] = ...;` のような**要素の書き込み**は、`volatile` の保護対象外です。
配列の要素を安全に共有したいなら、`AtomicIntegerArray` を使うのが定石です（第5節）。

---

## `volatile` のコスト

「便利なら全部 `volatile` にすればいいのでは?」 ―― 答えは「**やめておけ**」です。

`volatile` は、ふつうのフィールドアクセスより**コストが高い**です。具体的には、

- 書き込み: **メモリバリア命令**が挿入される（CPU のキャッシュとメインメモリを同期）
- 読み込み: **キャッシュからの値を信用しない**ので、メインメモリを見にいく
- JIT の最適化機会が**減る**（ホイストや並べ替えが禁止される）

ふつうの int の読み書きが 1 ns 以下なのに対し、`volatile` の読み書きは数 ns〜十数 ns 程度のオーバーヘッドがあります（実機依存）。
1 回 1 回は小さくても、**ループの中で多用**すると、それなりに効いてきます。

そして、「**ほとんどのフィールド**は、シングルスレッドで触られるか、設定 1 回だけ」です。
無闇に `volatile` を付けるのは、

- 性能を落とすだけ
- 「**ここは並行アクセスがある**」というコードの意図を**曖昧にする**

の二重で良くありません。
**「ここは並行で触る」と意識的な場所だけ**、`volatile` を付ける ―― それが**読み手にも親切な**書き方です。

## 典型例: ダブルバッファとフラグ

実務で `volatile` がぴったり当てはまるのは、次のようなケースです。

### 1. 停止フラグ・状態フラグ

すでに見たループの停止や、`READY`・`SHUTDOWN` などの状態管理。

```java
volatile State state = State.RUNNING;
```

### 2. 1 回きりの初期化フラグ

オブジェクトの初期化完了を知らせるフラグ。

```java
private volatile boolean initialized = false;

void init() {
    setup();
    initialized = true;   // ← この時点で setup() の効果が他スレッドから見える
}

boolean isReady() {
    return initialized;
}
```

`initialized = true` の前に行ったすべての書き込み（`setup()` 内の状態セットアップ）が、`initialized == true` を観測した他スレッドから**全部見える**ようになります。
これは、第2節で見た **happens-before の橋**が架かっているからです。

### 3. 参照の入れ替え（不変オブジェクトの差し替え）

「**設定オブジェクト**」を、リロード時に丸ごと差し替えるパターン。

```java
volatile Config currentConfig = Config.defaults();

void reload() {
    currentConfig = loadFromDisk();   // 参照の差し替えだけ
}

Config get() {
    return currentConfig;
}
```

新しい `Config` の中身は**不変**として、差し替えだけ `volatile` で原子化する。
読み手はロックなしで最新の設定を見られるし、書き手はリロードのときだけ参照を更新する。
これは、後の第6節「**不変性とスレッド安全性**」と組み合わせて、よく使うイディオムです。

---

## まとめると

- `volatile` は、**フィールドの可視性**を保証する最も軽量な手段
- `volatile` 書き込み → `volatile` 読み込みで **happens-before の橋**がかかる
- ただし、**`counter++` のような複合操作はアトミックにならない**
- 配列フィールドの `volatile` は、**参照**だけ保護、**要素**は保護しない
- コストはあるので、無闇に付けず、「**並行で触る**」と意識的な場所だけに使う
- 典型例は **停止フラグ・初期化フラグ・参照の差し替え**

次の節では、より強力で複合操作も守れる手段 ―― **`synchronized`** を見ていきます。
