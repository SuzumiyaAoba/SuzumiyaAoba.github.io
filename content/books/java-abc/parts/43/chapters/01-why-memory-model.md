## なぜメモリモデルが必要か ― 並行処理の落とし穴

並行処理のバグには、第29章で見た「**競合状態**」のほかに、さらに**根が深い**問題があります。
それが、「**書いた値が他のスレッドに見えない**」「**実行順が入れ替わって見える**」という、**順序と可視性**の問題です。
この節では、そもそも何が起きうるのかを、3 つの観点で整理します。

---

## 観点1: 可視性（visibility）

「**書いた値が、他のスレッドにいつ見えるのか?**」 ―― これが**可視性**の問題です。

次のコードを見てください。

```java
class Worker implements Runnable {
    boolean stopped = false;

    public void run() {
        while (!stopped) {
            // 何か仕事
        }
    }
}
```

別のスレッドから `worker.stopped = true;` と書き換えれば、ループは止まるはずです ―― **理屈の上では**。
ですが実際には、**永久に止まらないことがあります**。
なぜでしょうか?

理由は、**CPU と JIT がメモリを「直接」読み書きしない**ことにあります。

```text line-numbers=false
[Thread A] CPU コア1       [Thread B] CPU コア2
   │                              │
   │ ┌─キャッシュ─┐                │ ┌─キャッシュ─┐
   │ │ stopped: F │                │ │ stopped: T │ ← 書き込んだ
   │ └────────────┘                │ └────────────┘
   │                              │
   └──────► メインメモリ ◄──────────┘
              │
              │ いつ同期されるかは保証なし
```

スレッド B が `stopped = true` と書いても、その値は **B のキャッシュ**に書かれるだけ。
メインメモリへの反映や、A のキャッシュの更新は、**いつ起きるかわかりません**。
A は古い `stopped = false` をキャッシュに持ち続け、ループから抜けない、ということが**理論上ありうる**のです。

しかも、JIT がコードを最適化する過程で、

```java
while (!stopped) { ... }
```

を

```java
boolean local = stopped;
while (!local) { ... }
```

と書き換えてしまう（**ホイスト**、hoisting と呼ぶ最適化）こともあります。
こうなると、**他スレッドがどう書き換えても**、もう永久に見えません。

可視性の問題は、ハードウェアと最適化の両方が原因になりうる ―― これが Java 並行処理の最初の落とし穴です。

---

## 観点2: 順序（ordering） / 並べ替え（reordering）

「**書いた順番どおりに、他のスレッドから見えるのか?**」 ―― これが**順序**の問題です。

次のコードを考えます。

```java
class Pair {
    int x = 0;
    int y = 0;

    // Thread A
    void setOnA() {
        x = 1;        // ①
        y = 2;        // ②
    }

    // Thread B
    void readOnB() {
        if (y == 2) {           // ③
            System.out.println(x);   // ④
        }
    }
}
```

Thread B で `y == 2` が見えたなら、`x` は `1` のはず ―― ふつうはそう考えます。
ですが、Java の仕様では、

- **JIT** は ①と② を**並べ替える可能性**がある（依存関係がなければ）
- **CPU** は、書き込みを**並べ替えて**メモリに反映することがある
- **キャッシュ**の更新タイミングが**コア間で異なる**

ことから、Thread B から見ると「**`y == 2` なのに `x == 0`**」が起きうるのです。
④で `0` が出力されることがある、という不気味な現象です。

「**そんなバカな**」と思うでしょう。
ですが、これは**バグではなく仕様**です。Java は、**並行処理に明示的な同期がない**コードに対して、**特定の順序を保証していません**。
パフォーマンスのために、JIT・CPU・キャッシュが**勝手に並べ替える権利**を持っているのです。

---

## 観点3: アトミシティ（atomicity）

第29章で見た `counter++` の問題は、3 つめの観点 ―― **アトミシティ**（不可分性）に関わります。

```java
counter++;     // 内部では: counter = counter + 1
```

これは見た目 1 ステップですが、内部では「**読む → 加算 → 書く**」の 3 ステップに分かれています。
2 つのスレッドがほぼ同時に走ると:

```text line-numbers=false
Thread A: counter を読む（= 0）
Thread B: counter を読む（= 0）
Thread A: 0 + 1 = 1 を書く（counter = 1）
Thread B: 0 + 1 = 1 を書く（counter = 1）   ← Thread A の書き込みが「上書き」された
```

期待値は 2 なのに、結果は 1。
これは「**3 ステップが、途中で他スレッドに割り込まれずに 1 つの操作として完了する**」ことが保証されていないからです。

「**割り込まれずに完了する**」のが**アトミック**な操作です。
プリミティブの `++` や `+=` は、**アトミックではありません**。

> **`long` と `double` の落とし穴**
>
> JMM 仕様では、`int`・`reference`・`boolean` の代入はアトミックですが、`long` と `double`（64 bit）は、**実装によっては 2 回の 32 bit 書き込みに分割される**ことが許されています。
> つまり、`long x = ...;` のような代入で、別スレッドが**書き込みの途中の値**を読む可能性が理論上あります。
> 実機ではほぼ起きませんが、**`volatile long`** にすると確実にアトミックになります。

---

## 3 つを束ねるのが「メモリモデル」

ここまで見た 3 つの観点 ―― **可視性・順序・アトミシティ** ―― は、すべて「**スレッド間で、データはどう見えるか**」という同じ問題の別の顔です。
これらを 1 つのルールで定めたものが、**Java メモリモデル**（Java Memory Model、JMM）です。

JMM は、Java 5（JSR-133、2004 年）で初めて厳密に定義されました。
ざっくり言うと、JMM は次のことを定めています。

| 何を定めるか | 内容 |
|---|---|
| **どこから何が見えるか** | あるスレッドの書き込みは、どんな条件で他のスレッドから見えるか |
| **何が並べ替えられるか** | JIT・CPU は、どんな並べ替えが許されるか |
| **何がアトミックか** | どのサイズの代入が、必ず分割されずに完了するか |

そして、これらの保証を **プログラマが宣言する手段** として、

- `volatile`
- `synchronized`
- `final`
- `java.util.concurrent` の各種クラス（atomic を含む）

を用意しています。

> **JMM の歴史**
>
> Java 1.0〜1.4 の時代の JMM は曖昧で、**移植性のあるマルチスレッドコードがほぼ書けない**状態でした。
> JSR-133 で抜本的に書き直され、Java 5 以降、ようやく「**書けば動く**」マルチスレッドコードが可能になりました。
> 現在の `volatile` のセマンティクスは、Java 5 で大きく変わったものです。

---

## 「明示的に同期しない限り、何も保証されない」

JMM が定める基本姿勢は、こうです。

> 明示的に同期（`volatile`・`synchronized`・atomic など）を使わない限り、
> **他のスレッドからどう見えるか、何も保証しない**。

これは**冷たい**ように聞こえますが、実は逆で、

- **同期がなければ、JIT も CPU も思いきり最適化できる**（シングルスレッドとして扱える）
- **必要な箇所だけ同期する**ことで、性能と安全性を両立できる

という設計思想の表れです。
Java は、「**何でもかんでも安全**」ではなく、「**安全にしたい場所を明示する**」ことを求めます。
そのための道具立てが、これから 2〜5 節で見ていく `volatile` / `synchronized` / `atomic` です。

---

## まとめると

- 並行処理の落とし穴は、**可視性・順序・アトミシティ**の 3 つに整理できる
- **可視性**: 書いた値が、他のスレッドにいつ見えるか
- **順序**: 書いた順どおりに見えるか（JIT・CPU が並べ替える可能性）
- **アトミシティ**: 1 ステップに見える操作が、本当に途中で割り込まれないか
- これらを規定するのが **Java メモリモデル**（JMM）
- 基本姿勢: **明示的な同期がない限り、保証はない**
