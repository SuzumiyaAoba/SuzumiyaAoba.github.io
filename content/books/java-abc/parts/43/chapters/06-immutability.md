## 不変性とスレッド安全性

ここまで「**守る**」道具立てを見てきました。
ですが、最強の戦略はもう一段先にあります。それは ―― **「そもそも変えない」**。
**不変オブジェクト**（immutable object）は、それ自体が**完全にスレッド安全**です。

---

## 不変オブジェクトの定義

**不変オブジェクト**とは、

> 生成された後、**状態が一切変わらない**オブジェクト

のことです。
代表例は `String`、`Integer`、`LocalDate`、`Path` などです。
`String s = "abc"; s.toUpperCase();` を実行しても、`s` 自身は `"abc"` のまま ―― 新しい `String` オブジェクトが返るだけです。

不変オブジェクトは、

- どのスレッドから読まれても、**同じ値**しか返さない
- 書き換えられないので、**ロックが要らない**
- どこに渡しても、**変な改ざんを心配しなくていい**

という、並行処理の悩みの**ほとんど**を消してくれます。

---

## 不変クラスの作り方

不変なクラスを作るルールは、Java 16 以前は次のとおりでした。

1. クラスを `final` にする（継承で挙動を変えられないため）
2. すべてのフィールドを `private final` にする
3. セッターを書かない
4. **可変なオブジェクト**（`Date`、`List` など）を保持するときは、

   - コンストラクタで**防衛的にコピー**する
   - ゲッターでも**コピー**して返す

```java
public final class Person {
    private final String name;
    private final List<String> hobbies;

    public Person(String name, List<String> hobbies) {
        this.name = name;
        this.hobbies = List.copyOf(hobbies);   // ← 防衛的コピー
    }
    public String getName() { return name; }
    public List<String> getHobbies() { return hobbies; }   // copyOf は List.copyOf でイミュータブル
}
```

この書き方は、間違いやすく、ボイラープレートが多いのが弱点でした。
Java 14 以降の **レコード**（第17章）が、それを解決します。

---

## レコードは「不変の標準形」

`record` を使うと、不変オブジェクトの宣言が**1 行**で書けます。

```java
public record Person(String name, List<String> hobbies) {
    public Person {
        hobbies = List.copyOf(hobbies);   // 防衛的コピー
    }
}
```

- フィールドは自動的に `private final`
- 自動生成のコンストラクタ・ゲッター・`equals`・`hashCode`・`toString`
- クラス自体が暗黙に `final`

可変な引数を持つときの**防衛的コピー**だけ、コンパクトコンストラクタで明示すれば完璧です。

> **`List.copyOf` の落とし穴**
>
> `List.copyOf` は、Java 10 で追加された**イミュータブルリストを作る**メソッドです。
> 渡されたリストが**すでにイミュータブル**なら、コピーせずに**そのまま返す**こともあります（最適化）。
> 「**不変かもしれないし、コピーかもしれない、いずれにせよ不変リスト**」と覚えておけば実用上 OK です。

---

## なぜ「不変」だとスレッド安全なのか

JMM の観点で見ると、不変オブジェクトのスレッド安全性は、**`final` フィールドの保証**から来ます。
第2節で触れた **6 つめの happens-before**:

> コンストラクタが完了するまでに**確定**した `final` フィールドへの書き込みは、コンストラクタ完了後の**そのオブジェクトの任意のスレッドからの読み込み**に対して happens-before。

つまり、

1. すべてのフィールドが `final`
2. コンストラクタが完了
3. その後、**どのスレッドからも、同期なしで読める**

という保証です。
別のスレッドが「中途半端な状態の Person」を見ることはありません。

これに対し、可変なクラスでは、

```java
Person p = new Person();
p.name = "Alice";    // 別スレッドからは、まだ name が null に見えるかもしれない
```

という危険があります。
不変オブジェクトには、この危険が**構造的にない**のです。

---

## 「コピーが多くなって遅いのでは?」

不変オブジェクトの代表的な反論が「**毎回コピーは遅くないか?**」です。
たしかに、

- `String.replace(...)` は、新しい `String` を作る
- `record Person` のフィールドを 1 つだけ変えたいなら、新しい `Person` を作る

という具合に、変更=新規生成になります。
ですが、

- 多くの場合、**生成コストはわずか**（特に小さいオブジェクトなら、エスケープ解析（第41章）で**スタックに置かれる**）
- 共有が安全なので、**ロックや防衛的コピーが不要**になり、トータルで速い
- GC が短命オブジェクトに最適化されているので、すぐに回収される

そして何より、

- **読み書きを区別しなくていい**
- **正しさが言語仕様で保証**される

という、**人間にとっての**読みやすさ・正しさが大きいです。
パフォーマンスが本当に問題になるのは、ベンチで測ってから ―― それがプロの判断です。

---

## 「変更する」 ―― ビルダーパターンと `with` メソッド

不変オブジェクトのフィールドを 1 つだけ変えたい、はよくある要求です。
これには 2 つのイディオムがあります。

### ビルダーパターン

`StringBuilder` のように、**変更を蓄積するビルダー**を別途用意します。

```java
Person p = new Person.Builder()
    .name("Alice")
    .hobbies(List.of("Reading"))
    .build();
```

複数のフィールドをまとめて設定するときに読みやすい一方、ボイラープレートが多くなります。
**Lombok の `@Builder`** などで自動生成するのが現代風です。

### `with` メソッド

「**ある 1 つのフィールドだけ変えた新しいインスタンス**」を返すメソッド。

```java
public record Person(String name, List<String> hobbies) {
    public Person withName(String newName) {
        return new Person(newName, hobbies);
    }
}
```

```java
Person p2 = p.withName("Bob");   // p は変わらず、p2 が新たに作られる
```

関数型プログラミングの言語では「**copy with modifications**」と呼ばれる定番の手筋です。
レコードと相性がよく、Java 25 ではまだ**自動生成されません**が、その提案（withers）は進行中です。

## 不変なコレクション

`java.util` のコレクションには、不変版が用意されています。

| 不変な作り方 | 何ができるか |
|---|---|
| `List.of(...)` | 任意の要素から不変リスト |
| `Set.of(...)` | 不変セット |
| `Map.of(...)` | 不変マップ |
| `List.copyOf(coll)` | 既存コレクションから不変リスト |
| `Collections.unmodifiableList(list)` | 既存リストの**ビュー**（中身は外で変えられる可能性！） |

`Collections.unmodifiableList` の罠は、**元のリストを変えれば、ビューも変わって見える**ことです。
真の不変が欲しいなら、`List.copyOf` を使うのが安全です。

---

## 「不変・スレッドセーフ」の階層

クラスの「**スレッド安全性のレベル**」は、おおむね 4 段階に整理できます（『Java Concurrency in Practice』の有名な分類）。

1. **イミュータブル**: 不変なので、何もしなくて安全（例: `String`、`record` でフィールドも不変）
2. **スレッドセーフ**: 内部で同期を済ませているので、外から使う側は何もしなくていい（例: `ConcurrentHashMap`）
3. **条件付きスレッドセーフ**: 単一操作はスレッドセーフだが、**複数操作の組み合わせ**には外部の同期が要る（例: `Collections.synchronizedMap`）
4. **スレッドホスタイル / 単スレッド**: 並行アクセスは不可。呼び出し側が同期する

クラスを設計するときは、これらのどれを目指しているか、ドキュメントで明示しましょう。

---

## 「並行処理の正しい設計の原則」

並行処理を扱う設計の原則は、ひとことで言えば:

> **共有するなら不変、変えるなら共有しない。**

- スレッド間で共有するなら、**不変オブジェクト**にする
- 変えたいなら、**そのスレッド専用**にする（`ThreadLocal`、引数で渡す、関数の中だけで使う）

この原則だけ守っていれば、`volatile` も `synchronized` も `Atomic` も、**ほとんど要らなくなります**。
道具立てを知った上で、**使わずに済ます**設計が、究極のスレッド安全性です。

---

## まとめると

- **不変オブジェクト**は、それ自体がスレッド安全 ―― ロック不要
- レコードは、不変クラスの**標準形**
- JMM の**`final` フィールドの保証**から、同期なしで安全に読める
- 不変だとコピーが増えるが、**読みやすさと正しさ**が勝る
- 「**`with` メソッド**」「**ビルダー**」で、1 つだけ変えたインスタンスを作る
- スレッド安全性は **イミュータブル > スレッドセーフ > 条件付き > スレッドホスタイル**
- 究極の原則: **「共有するなら不変、変えるなら共有しない」**
