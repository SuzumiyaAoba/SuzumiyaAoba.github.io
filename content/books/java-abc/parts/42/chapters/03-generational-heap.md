## 世代別ヒープ ― Eden・Survivor・Old

前節で、ヒープを「**若い**」と「**古い**」の 2 つに分ける世代別 GC の考え方を見ました。
この節では、Young Generation の中身（**Eden** と 2 つの **Survivor**）を含めて、Java のヒープ構造を具体的に見ていきます。

---

## ヒープ全体の構造

Java の世代別 GC では、ヒープは次のように分かれています。

```text
┌─────────────────── ヒープ全体 ────────────────────┐
│                                                  │
│ ┌───── Young Generation ─────┐ ┌─── Old Gen ───┐ │
│ │                            │ │               │ │
│ │ ┌──────┐ ┌────┐ ┌────┐    │ │               │ │
│ │ │ Eden │ │ S0 │ │ S1 │    │ │   Old (Tenured) │ │
│ │ └──────┘ └────┘ └────┘    │ │               │ │
│ │                            │ │               │ │
│ └────────────────────────────┘ └───────────────┘ │
└──────────────────────────────────────────────────┘
```

- **Eden**（エデン）: 新しく `new` されたオブジェクトが、最初に置かれる場所
- **Survivor 0 / Survivor 1**（S0 / S1）: マイナー GC を生き延びたものが、一時的に住む場所
- **Old**（Tenured、テニュアード）: 何回も生き残った長寿オブジェクトの住処

---

## なぜ Survivor が 2 つあるのか

「Eden と Survivor が 1 つで十分なのでは?」と思うかもしれません。
2 つに分けている理由は、**コピー GC** の動きを思い出すと分かります。

```text
[マイナー GC 前]
Eden: [A, B, C, D, E]    （いっぱい）
S0  : [X, Y]              （前回生き残った組）
S1  : (空)                ← コピー先

[マイナー GC]
1. Eden と S0 を一緒にスキャン
2. 生きてるオブジェクトを S1 にコピー
3. Eden と S0 を空にする

[マイナー GC 後]
Eden: (空)
S0  : (空)
S1  : [生き残ったオブジェクト]
```

次のマイナー GC では、**S0 と S1 の役割が入れ替わります**（S0 がコピー先、S1 がスキャン側）。
**いつも片方が空っぽ**なのは、コピー GC のために**コピー先を確保**しておくためです。

---

## 「年齢」と昇格（Promotion）

オブジェクトには、「**今までに何回のマイナー GC を生き延びたか**」という**年齢**（age）が記録されます。
マイナー GC を生き延びるたびに +1 されます。

ある閾値（多くの GC で 15 など）に達したオブジェクトは、「**もう Young にいる必要なし**」と判断され、**Old Generation に移動**します。
これを **昇格**（Promotion、テニューア）と呼びます。

```text
Young（Eden / Survivor）
     │ 何度ものマイナー GC を生き残った
     ▼  （年齢が閾値超え、または Survivor が満杯）
Old（Tenured）
```

昇格条件は GC によって少しずつ違いますが、おおむね次のとおりです。

1. **年齢が閾値を超えた** ―― 長寿が確定したと判断
2. **Survivor が溢れた** ―― 行き先がないので、Old に押し込む（**早期昇格**）

「早期昇格」は、本当はまだ Young にいたほうがよかったオブジェクトを Old に押し上げてしまうので、後のメジャー GC を重くする原因になります。
チューニングのポイントの一つです（第6節）。

---

## ライフサイクルの全体図

ここまでをまとめて、オブジェクトの一生を図にすると次のようになります。

```text
[new]
  │
  ▼
┌──────┐                   ┌──────┐
│ Eden │ ──マイナー GC───► │  S0  │
└──────┘                   └──────┘
                              │
                              │ 次のマイナー GC
                              ▼
                           ┌──────┐
                           │  S1  │ ←→ S0 と入れ替わり続ける
                           └──────┘
                              │
                              │ 年齢が閾値を超える
                              ▼
                           ┌──────┐
                           │ Old  │ ──メジャー GC──► [回収]
                           └──────┘
```

「Eden → Survivor のあいだを行き来しながら、生き延びたら Old へ」 ―― これが世代別 GC の中での Java オブジェクトの一生です。

---

## 領域サイズのバランス

それぞれの領域のサイズは、GC とヒープサイズに応じて自動調整されますが、起動オプションで明示的に指定できます。

| オプション | 意味 |
|---|---|
| `-Xms` / `-Xmx` | ヒープ全体の初期 / 最大サイズ |
| `-Xmn` | Young Generation のサイズ |
| `-XX:NewRatio=N` | Old : Young の比率（既定は 2、Old が 2/3） |
| `-XX:SurvivorRatio=N` | Eden : Survivor の比率（既定は 8、Eden が 8/10） |
| `-XX:MaxTenuringThreshold=N` | 昇格までの年齢上限（既定は 15） |

たとえば `-Xmx2g -XX:NewRatio=1` なら、ヒープ 2 GB のうち Young 1 GB ・ Old 1 GB の構成になります。

ただし、現代の G1（次節）や ZGC では、**領域は固定の Young/Old ではなく、リージョン単位**で動的に決まります。
上のオプションのいくつかは、**G1 では効きません**。
覚えるべきは「**Young と Old の比率を変えると、何が変わるか**」の感覚です。

| 操作 | 何が起こりやすくなるか |
|---|---|
| Young を大きくする | マイナー GC の**頻度が減る**が、**1 回あたりが重くなる** |
| Young を小さくする | マイナー GC の**頻度が増える**が、1 回は軽い。早期昇格が起きやすい |
| Old を大きくする | メジャー GC の**頻度が減る**が、**1 回あたりが重くなる** |

レイテンシ重視のシステムなら、「**1 回の停止を短く**」が大事 ―― 領域は小さめにして頻度で回す方針が合います。
スループット重視なら、「**合計の GC オーバーヘッドを小さく**」 ―― 領域を大きめにして頻度を下げる方針が合います。

---

## 「Humongous」 ―― 巨大オブジェクト

G1 GC（次節）の用語ですが、**Humongous**（ヒューマンガス、巨大）という分類があります。

> リージョン（既定で 1 MB〜32 MB 程度の領域）の**半分**を超える大きさのオブジェクト

を指します。
たとえば 5 MB の `byte[]` を確保すると、それは Humongous 扱いになり、Young / Old の通常パスとは別に**専用領域**に配置されます。
Humongous は、GC の挙動も少し変則的になります。

実際、第5節で見る GC ログにも `G1 Humongous Allocation` というキーワードが出てきます。
「**大きな配列やバッファは、専用の挙動になる**」とだけ覚えておきましょう。

---

## まとめると

- Java の世代別ヒープは、**Eden + Survivor 0 + Survivor 1 + Old** の 4 領域
- Survivor が 2 つあるのは、**コピー GC** の方式上必要だから
- 何度ものマイナー GC を生き延びたら、**昇格**して Old へ
- **早期昇格**は、Survivor が小さすぎるとき起き、後のメジャー GC を重くする
- 領域サイズの比率は、**レイテンシ重視 vs スループット重視**でチューニング
- G1 では、**Humongous** な巨大オブジェクトに専用パスがある

次の節では、これらの仕組みを実装した Java の代表的な GC ―― **G1・ZGC・Parallel・Serial** を、用途とともに見ていきます。
