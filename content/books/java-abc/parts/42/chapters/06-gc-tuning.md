## GC のチューニング ― ヒープサイズと目標から考える

GC は、設定しすぎても、設定しなさすぎても、よくありません。
この節では、**何を目標にチューニングするか**を整理し、典型的な設定パターンを見ていきます。

---

## まず「目標」を決める

GC チューニングで最初にやることは、**コードを書く**でも、**フラグを足す**でも、ありません。
**目標**を 1 つ決めることです。

| 目標 | 重視する指標 |
|---|---|
| **レスポンスは速く** | **GC の停止時間**（pause time）の最大値 |
| **大量にさばきたい** | **GC のオーバーヘッド**（total GC time / wall clock） |
| **メモリを節約したい** | **ヒープサイズ** |

これらは多くの場合**トレードオフ**です。
全部を同時に最大化することはできません。
「**今、何が困っているのか**」「**どれを犠牲にしてもよいか**」を決めるのが、チューニングの第一歩です。

---

## 「目標を SLO で書く」

漠然と「レスポンス重視」と決めても、現場では合意が取りにくいです。
お勧めは、**SLO**（Service Level Objective、サービスレベル目標）の形で書くことです。

> 例: **99 パーセンタイルのレスポンスを 100 ms 以下に維持する**

この場合、GC の停止時間の P99 が**この値の何分の一以下**であるべきか ―― が、JVM 側の目標値になります。
おおむね、**SLO の 1/10〜1/5** が、GC 停止時間の上限の目安です。
SLO が 100 ms なら、GC 停止は **10〜20 ms 以下**を目指す、といった具合です。

---

## ヒープサイズの決め方

ヒープサイズの決め方には、おおむね 3 つの考え方があります。

### 1. 「**観測値の 2〜3 倍**」

本番想定の負荷で動かしたとき、**GC 後のヒープ使用量（live size）の 2〜3 倍**を `-Xmx` にする ―― これが定石です。

たとえば、定常状態で GC 後に 500 MB ぶんのオブジェクトが生き残っているなら、`-Xmx1500m` あたりが目安です。
**ヒープ使用量＝ live size + GC 用の余裕**で、余裕が少なすぎると **GC が頻発**し、多すぎると**メモリの無駄**になります。

### 2. 「**コンテナの 75 %**」

コンテナで動かすなら、`-XX:MaxRAMPercentage=75.0` のような割合指定が便利です。
コンテナのメモリ上限が変わっても、自動で追随します。

```text line-numbers=false
$ java -XX:MaxRAMPercentage=75.0 -XX:+UseG1GC -jar myapp.jar
```

残りの 25 % は、**メタスペース・スタック・JIT のコードキャッシュ・ネイティブバッファ**などに使われます。

### 3. 「**`-Xms = -Xmx`** で固定**」**

サーバー用途では、`-Xms` と `-Xmx` を**同じ値**にすることが多いです。

```text line-numbers=false
$ java -Xms2g -Xmx2g MyApp
```

理由は、

- ヒープ拡張のたびに走る OS への要求が省ける
- 起動直後から本来のサイズで動くので、定常状態に到達するのが速い
- メモリ使用量が予測しやすい（運用が楽）

逆に**起動時のメモリを節約したい**なら、`-Xms` を小さく、`-Xmx` を大きく、で動的に伸ばす設定もありますが、本番サーバーでは固定がおすすめです。

---

## G1 のチューニング ―― 「目標停止時間」を渡す

G1 では、第4節で触れたとおり、**目標停止時間**を渡すのが最も効くチューニングです。

```text line-numbers=false
$ java -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -Xmx2g MyApp
```

`-XX:MaxGCPauseMillis=100` は、「**GC の停止時間を 100 ms 以下に収めようとしてくれ**」というヒントです（**絶対の保証ではない**点に注意）。
G1 は、この目標に合わせて、

- リージョンサイズ
- 1 回の GC で回収するリージョン数
- Young / Old のバランス

を自動調整してくれます。
たいていのアプリは、**`-Xmx` と `-XX:MaxGCPauseMillis`** の 2 つを設定するだけで、かなり良い線にチューニングできます。

---

## ZGC のチューニング ―― ほぼ「何もしない」

ZGC は「**チューニングしない**」ことが、ある意味で正しい使い方です。

```text line-numbers=false
$ java -XX:+UseZGC -Xmx4g MyApp
```

ZGC は STW を 1 ms 以下に抑えるよう設計されているので、`-XX:MaxGCPauseMillis` のようなヒントは効きません。
**ヒープサイズだけ十分に大きく**取って、後は任せます。
強いて言えば、`-XX:SoftMaxHeapSize` で「**普段はここまでに収めようとして**」という柔軟な上限を指定するくらいです。

---

## チューニングのアンチパターン

「**やってはいけない**」を 5 つにまとめます。

### 1. 「とりあえず `System.gc()`」

`System.gc()` はフル GC を引き起こす可能性があり、**自分でレイテンシ事故**を起こします。
本番コードで呼ぶ理由は、ほぼありません。

### 2. 「とりあえずヒープを大きく」

大きくすればよい、ではありません。

- 大きすぎると、1 回の GC が**重く**なる
- メタスペース・スタックなど**他の領域**が圧迫される
- コンテナでは**OS から OOM Killer に殺される**

「**観測してから合わせる**」が原則です。

### 3. 「フラグの多重がけ」

`-XX:NewRatio`、`-XX:SurvivorRatio`、`-XX:MaxTenuringThreshold`、`-XX:G1HeapRegionSize` …
たくさんのフラグを同時に変えると、**何が効いたか分からなくなります**。
**変えるのは 1 つずつ**、効果を測ってから次へ。

### 4. 「**古い知識**で G1 をチューニング」

Java 8 時代の Parallel GC を前提とした記事を真に受けて、G1 にも同じ設定を入れるのは事故のもとです。
`-XX:NewRatio` が一部の GC で効かない、`-Xmn` が G1 では非推奨、など。
**今動かしている GC のドキュメント**を当たりましょう。

### 5. 「**テストの値**を本番に持ち込む」

ローカル環境（メモリ 32 GB の Mac）で **`-Xmx16g`** にしたら速くなった ―― この設定を、コンテナで動かす本番（メモリ 2 GB）にそのまま持っていくと、即 OOM です。
**本番の条件で**測ってから決めましょう。

## それでも詰まったときの手筋

「いろいろ調整したけど、まだ GC で詰まる」 ―― そんなときの**段階的な手筋**を 3 つ。

### 手筋1: **GC ログを取り直す**

`-Xlog:gc*:file=gc.log` で詳細ログを取り、GCViewer や GCEasy で可視化。
**ピーク使用量、GC 頻度、Before/After のヒープサイズ**を見ます。
リークなのか、ヒープ不足なのか、GC アルゴリズムが合っていないのかが、ここでだいたい分かります。

### 手筋2: **ヒープダンプを取る**

```text line-numbers=false
$ jcmd <pid> GC.heap_dump filename=heap.hprof
```

これで取れる `.hprof` ファイルを **Eclipse MAT**（Memory Analyzer Tool）で開くと、

- どのクラスが**何個・何 MB** 占めているか
- どの参照経路で**保持され続けている**か
- **疑わしいリーク**の自動検出

が分かります。
リーク調査は、ここまでやって初めて「**確実な犯人**」が見つかります。

### 手筋3: **GC を切り替えてみる**

- G1 で詰まっている → ZGC を試す
- バッチで遅い → Parallel を試す

切り替えはフラグ 1 つです（`-XX:+UseZGC` など）。
GC の選択そのものがミスマッチだった、というケースも、本番では珍しくありません。

---

## まとめると

- GC チューニングは「**目標を 1 つ決める**」ことから始める
- **SLO の 1/10〜1/5** を、GC 停止時間の上限の目安に
- ヒープサイズは「**観測値の 2〜3 倍**」、または `-XX:MaxRAMPercentage`
- G1 なら **`-Xmx` + `-XX:MaxGCPauseMillis`** で大半は済む
- ZGC は **ほぼチューニング不要**、ヒープサイズだけ取る
- 詰まったら、**ログ → ヒープダンプ → GC 切り替え**の順で診る
