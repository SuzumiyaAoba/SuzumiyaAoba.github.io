## GC の基本アルゴリズム ― マーク&スイープ・コピー・世代別

第1節で、GC は「**ルートから辿れないオブジェクトを回収する**」ことを学びました。
では、それを実際にどう実装するのか。
基本となる 3 つのアルゴリズムを、図で見ていきます。

---

## アルゴリズム1: マーク&スイープ（Mark and Sweep）

最も素朴で、もっとも分かりやすい GC アルゴリズムです。
名前のとおり、**2 段階**で動きます。

### 1. マーク（mark）

ルートから到達可能なオブジェクトを、すべて辿って「**生きている**」と印を付けます。

```text
[マーク前]
ルート ─► [A] ─► [B] ─► [C]
[D] ─► [E]  （ルートから来てない）

[マーク後]
ルート ─► [A]✓ ─► [B]✓ ─► [C]✓
[D]    ─► [E]    （未マーク）
```

### 2. スイープ（sweep）

ヒープを頭から舐めて、**マークされていない**オブジェクトを「空きリスト」に戻します。
マークされたオブジェクトは、**マークを消して**残します。

```text
[スイープ後]
ルート ─► [A] ─► [B] ─► [C]
[空き] [空き]
```

| 長所 | 短所 |
|---|---|
| 実装がシンプル | **断片化**（fragmentation）が起きる |
| オブジェクトを動かさない | スイープに**ヒープ全体**を舐める時間が必要 |
| 循環参照に強い | 確保速度が落ちていく |

**断片化**は、回収を繰り返すうちに「**小さな空き**」が穴ぼこのように残り、大きなオブジェクトが入る場所がなくなる現象です。
ヒープ全体としては空きがあるのに、`new` が失敗する ―― そんな状態が起きます。

---

## アルゴリズム2: コピー GC（Copying GC）

断片化問題を、もっと豪快に解決するのが**コピー GC** です。
ヒープを**半分に分け**、片方しか使いません。

```text
[使用中]                    [空き]
┌──────────────┐           ┌──────────────┐
│ A  B  C  D  E│           │              │
└──────────────┘           └──────────────┘
   From 領域                  To 領域
```

GC が起動すると、

1. **From 領域**を端から走査
2. ルートから辿れる生きたオブジェクトを、**To 領域**にコピー
3. コピー後の参照を、全部書き換える
4. 終わったら、**From と To を入れ替える**

```text
[コピー後]
┌──────────────┐           ┌──────────────┐
│              │           │ A  B  C      │
└──────────────┘           └──────────────┘
   From 領域 (空)            To 領域 (詰めて再配置)
```

| 長所 | 短所 |
|---|---|
| **断片化しない**（詰めて置き直すから） | ヒープの**半分しか使えない** |
| 生きてるオブジェクトだけ触る ―― ゴミが多いと**速い** | 生きてるオブジェクトが多いと**コピーコストが重い** |
| 確保は単に**末尾を伸ばすだけ**で速い | 参照書き換えが要る |

特に「**生きてるオブジェクトが少ない**領域では速い」という性質が、次の世代別 GC の発想につながります。

---

## アルゴリズム3: マーク&コンパクト（Mark and Compact）

コピー GC の「半分しか使えない」を回避するアルゴリズムです。
マーク&スイープと同じく、まずマーク。
そのあと、**生きてるオブジェクトを端から詰め直し**（コンパクト）ます。

```text
[マーク後]
[A]✓ [   ] [B]✓ [   ] [C]✓

[コンパクト後]
[A] [B] [C] [          空き           ]
```

| 長所 | 短所 |
|---|---|
| ヒープ全部を使える | 移動コストが重い |
| 断片化しない | 並行化が難しい |

これら 3 つの基本アルゴリズムを「**いつ・どこに**」適用するかを工夫したのが、次に見る**世代別 GC** です。

---

## 「ほとんどのオブジェクトはすぐ死ぬ」 ―― 弱世代仮説

40 年以上前から知られている、**Java を含む多くの言語**で当てはまる経験則があります。

> **オブジェクトの大半は、生まれてすぐ死ぬ。**
> 長生きするオブジェクトは、ますます長生きする。

これを **弱世代仮説**（Weak Generational Hypothesis）と呼びます。
Web アプリで言えば、

- **リクエスト処理中の一時オブジェクト**（DTO、文字列、ラムダ、Stream）は、**1 リクエストで死ぬ**
- **キャッシュ・接続プール・設定オブジェクト**は、**アプリ起動から終了までずっと生きる**

の極端な二極化が、ふつうに起こります。
この性質を活用すれば、GC を**もっと効率化**できます。

---

## 世代別 GC ―― 「若い者と古い者を分ける」

弱世代仮説をそのまま GC に持ち込んだのが、**世代別 GC**（Generational GC）です。
ヒープを「**若い世代**」（Young Generation）と「**古い世代**」（Old Generation）に分けて扱います。

```text
ヒープ
┌─────────────────────────────────────┐
│  Young Generation    Old Generation │
│ ┌──────────────────┐ ┌────────────┐ │
│ │ Eden  S0   S1    │ │            │ │
│ └──────────────────┘ └────────────┘ │
└─────────────────────────────────────┘
```

- **Young**: 新しく生まれたオブジェクトの住処。**ほとんどがすぐ死ぬ**から、コピー GC で速く回す
- **Old**: 何度の GC を生き延びた長寿オブジェクトの住処。回収頻度は低く、マーク&コンパクトでまとめて掃除

それぞれを「**ほとんどがゴミ**」「**ほとんど生きている**」と決めつけてしまうことで、

- Young は **頻繁に、速く**回せる
- Old は **稀に、入念に**回す

という、メリハリの効いた GC ができます。

実際の GC では、Young が **Eden + Survivor 2 つ**に分かれていますが、これは次の節で詳しく見ます。

---

## マイナー GC とメジャー GC

世代別 GC が動くタイミングには、2 種類あります。

| 種類 | いつ起きる | 範囲 | 速さ |
|---|---|---|---|
| **マイナー GC**（Minor / Young GC） | Young が埋まったとき | Young だけ | **速い**（ms 単位） |
| **メジャー GC**（Major / Full GC） | Old が埋まったとき | Young + Old | **遅い**（百 ms〜秒） |

Web アプリで「**たまに長い間止まる**」と感じるのは、たいていメジャー GC です。
逆に言えば、

- マイナー GC が**頻発**しても、合計時間が短ければ問題は出にくい
- メジャー GC は**頻度を下げる**ことが、レイテンシ改善の王道

このメリハリを意識すると、GC ログ（第5節）の読み方が一気に身につきます。

---

## まとめると

- 基本アルゴリズムは 3 つ:
  - **マーク&スイープ** ―― シンプルだが**断片化**する
  - **コピー GC** ―― **断片化しない**が、ヒープを半分しか使えない
  - **マーク&コンパクト** ―― 詰め直すが、移動コストが重い
- **弱世代仮説**（オブジェクトの大半はすぐ死ぬ）から、**世代別 GC**が生まれた
- Young は速く、Old は入念に
- **マイナー GC は速く、メジャー GC は重い** ―― レイテンシで気にすべきは後者
