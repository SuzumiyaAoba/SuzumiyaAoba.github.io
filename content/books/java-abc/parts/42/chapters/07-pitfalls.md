## よくあるつまずき

GC は「**自動だから何もしなくていい**」と思われがちですが、運用ではそうもいきません。
ここでは、Java の GC でとくに踏みやすい 7 つのつまずきを整理します。

---

## つまずき1: 「`System.gc()` を呼べばすっきり片付く」

業務コードで `System.gc()` を呼ぶのは、ほぼ常に**間違い**です。

- 呼んでも、実装が無視できる仕様（`-XX:+DisableExplicitGC` で無効化可）
- 呼ぶと**フル GC が走る可能性**があり、長い STW を自分から作る
- ライブラリ・テスト・本番、どこでも一度の呼び出しが**重い**

「メモリが減るのを確認したい」「キャッシュをクリアしたい」 ―― そういう動機はわかりますが、ふつうは GC ではなく、**参照を切る**（コレクションから remove する、変数に `null` を入れる）ことで対処します。

唯一許せる場面は、`jcmd` で「**今のヒープの実体を測りたい**」ような、**運用・診断目的**だけです。

---

## つまずき2: 「`new` してすぐ捨てるオブジェクトはコストゼロ」

「すぐ捨てるならコピー GC で速いから、いくら作っても無料」 ―― 半分は本当、半分は罠です。

たしかに、生まれてすぐ死ぬオブジェクトは Young の 1 回目の GC で安く回収されます。
ですが、

- 大量に作ると、**マイナー GC の頻度が増える**（合計の停止時間は無視できなくなる）
- 一時オブジェクトが **Survivor に溢れる**と、**早期昇格**で Old を圧迫する
- ループの中で `String + String` を繰り返すと、毎回**新しい `String` オブジェクト**ができる

`for (int i = 0; i < n; i++) { s = s + "..."; }` の遅さの原因は、ここです。
ループでの文字列連結は **`StringBuilder`**（第28章）が定石です。

---

## つまずき3: 「リスナ／コールバックを外し忘れる」

Java で**気づきにくいリーク**の代表が、**リスナの外し忘れ**です。

```java
button.addClickListener(e -> log.info("clicked"));
```

このコードでは、`button` がラムダ式を保持し、ラムダ式は**そのクラスのインスタンス**を暗黙に参照します。
ボタンが長生きするのに、画面が閉じても**リスナが残り続ける**と、画面 1 つぶんのオブジェクトが**永遠に GC されません**。

対処は単純に、

```java
button.removeClickListener(handler);
```

を必ず呼ぶこと。`AutoCloseable` で `close()` の中で removeListener するパターンも有効です。

同じ理由で、**Observer パターン**（第34章）の解除忘れも、定番のリーク源です。

---

## つまずき4: 「`ThreadLocal` を `remove` しない」

`ThreadLocal` は、**スレッドごとに値を持てる**便利な仕組みですが、

- スレッドプールから**再利用**されるスレッド上で
- `set` だけして `remove` しない

と、前のリクエストで入れた値が**残り続け**、リクエストごとに溜まっていきます。
仮想スレッド（第44章）と違って、**プラットフォームスレッドは長生き**するので、`ThreadLocal` リークは特に深刻になります。

```java
try {
    requestContext.set(ctx);
    // ... 処理 ...
} finally {
    requestContext.remove();   // ← これを忘れない
}
```

`try-finally` で `remove` を必ず呼ぶ ―― これが鉄則です。

---

## つまずき5: 「`-Xmx` を増やせば速くなる」

「ヒープを大きくしたほうが GC が走らなくて速いはず」 ―― 半分は正しいですが、**やりすぎは逆効果**です。

- ヒープが大きいと、**1 回の GC が重く**なる（マークも長く、コピーも長く）
- ヒープが大きいと、メモリページの**TLB ミス**が増えて CPU が遅くなる
- コンテナでは、ヒープを大きくしすぎると **メタスペース・スタック・JIT** のメモリが圧迫される

「**GC が頻発している**」と確認してから増やすのは正しい判断ですが、「**遅いから増やす**」は短絡的です。
**ログを取る → 原因を切り分ける → ヒープが原因と確かめてから増やす**、の順を守りましょう。

---

## つまずき6: 「キャッシュは `HashMap` で十分」

「メモリ上のキャッシュは `HashMap` でいいや」 ―― 大規模アプリでよくあるリーク源です。

- キーが`String` の合成キーなど**ユニークに増える**性質なら、エントリが**無限に増え続ける**
- 失効ロジック（TTL、LRU）がないと、ヒープを食い潰す

対策として:

1. **`Map` のサイズ上限を決める**（`LinkedHashMap` で LRU）
2. **キャッシュライブラリを使う**（**Caffeine**、Guava Cache など）
3. **`WeakHashMap`** で、参照されなくなったら GC されるキャッシュにする
4. **`SoftReference`** を使い、**メモリが足りなくなったら捨ててよい**キャッシュにする

ふつうのアプリは Caffeine がおすすめです（第43章周辺の関心事を超えるので、本書では触れません）。

## つまずき7: 「GC のチューニングが先、コードの見直しは後」

ベテランがやりがちな逆順です。

- フル GC 連発 → JVM フラグを色々試す → 解決せず → 結局**コードが悪かった**

GC のチューニングは、**コードに問題がない**ことを前提とした最終手段です。
最初に見るべきは、

1. **不要に大きなオブジェクトを保持していないか**
2. **コレクションへの追加が、削除より速くないか**
3. **キャッシュに上限と TTL があるか**
4. **リスナ・`ThreadLocal` が掃除されているか**

これらの**ヒープを増やさない設計**が先、JVM フラグは後、です。

---

## まとめると ―― 「GC は忘れない」

GC を恐れずに付き合うコツは、たった 1 つです。

> **GC は忘れません。あなたが「不要」と忘れた瞬間、それを参照する変数があれば、GC は「まだ必要」と判断します。**

「**不要なら、参照を切る**」 ―― この一言を、ロジック設計の柱に置く。
これだけで、Java の GC に振り回されることは、ぐっと減ります。
