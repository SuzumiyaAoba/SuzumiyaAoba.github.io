## はじめに ― この章で学ぶこと

第41章で、オブジェクトが置かれる場所 ―― **ヒープ**を学びました。
ヒープは、`new` するたびに使われていく一方の領域です。
「もう使わないオブジェクト」を**そのままにしておけば**、メモリはあっという間に尽きます。

それでも、本書のサンプルが**「いつかメモリが足りなくなる」ことなく**動いてきたのは、JVM の中で、絶えず**お掃除**が走っているからです。
そのお掃除の名が、**ガベージコレクション**（Garbage Collection、GC）です。

第42章では、

- そもそも GC とは何をしているのか
- どんな**アルゴリズム**で動くのか
- Java 25 にはどんな GC があり、どう**使い分けるか**
- 本番運用で見ることになる **GC ログ**の読み方

を扱います。

---

## なぜ GC を学ぶ必要があるのか

C や C++ では、`malloc` / `free` を**手で**書かないと、メモリリークやダングリングポインタを生みました。
Java はそれを**自動化**しました。
書く側からすると「**GC があるから、メモリのことは考えなくていい**」と感じてしまいますが、これは半分しか正しくありません。

| 自動化されたこと | 自分で考えるべきこと |
|---|---|
| `free` を呼ぶタイミング | **参照を持ち続けない**こと（持つとリーク） |
| メモリ確保のアルゴリズム | **GC アルゴリズムの選択**（用途に合うか） |
| ヒープ全体のスキャン | **GC が止める時間**（レイテンシ） |

特に最後の「**GC が止める時間**」――これがそのまま、**ユーザーが体感する遅延**になります。
レイテンシ要件の厳しいシステムでは、GC の選択を間違うだけで、本番事故になります。

---

## GC のイメージ ――「もう触られないオブジェクトを集める」

GC のやっていることを一言で言うと、

> **どこからも参照されていない**オブジェクトを見つけて、その場所を**空ける**

これだけです。
「**参照されているか**」は、ヒープを**ルート**（スタック上の変数や、static フィールドなど）から手繰っていって、辿り着けるかどうかで決めます。

```text
[スタックの変数] ──► オブジェクト A ──► オブジェクト B
                                    │
                                    ▼
                              オブジェクト C  ← 辿れる（生）
                              
                              オブジェクト D ──► オブジェクト E  ← 辿れない（ゴミ）
```

GC は、こうして「辿れる」ものを残し、「辿れない」ものを**回収**します。
ただ、この単純な原理を、**動いているプログラムの上で**、**できるだけ止めずに**、**ギガバイト級のヒープに対して**やる ―― そこから、現代の GC の難しさと工夫が生まれます。

第42章は、次の8つの節で構成されています。

| 節 | タイトル | 内容 |

## この章を読み終えると

第42章を読み終えるころには、次のことができるようになっています。

- 「**GC が消すもの／消さないもの**」を、参照の到達可能性で説明できる
- マーク&スイープ・コピー・世代別の**基本アルゴリズム**を区別できる
- Java の標準 GC（**G1・ZGC・Parallel・Serial**）の使い分けが言える
- `-Xlog:gc` で取った GC ログを、最低限**読める**
- 「**メモリリーク**」が、Java でも起きる典型パターンを挙げられる
- 適切な**ヒープサイズ**と GC を、用途から選べる

それでは、最初の節「ガベージコレクションとは」から始めましょう。
