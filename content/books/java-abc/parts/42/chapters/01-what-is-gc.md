## ガベージコレクションとは

**ガベージコレクション**（Garbage Collection、ごみ収集）は、不要になったメモリを**自動で**回収する仕組みです。
Java のヒープに `new` で置かれたオブジェクトは、いつかは GC に回収されます。
この節では、その大原則を整理します。

---

## 「不要」とは何か ―― 到達可能性

GC が「これは捨ててよい」と判断する基準は、たった 1 つです。

> どこからも**辿れない**オブジェクトは、不要

ここで言う「辿れる」とは、**ルート**（root、起点）から参照を辿って、そのオブジェクトに行き着けることです。
ルートには次のものが含まれます。

| ルート | 例 |
|---|---|
| 各スレッドの**スタック**にあるローカル変数 | `void m() { Foo f = new Foo(); ... }` の `f` |
| **static フィールド** | `class Cache { static Map<...> m = ...; }` |
| **JNI / Panama** から参照されているハンドル | ネイティブ側で握っているもの |
| 実行中のスレッド自身 | `Thread` インスタンス |

このルートから、参照のつながりを辿って到達できるオブジェクトを「**到達可能**」（reachable）、辿れないものを「**到達不可能**」（unreachable）と呼びます。

```text
ルート
  │
  ▼
[A] ─► [B] ─► [C]
              ▲
              │（A から B 経由で到達可能）

[D] ─► [E]        （ルートから辿れない＝ゴミ）
  ▲      │
  │      ▼
  └──────[F]      （E と F が互いに参照しても、ルートから来てなければゴミ）
```

特に重要なのが、最後のケースです。
**互いに参照し合うオブジェクト**も、**ルートから辿れなければ**まとめて回収されます。
これが、参照カウント方式の言語にあった**循環参照のリーク**問題を、Java は構造的に持たない理由です。

---

## 「使わない」と「参照されない」は違う

ここで、初心者が最も陥りやすい誤解があります。

> **使わなくなった**オブジェクトは、自動で消える?

答えは「**いいえ**」です。
GC が見ているのは「**使うかどうか**」ではなく、「**参照が残っているかどうか**」です。
「もう使わない」と頭でわかっていても、変数や Map に**入れっぱなし**だと、GC からは「**まだ使う気だな**」と見えます。

これが、Java で**メモリリーク**が起きる仕組みです。
たとえば、

```java
class Cache {
    static Map<String, byte[]> cache = new HashMap<>();
    static void put(String key, byte[] data) {
        cache.put(key, data);
    }
}
```

このコードに `remove` を書かないまま `put` し続けると、`Cache.cache` から無限にオブジェクトが辿れる状態が続き、**永久に GC されません**。
ヒープがどんどん埋まり、最後は `OutOfMemoryError: Java heap space`。

Java の GC は**忘れません**。**あなたが忘れない限り、GC も忘れない**のです。

---

## GC が走る瞬間

GC は、いつ走るのでしょうか。
ざっくり言うと、**ヒープに空きがなくなりそうになったとき**です。
具体的なきっかけはいくつかあります。

| きっかけ | 概要 |
|---|---|
| **割り当て失敗** | `new` しようとして、空き領域がない |
| **しきい値超過** | ヒープ使用量が、設定した目標を超えた |
| **明示的な要求** | `System.gc()` の呼び出し（実装依存。**推奨しない**） |
| **アイドル時** | 一部の GC は、暇なときに先回りで回収する |

`System.gc()` は、JVM 仕様としては「**ヒント**」にすぎず、無視してよいとされています（実際、`-XX:+DisableExplicitGC` で無効化できます）。
本書では呼ばない方針で進めます。

---

## 「Stop The World」

GC を語るとき、必ず出てくる言葉が **Stop The World**（STW、世界を止める）です。

> GC が、**プログラム全体の実行を一時停止**して、ヒープを掃除すること

なぜ止めるのか?
動いているスレッドが**ヒープを書き換えながら**、GC が参照を追って回るのは、原理的に**整合性が取れない**からです。
一瞬でも止めれば、参照グラフの**写真**が確定し、その上で安全に回収できます。

ただ、止める時間が長いと、利用者から見ると「**画面が固まった**」「**API が遅い**」と感じます。
そこで現代の GC は、

- **マークの並行化**: 止めずにマークを進める
- **コピーの並行化**: 止めずにオブジェクトを移動する
- **領域ごとの回収**: 全部を一度に掃除しない（インクリメンタル）

といった工夫で、STW 時間を**極限まで短く**しています。
第4節で見る **ZGC** などでは、STW がミリ秒以下です。

---

## ファイナライザ ―― 触らないでください

ここで、注意点を 1 つ。Java には `Object` の `finalize()` という、**「GC される直前に呼ばれるメソッド」** がありました。
ですが、**現在は使ってはいけません**。

| 理由 | |
|---|---|
| 呼ばれる**保証がない** | プログラム終了時にも呼ばれない可能性 |
| 呼ばれる**順序が決まらない** | 依存関係のあるリソース解放には使えない |
| **GC の挙動を狂わせる** | ファイナライザがあると、世代を進めなくなる |
| Java 9 で **`@Deprecated`**、Java 18 で**廃止予告**（JEP 421） | 公式に「使うな」 |

リソース解放は、`try-with-resources`（第20章・第30章）と `AutoCloseable` を使う、というのが Java 7 以降の答えです。
ファイナライザは、コードレビューで見かけたら**赤信号**だと思ってください。

代替として、ファイナライザより安全な仕組みに `java.lang.ref.Cleaner` がありますが、ふつうのアプリでは出番はほぼありません。
リソース管理は **`try-with-resources` 一択**と覚えておくのが安全です。

---

## まとめると

- GC は「**ルートから辿れないオブジェクト**」を回収する
- **使うかどうか**ではなく、**参照が残っているか**で判断する
- Java で起きるメモリリークは、**入れっぱなしのコレクション**が代表
- GC は**ヒープ不足が近いとき**に走る。`System.gc()` は呼ばない
- **Stop The World** で一瞬止まる。現代の GC はそれを最小化している
- **`finalize()` は使わない**。リソース解放は `try-with-resources`

次の節では、「**辿れないものを回収する**」を具体的にどう実装するか ―― 基本的な GC アルゴリズムを見ていきます。
