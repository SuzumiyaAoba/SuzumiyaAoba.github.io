## 用語集 ― この章で学んだ言葉

第33章では、Mockito を使ったモックテストの書き方と、テストしやすい設計を学びました。
節ごとに、言葉を整理しておきます。

---

## モックの基礎（第1〜2節）

| 用語 | 英語・読み | 意味 |
|---|---|---|
| モック | mock | テスト用の身代わり。呼び出しを記録し、検証できる |
| スタブ | stub | 「呼ばれたらこれを返す」と決め打ちする身代わり |
| フェイク | fake | 本物に近い軽量実装（インメモリ DB など） |
| Mockito | ― | Java で広く使われるモックライブラリ |
| `mockito-junit-jupiter` | ― | Mockito と JUnit 5 を連携させるアダプタ |
| `mock(クラス.class)` | ― | モックインスタンスを作る |
| `@ExtendWith(MockitoExtension.class)` | ― | JUnit 5 に Mockito を有効化する |
| `@Mock` | ― | フィールドをモックにする |
| `@InjectMocks` | ― | モックを注入したテスト対象を作る |

---

## スタブと検証（第3〜4節）

| 用語 | 意味 |
|---|---|
| `when(モック.メソッド(...)).thenReturn(値)` | 戻り値を仕込む |
| `thenThrow(例外)` | 例外を投げさせる |
| `thenAnswer(invocation -> ...)` | 引数に応じて動的に返す |
| `doThrow(...).when(モック).メソッド(...)` | `void` メソッドに例外を仕込む |
| `doAnswer(...).when(...)` | `void` メソッドに動的処理を仕込む |
| `verify(モック).メソッド(...)` | 呼び出しを検証する |
| `times(N)` / `never()` / `atLeast(N)` | 回数指定 |
| `verifyNoInteractions(モック)` | 一切呼ばれていないことを断定 |
| `verifyNoMoreInteractions(モック)` | verify で確認した以外には呼ばれていないことを断定 |
| `InOrder` | 呼び出し順序を検証 |
| `ArgumentCaptor` | 呼び出し時の引数を取り出して検証 |

---

## 引数マッチャー（第5節）

| マッチャー | 意味 |
|---|---|
| `any()` | 何でもよい（null 含む） |
| `anyString()` / `anyInt()` / `anyLong()` | 何らかの非 null プリミティブ / `String` |
| `any(クラス.class)` | 何らかのそのクラスのインスタンス |
| `eq(値)` | この値と等しい（他のマッチャーと混在時に必要） |
| `argThat(ラムダ)` | 条件を自由に書く |
| `nullable(クラス.class)` | null を許す |

---

## 設計と落とし穴（第6〜7節）

| 用語 | 意味 |
|---|---|
| 依存性の注入（DI） | Dependency Injection。依存を外から渡す設計（第36章で本格的に） |
| コンストラクタ注入 | コンストラクタの引数で依存を受け取る形 |
| 境界（boundary） | 外の世界（DB・HTTP・メール）との接点 |
| 振る舞いベースのテスト | 戻り値ではなく呼び出し自体で検証するテスト |
| `InvalidUseOfMatchersException` | マッチャーと具体値の混在で出るエラー |
| `mockStatic` | `static` メソッドをモック化する技。設計悪化のサインでもある |

---

## おわりに ― 「テストを書く力」がそろいました

第32章と第33章で、現代の Java におけるユニットテストの**書き方**がそろいました。

- **JUnit 5** で、純粋なロジックのテストを書ける
- **Mockito** で、依存を切り離したテストを書ける
- **テストしやすい設計**を意識して、本番コードを書ける

これで、書いたコードに**自信を持って手を入れられる**状態になりました。
ここから先は、

- どんな**構造**でクラスを組み合わせると、変更に強くなるか（**デザインパターン**、第34章）
- それらを支える、**設計原則**（**SOLID**、第35章）
- 依存先を、自動で組み立ててくれるしくみ（**DI と Spring**、第36章）

と、設計の話へ広げていきます。
書く力に加えて、「**どう設計するか**」を学んでいきましょう。
