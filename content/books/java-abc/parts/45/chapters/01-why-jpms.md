## なぜモジュールシステムが必要か

モジュールシステム（JPMS）を学ぶ前に、**なぜそれが必要になったのか**を整理しましょう。
背景が分かれば、各機能の必然性がすっと入ってきます。

---

## 背景1: クラスパス地獄

Java 1.0 から続いてきた**クラスパス**（classpath）は、シンプルで強力ですが、規模が大きくなると問題を起こします。

```text
$ java -cp lib/spring-core-5.0.jar:lib/spring-beans-5.0.jar:lib/log4j-2.0.jar:... MyApp
```

このような長いリストを管理する中で、

### 1. 「**重複した jar**」

同じライブラリの異なるバージョンが、クラスパスに 2 つ入る、ということが起きます。
たとえば、`A.jar` が `slf4j-1.7` を要求し、`B.jar` が `slf4j-2.0` を要求すると、

```text
-cp slf4j-1.7.jar:slf4j-2.0.jar:A.jar:B.jar
```

このとき、どちらが「勝つ」かは**並び順**で決まります。
`slf4j-1.7.jar` が先なら、`B.jar` は古い API を見て動かしません。

ビルドツール（第31章の Maven / Gradle）はこの問題を**dependency conflict** として検出し、調整する仕組みがあります。
ですが、ランタイムでも、本質的にはクラスパスの最初に勝った 1 つだけが見えます。

### 2. 「**ClassNotFoundException at runtime**」

`Foo.jar` をコンパイル時に渡しても、実行時のクラスパスに入れ忘れると、

```text
java.lang.ClassNotFoundException: com.foo.Bar
```

が**動かしてから**判明します。
依存関係が**コードと一緒に管理されていない**ので、こうした事故が起きます。

### 3. 「**全部 public**」

パッケージ内で `public class Internal` と宣言すると、それは**JVM 上のすべてのクラスから**見えます。
「これは内部用」とコメントに書いても、誰かが import してしまえば、もう外せません。

これを防ぐために、`com.example.foo.internal` という命名規約を作ったり、Javadoc に `@hidden` を書いたりしてきましたが、**強制力はゼロ**でした。

## 背景2: JDK 自身の保守困難

Java の標準ライブラリは、`java.lang.*`、`java.util.*` の**公開 API** だけでなく、`com.sun.*`、`sun.misc.*`、`jdk.internal.*` などの**内部実装**を抱えています。

たとえば、`sun.misc.Unsafe` という有名な「**禁断の API**」がありました。
これを使うと、

- ヒープに直接バイトを書き込める
- GC をバイパスして大きな配列を作れる
- ロックなしの強力な並行プリミティブを使える

など、本来は不可能なことが「裏技」でできました。
そして、**Cassandra・Netty・Spring** など、世界中の主要ライブラリが、こぞって `Unsafe` を使うようになりました。

問題は、`Unsafe` は **JDK の内部実装**であり、Oracle が「**いつ変更してもよい**」ものだったことです。
変更したら、世界中のライブラリが壊れる。
壊さないために、本来は変更したかったコードを**ずっと維持し続ける**羽目になる。

これが「**JDK が変えられない**」という構造的な問題でした。
モジュールシステムを使えば、JDK 内部のパッケージを「**閉じた**」ものとして宣言できます。
`java.base` から `jdk.unsupported` を取り出さない限り `Unsafe` には触れない ―― このように、**境界を明示**できるようになりました。

## 背景3: 起動の遅さ・サイズの大きさ

Java 8 までの JDK は、`rt.jar` という**巨大な 1 つの jar** で配布されていました[^jep261-jpms]。
中には、Swing、CORBA、JavaFX、JNDI、JDBC、XML、暗号、ロギング ... と、ありとあらゆるものが入っていました。

ですが、ふつうの Web アプリは、Swing も CORBA も使いません。
それでも、メモリには**全部**ロードされ、起動時の探索コストもかかっていました。

モジュール化により、

- 使うモジュールだけを**カスタム JDK** にする（`jlink`）
- 起動時に**必要なものだけ**ロードする
- コンテナのサイズを**数十 MB**まで小さくできる

ことが可能になりました。
これは、**コンテナ・サーバーレス**の時代に、特に重要な変化でした。

---

## 背景4: 信頼できる依存関係

ビルドツール（Maven、Gradle）は、依存関係を `pom.xml` や `build.gradle` に書きます。
ですが、これらは**ビルド時**の話で、**ランタイム**には反映されません。

「**本当は `B` に依存しているのに、`pom.xml` には書いてない**」というケースで、たまたまクラスパスに `B.jar` があれば動いてしまいます。
ある日、`B` を除いたら**他のところで壊れる** ―― そんなことが起きていました。

モジュールシステムでは、**`module-info.java`** に書かれた `requires` だけを使えます。
書いてないモジュールは、**コンパイル時と実行時の両方**で見えません。
依存が**明示**され、**検証**される ―― これがモジュールシステムの大きな価値です。

## モジュールがあると変わる景色

これらの問題を、JPMS は次のように解決します。

| 問題 | JPMS での解決 |
|---|---|
| クラスパス地獄 | **モジュールパス**：明示的な依存だけが見える |
| 全部 public | **`exports`**：公開したいパッケージだけを宣言 |
| 内部 API の暗黙利用 | JDK 内部はモジュール化され、`--add-exports` が必要 |
| 重複 jar | 同じモジュール名は**重複できない**（コンパイル時エラー） |
| `ClassNotFoundException` at runtime | **コンパイル時**に依存不足を検出 |
| 起動の遅さ | `jlink` でカスタム JDK が作れる |

---

## 「**強い気持ち**」と「**現実**」

ここまで読むと、「JPMS は素晴らしい」と思うでしょう。
ですが、**現実には**、大半の Java アプリは依然としてクラスパスベースです。
理由は、

- 既存ライブラリの**モジュール化が遅れた**
- Spring などのフレームワークが、当初は JPMS を**全面採用しなかった**
- ライブラリ作者から見ると、`module-info.java` を維持するのが**負担**

から。

ただ、Java 9 リリースから10 年近くたって、状況は変わりつつあります。

- Java 標準ライブラリは完全にモジュール化済み
- JDK 内部 API は段階的に**閉じられ**ている（Java 16 で `--illegal-access=deny` が既定）
- 主要ライブラリも、`Automatic-Module-Name` を付けて段階的に対応

「**全面採用**」しないチームでも、**JDK のモジュール化の影響**は受けます。
たとえば、リフレクションで `sun.misc.Unsafe` に触れていたコードは、Java 17 以降では `--add-opens` が必要になっています。

つまり、JPMS を**使わない**選択をしても、**ふるまいの違い**は知っておく必要があります。
これが、本書で扱う動機です。

---

## まとめると

- クラスパスの問題は、**重複・暗黙の依存・全部 public**
- JDK 内部 API（`sun.misc.Unsafe`）の濫用が、長年 JDK 進化を阻んでいた
- JPMS は、**モジュール単位の境界**で、これらを整理する
- 多くのアプリは**まだクラスパス**だが、JDK のモジュール化の**影響は全員が受ける**
- ライブラリ作者、カスタム JDK、厳密な境界が必要なチームには、JPMS は強力

[^jep261-jpms]: JEP 261: Module System, [https://openjdk.org/jeps/261](<https://openjdk.org/jeps/261>)。Java 9（2017年）で導入された Java Platform Module System（JPMS、別名 Project Jigsaw）。`module-info.java`、`requires`、`exports`、`jlink` などを規定する。詳細仕様は JSR 376（[https://jcp.org/en/jsr/detail?id=376](<https://jcp.org/en/jsr/detail?id=376>)）。
