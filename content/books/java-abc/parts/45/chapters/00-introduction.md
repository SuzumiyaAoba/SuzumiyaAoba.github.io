## はじめに ― この章で学ぶこと

ここまで本書では、コードを「**ファイル → パッケージ → クラスパス**」という単位で組み立ててきました。
ですが、Java 9 から、もう一段上の単位 ―― **モジュール**（module）が加わりました。
それが、**Java Platform Module System**（JPMS）です。

第45章では、

- なぜモジュールが必要になったのか
- `module-info.java` の書き方
- パッケージの**公開**と**依存**の宣言
- 既存コードへの影響

を扱います。

---

## クラスパスの限界

Java 8 までの世界では、コードの構成単位は **パッケージ**（`com.example.foo`）と**クラスパス**（`-cp lib/foo.jar:lib/bar.jar`）でした。
これで何でもできましたが、規模が大きくなると、いくつもの**綻び**が出てきます。

| 問題 | 症状 |
|---|---|
| **「全部 public」しかない** | パッケージ内で `public` にすれば、誰でも使える。「ライブラリ内部用」を隠せない |
| **依存関係が暗黙** | jar A は jar B が要る、と書いてある場所がない。実行時に `NoClassDefFoundError` で発覚 |
| **クラスパスは平面** | 同じクラス名の jar が 2 つあると「**前のほうが勝つ**」――暗黙の優先順位 |
| **JDK 本体も平面** | `com.sun.*` や `sun.misc.*` が、本来内部用なのに**全部見える** |
| **起動が遅い** | クラスパス全部を舐めないと、使えるクラスが分からない |

特に「**本当はライブラリ内部用なのに、外から触れてしまう**」問題は、長年 Java の悩みの種でした。
モジュールシステムは、これを根本から整理するために導入されました。

---

## モジュールとは ―― 「パッケージの上の単位」

**モジュール**は、

- 複数のパッケージをまとめた**部品**で
- **何を公開するか**（exports）と
- **何に依存するか**（requires）を
- **明示的に宣言**したもの

です。
イメージでは、

```text line-numbers=false
[モジュール: com.example.app]
   依存: java.base, com.example.lib

   公開パッケージ:
     com.example.app.api
   非公開パッケージ:
     com.example.app.internal
```

`module-info.java` というメタファイルを 1 つ書くだけで、

- 「**この api パッケージは外に公開する**」
- 「**internal パッケージは外からは見えない**」
- 「**動かすには com.example.lib が必要**」

がコンパイラと JVM の両方から**検証**されます。
クラスパスの「全部見えて、依存は暗黙」と比べて、**ずっと整然**としています。

## JDK 自身がモジュール化された

Java 9 で、JDK 本体も **97 個のモジュール**に分解されました（数は版によって変わります）。
代表的なものを挙げます。

| モジュール | 何が入っているか |
|---|---|
| `java.base` | `java.lang`、`java.util`、`java.io` など、コアの中のコア |
| `java.sql` | JDBC API |
| `java.xml` | XML 関連 |
| `java.logging` | `java.util.logging` |
| `java.desktop` | Swing、AWT、画像 |
| `java.net.http` | Java 11 で標準化された HTTP クライアント |
| `java.compiler` | コンパイラ API |
| `jdk.jfr` | JFR（第48章） |

このおかげで、**使わないモジュールを抜いた**カスタム JDK を作れるようになりました。
それが、第45章の終わりに少し触れる **`jlink`** です。

---

## 「**モジュール化しない**」という選択肢

ここで早めに言っておきます。
**多くのアプリは、モジュールシステムを使わなくても済みます**。
Spring Boot を含む多くのアプリやライブラリは、**まだクラスパスベース**で運用されています。

ただ、

- ライブラリ作者（公開 API と内部実装を分けたい）
- カスタム JDK を作りたい人
- 「**外から触られたくない**」内部を厳密に管理したい人

には、JPMS は強力な道具です。
本書では、「**何をしているか分かる**」レベルまで踏み込み、必要になったときに使えるようにします。

## この章を読み終えると

第45章を読み終えるころには、次のことができるようになっています。

- なぜ JPMS が導入されたかを、自分の言葉で言える
- 最小限の `module-info.java` を書ける
- `exports` と `requires` の方向を読み解ける
- リフレクションが絡んだときの `opens` の使い方が分かる
- 既存の jar をモジュール化せず混在させる方法が分かる
- **「クラスパス vs モジュールパス**」の境界線が引ける

それでは、最初の節「**なぜモジュールシステムが必要か**」から始めましょう。
