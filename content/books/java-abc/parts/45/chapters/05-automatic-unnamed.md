## 自動モジュールと無名モジュール

JPMS で書いた新しいコードと、**まだ `module-info.java` を持たない既存の jar** ―― 両者をどう共存させるか。
JPMS は、そのための**2 つの橋**を用意しています。

- **自動モジュール**（Automatic Module）
- **無名モジュール**（Unnamed Module）

この節では、その違いと使い分けを見ていきます。

---

## クラスパス時代の jar に起きること

`module-info.class` を持たない jar をモジュールパスに置くと、

- 自動的に**ある名前のモジュール**として扱われる
- すべてのパッケージを **`exports`** している扱い
- すべてのパッケージを **`opens`** している扱い
- **他のすべてのモジュール**を `requires` している扱い（ふんわり依存）

これが**自動モジュール**です。
逆に、`-cp` でクラスパスに置いた場合は、**無名モジュール**として扱われます。

| 配置 | 種別 |
|---|---|
| `--module-path` に置いた、`module-info` ありの jar | **名前付きモジュール**（命名済み） |
| `--module-path` に置いた、`module-info` なしの jar | **自動モジュール** |
| `-cp` または `-classpath` に置いた jar | **無名モジュール** |

---

## 自動モジュールの名前はどう決まるか

`module-info` のない jar には、どうやって「モジュール名」を付けるのでしょうか?
答えは 2 段階です。

1. **`META-INF/MANIFEST.MF`** に `Automatic-Module-Name: ...` があれば、それを使う
2. なければ、**jar ファイル名**から推測する（`foo-bar-1.2.jar` → `foo.bar`）

特に 1 の `Automatic-Module-Name` は、ライブラリ作者からの「**こう呼んでほしい**」という宣言です。
Spring など主要ライブラリは、**完全な JPMS 化はしないが、`Automatic-Module-Name` だけは付けてある**、という中間的な対応をしています。

これにより、JPMS を使うユーザーは:

```java
module com.example.app {
    requires spring.core;     // Spring が Automatic-Module-Name で宣言した名前
}
```

と書けます。ライブラリ側は、`module-info.java` を書かずに済む ―― **ゆるい移行**が可能な仕組みです。

---

## 無名モジュール ―― 「クラスパス世界」を残す

`-cp` または `-classpath` で渡された jar は、**無名モジュール**（unnamed module）として扱われます。
これは事実上、**従来のクラスパス世界**そのものです。

- すべてが `public` で見える
- すべてのモジュールを **読める**
- ふつうに JPMS の境界を**越える**

無名モジュールは、

- **名前のあるモジュールから**は **読めない**（`requires` できない）
- 名前のあるモジュールが公開していないパッケージは**見えない**

という、興味深い非対称性を持ちます。
つまり、無名モジュールから名前付きモジュールへの依存はある程度許されるが、その逆は厳格、というのが基本姿勢です。

---

## 「**移行戦略**」 ―― いきなり全部やらない

実際に既存のアプリを JPMS に乗せるとき、**いきなり全部モジュール化**するのは現実的ではありません。
段階的な戦略があります。

### ステージ1: クラスパスのまま、Java 11 〜 25 で動かす

- すべてのコードと依存は `-cp` に
- 無名モジュールとして扱われる
- 「**JPMS を全く使っていない**」状態

これでも、Java 25 のサンプルは大半が動きます。
JDK 内部 API へのアクセスが必要なら、`--add-opens` を渡す。

### ステージ2: 既存依存を自動モジュールとして使う

- 自分のコードに `module-info.java` を書く
- 既存ライブラリは `--module-path` に置いて、**自動モジュール**として `requires`
- ライブラリ側の対応を**待たない**で先に進める

### ステージ3: 全部命名済みモジュールに

- 主要ライブラリも `module-info` 対応
- 完全に JPMS の世界

ステージ 1 と 2 は、現実の多くのプロジェクトの**現在地**です。
ステージ 3 まで行くのは、相当のメリットがあると判断したチームだけです。

---

## 「Split Package」 ―― 同じパッケージを 2 つのモジュールが提供する罠

JPMS の世界では、**同じパッケージ名**を**複数のモジュール**が公開することは**禁止**されます。
これを **Split Package** と呼びます。

```text line-numbers=false
モジュール A     exports com.example.util
モジュール B     exports com.example.util    ← NG
```

クラスパス時代は、これがしばしば起きていました（同名のパッケージを別々の jar に分けて持つなど）。
JPMS では起動時に**`LayerInstantiationException`** で弾かれます。

この問題のせいで、たとえば古い `javax.annotation` を 2 つの jar が含んでいる、というような状況が、**JPMS にしようとした瞬間に**詰まります。
**移行戦略の最大の壁**になることがあるので、覚えておきましょう。

---

## どう判断するか

「うちのプロジェクトは JPMS にすべきか?」 ―― この問いには、次の指針が役立ちます。

| 状況 | 推奨 |
|---|---|
| **ライブラリ作者**（公開する jar を作る） | **`module-info.java` を書くか、最低限 `Automatic-Module-Name`** |
| **アプリ作者**（Spring Boot で動かす） | **当面はクラスパスのまま**でも問題ない。JDK モジュール化の**影響だけ**理解する |
| **コンテナサイズを削りたい** | **`jlink`** のために、JPMS 移行を検討（次節） |
| **境界を厳密にしたい** | **`module-info`** で `exports` を絞る |

「いますぐ全部 JPMS」ではなく、「**何が嬉しいか**で判断する」 ―― それが、現実的な答えです。

---

## まとめると

- `module-info` なしの jar には、3 種類の置き場所がある:
  - **モジュールパス → 自動モジュール**
  - **クラスパス → 無名モジュール**
- 自動モジュール名は、`Automatic-Module-Name` か jar ファイル名から決まる
- 無名モジュールは、**クラスパス世界**そのもの。名前付きモジュールからは読めない
- 移行は段階的に。**いきなり全部やらない**
- **Split Package** は、JPMS 移行の最大の壁になることがある

次の節では、JPMS を使ってプロジェクトを組む実務的な側面 ―― **Maven / Gradle との統合**と **`jlink`** を扱います。
