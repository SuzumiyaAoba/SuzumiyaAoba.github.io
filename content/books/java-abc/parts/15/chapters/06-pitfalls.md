## よくあるつまずき

ポリモーフィズムで、初心者がつまずきやすいポイントを、まとめて確認します。

---

## 1. 親の型から、子だけのメソッドを呼ぼうとする

第2節で見たとおり、親の型の変数からは、**親が持つメソッドしか呼べません**。

```java
Animal a = new Dog();
a.fetch();   // ✕ fetch() は Dog だけのメソッド → コンパイルエラー
```

`a` の中身は `Dog` ですが、`a` の**型は `Animal`** です。`Animal` に `fetch()` はないので、呼べません。
`Dog` 独自の機能を使いたいときは、第5節の `instanceof` で確認してから、ダウンキャスト（またはパターンマッチング）します。

```java
if (a instanceof Dog dog) {
    dog.fetch();   // ◯ Dog として扱えば呼べる
}
```

---

## 2. instanceof せずにダウンキャストする

中身が違う型なのに、無理にダウンキャストすると、**実行時に `ClassCastException`** になります。

```java
Animal a = new Cat();
Dog dog = (Dog) a;   // 中身は Cat なのに Dog へキャスト → ClassCastException
```

ダウンキャストの前には、**必ず `instanceof` で中身を確認**します。
パターンマッチング `if (a instanceof Dog dog)` を使えば、確認とキャストを安全にまとめてできます。

---

## 3. オーバーライドしていないと、ポリモーフィズムにならない

ポリモーフィズム（動的束縛）は、**メソッドをオーバーライドしている**ことが前提です。
子クラスでオーバーライドしていなければ、当然、親のメソッドがそのまま使われます。

```java
class Animal { void cry() { IO.println("（動物の鳴き声）"); } }
class Fish extends Animal { }   // cry() をオーバーライドしていない

Animal a = new Fish();
a.cry();   // → （動物の鳴き声）  ※ Fish 独自の鳴き声にはならない
```

`Fish` は `cry()` をオーバーライドしていないので、親 `Animal` の `cry()` がそのまま動きます。
「中身に応じたふるまい」をさせたいなら、**子クラスで `@Override` して、ふるまいを定義**する必要があります。

---

## 4. ポリモーフィズムが効くのは「メソッド」

動的束縛（中身の型で決まる）が働くのは、**メソッドの呼び出し**です。
一方、フィールドへの直接アクセスは、変数の型で決まり、動的束縛は働きません。

ただし、第12章で学んだとおり、**フィールドは原則 `private`** にして、外から直接アクセスしないのが基本でした。
`private` フィールドには、そもそも外からアクセスできないので、この問題に出会うこと自体、ほとんどありません。
「**ポリモーフィズムの主役は、オーバーライドしたメソッド**」と覚えておけば十分です。
データのやりとりも、ゲッターなどの**メソッド**を通せば、ちゃんとポリモーフィズムが効きます。

---

## まとめ

- 親の型からは、**親が持つメソッドだけ**呼べる。子独自の機能は `instanceof` ＋ ダウンキャストで
- `instanceof` で確認せずにダウンキャストすると **`ClassCastException`**
- ポリモーフィズムは**オーバーライドが前提**。していないと親のメソッドが動く
- 動的束縛が効くのは**メソッド**。フィールドは `private` にし、メソッド越しに扱うのが基本

次は、この章で学んだ言葉を、用語集としてまとめます。
