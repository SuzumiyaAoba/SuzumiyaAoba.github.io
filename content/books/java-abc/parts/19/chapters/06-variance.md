## 変性とは ― 共変・反変・不変

第14章で、**継承**による「is-a の関係」を学びました。
たとえば `Integer` は `Number` の子（サブ型）なので、「`Integer` は `Number` の一種である」と言えます。
そのため、`Number` の変数に `Integer` を入れられました。

```java
Number n = Integer.valueOf(3);   // Integer は Number の一種なので OK
```

ここで、素朴な疑問がわきます。
中身どうしにこういう親子関係があるとき、それを**入れ物**にまで広げられるのでしょうか。
たとえば「`Integer` のリスト」は、「`Number` のリスト」の一種として扱えるのでしょうか。

```java
List<Number> nums = new ArrayList<Integer>();   // これは許される？
```

この「**中身の親子関係が、入れ物どうしの親子関係にどう伝わるか**」を決めるルールを、**変性**（Variance）と呼びます[^jls-subtyping]。
実は答えは「場合による」で、Java では**配列**と**ジェネリクス**とで、ふるまいが違います。
この節では、まずその考え方の枠組みを、言葉として整理します。

> **補足: ここから先（第6〜9節）は発展的な内容です**
>
> 変性・ワイルドカードは、ジェネリクスの中でも一段階進んだ話題です。
> 前節までで「`<>` を読めて、使える」ようになっていれば、第3部に進む準備はできています。
> ここから先は、「`List<? extends Number>` のような記号を見かけて気になったとき」に読み返す、くらいの気持ちで構いません。

## 3つの「変わり方」

「中身」に親子関係があるとき、「入れ物」どうしの関係は、次の3通りに分かれます。
ここでは、動物の階層 ―― `Animal`（動物）を親に、`Cat`（猫）を子とする関係 ―― を例にします。

```java
class Animal {}
class Cat extends Animal {}   // Cat は Animal の子（Cat is-a Animal）
```

「入れ物」を仮に `箱<T>` と書くとき、`Cat` が `Animal` の子であることをふまえて、`箱<Cat>` と `箱<Animal>` の関係を考えます。

| 変わり方 | 読み | 意味 | 入れ物どうしの関係 |
|---|---|---|---|
| **共変** | Covariant | 中身の関係が、そのまま伝わる | `箱<Cat>` は `箱<Animal>` の子 |
| **反変** | Contravariant | 中身の関係が、**逆転**して伝わる | `箱<Animal>` は `箱<Cat>` の子 |
| **不変** | Invariant | 関係が**伝わらない** | `箱<Cat>` と `箱<Animal>` は無関係 |

言葉で並べると難しく見えますが、要点はシンプルです。

- **共変**（きょうへん）… 「子の入れ物は、親の入れ物の一種」。中身の向きと**同じ**。
- **反変**（はんぺん）… 「親の入れ物は、子の入れ物の一種」。中身の向きと**逆**。
- **不変**（ふへん）… 「子の入れ物と親の入れ物は、別物」。どちらの代入もできない。

> **補足: 「共変」「反変」「不変」の覚え方**
>
> 「変」は「変化（＝親子関係）」のこと。
> **共**変は中身と入れ物が「共に同じ向き」、**反**変は「反対向き」、**不**変は「変化が伝わらない（不通）」、と読むと覚えやすいです。
> なお、ここでの「不変」は、第13章の「不変オブジェクト（Immutable）」とは**別の意味**なので注意してください（同じ日本語ですが、英語は Invariant と Immutable で別物です）。

## なぜ「場合による」のか

変性が3通りに分かれるのは、「**入れ物をどう使うか**」で、安全な向きが変わるからです。
イメージだけ、先につかんでおきましょう。

- **読み出すだけ**の入れ物なら、**共変**が自然で安全です。
  「`Cat` の入れ物」から取り出したものは、必ず `Animal` でもあります。だから「`Animal` の入れ物」として読む分には、問題が起きません。
- **書き込むだけ**の入れ物なら、**反変**が自然で安全です。
  「`Animal` なら何でも入る入れ物」には、`Cat` を入れても当然だいじょうぶです。だから「`Cat` を入れる先」としては、より広い「`Animal` の入れ物」も使えます。
- **読み書き両方**する入れ物は、どちらの都合も満たさなければならず、結局**不変**（別物扱い）にするのが安全です。

この「読み・書き・両方」と「共変・反変・不変」の対応が、次節以降の主役になります。

## Java では、どこに現れるのか

変性は抽象的な話に聞こえますが、Java の身近なところに、すでに顔を出しています。
この章の残りで、順番に確かめていきます。

| 登場する場所 | 変性 | 学ぶ節 |
|---|---|---|
| **配列**（`Cat[]` と `Animal[]`） | 共変 | 第7節 |
| **ジェネリクス**（`List<Cat>` と `List<Animal>`） | 不変（既定） | 第7節 |
| **`? extends`**（上限境界ワイルドカード） | 共変 | 第8節 |
| **`? super`**（下限境界ワイルドカード） | 反変 | 第8節 |
| **メソッドの戻り値**（オーバーライド） | 共変 | 第9節 |

「配列は共変なのに、ジェネリクスはなぜ不変なのか」 ―― まずはこの対比から見ていきましょう。

## まとめ

- **変性**（Variance）とは、「中身の親子関係が、入れ物どうしの関係にどう伝わるか」のルール
- **共変**は同じ向き（子の入れ物は親の入れ物の一種）、**反変**は逆向き、**不変**は伝わらない（別物）
- 入れ物を「読むだけ」なら共変、「書くだけ」なら反変、「両方」なら不変が安全、という対応がある
- Java では、配列は共変、ジェネリクスは既定で不変。ワイルドカードで共変・反変を選べる
- ここでの「不変（Invariant）」は、第13章の「不変オブジェクト（Immutable）」とは別の意味

次の節では、「**配列は共変、ジェネリクスは不変**」という対比を、実際のコードとエラーで確かめます。

[^jls-subtyping]: 型どうしの「子（サブ型）・親（スーパー型）」の関係は、*The Java® Language Specification, Java SE 25 Edition*, §4.10 "Subtyping," [https://docs.oracle.com/javase/specs/jls/se25/html/jls-4.html#jls-4.10](<https://docs.oracle.com/javase/specs/jls/se25/html/jls-4.html#jls-4.10>) で定義されている。「共変・反変・不変（covariance / contravariance / invariance）」は型システム一般の用語で、Java では配列（§4.10.3）が共変、ジェネリクスの型引数が既定で不変、ワイルドカード（§4.5.1）で共変・反変を表す、という形で現れる。
