## はじめに ― この章で学ぶこと

第41章で見たとおり、Java は「**バイトコードを実行時に翻訳**」しながら動きます。
そのおかげで、

- **起動が遅い**（JIT のウォームアップが必要）
- **メモリ使用量が大きい**（JIT・GC・JVM 自身の常駐コスト）
- **CPU を一定割合食う**（JIT コンパイル中）

という、Java の伝統的な弱点が生まれます。
これらを根本から解消するのが、**GraalVM ネイティブイメージ**です。

第49章では、

- ネイティブイメージとは何か
- GraalVM のセットアップ
- ビルドの実践
- リフレクション・リソースのメタデータ
- Spring Boot のネイティブイメージ対応
- そして**トレードオフ**

を扱います。

---

## 「起動が遅い」が許されない時代

Java が長年抱えていた「**起動の遅さ**」は、しかし以前はそれほど問題視されていませんでした。

- サーバーは一度起動すれば**何日も動き続ける**
- 起動コストは**初回だけ**
- 安定運用が大事

なら、起動の遅さは小さなトレードオフでした。
ですが、**コンテナ・サーバーレス**の時代になって、状況が変わりました。

- スケールアウト時、**新インスタンスがすぐ立つ**ことが求められる
- AWS Lambda などのサーバーレスは、コールドスタートが**ユーザーが体感する遅延**になる
- マイクロサービスは**頻繁に**起動・終了する

「Spring Boot で 10 秒待たないと API が返らない」 ―― この古い感覚は、現代のクラウドでは許容しにくくなっています。

---

## ネイティブイメージの発想

GraalVM ネイティブイメージは、

> Java のコードを、**コンパイル時に**ネイティブコードに翻訳しておき、JVM なしで動かす

という、大胆な発想です。
JIT を**ビルド時に**やってしまうので、

- **起動は数ミリ秒〜数十ミリ秒**（数桁速い）
- **メモリ使用量も大幅減**（JVM や JIT のオーバーヘッドが消える）
- **配布物はOSネイティブの実行ファイル**

このような利点が得られます。
Lambda、Cloud Run、Knative ―― サーバーレス系プラットフォームでの Java は、ネイティブイメージで一気に**普通の選択肢**になりました。

---

## トレードオフを正直に

「ネイティブイメージ最高」と言いたいところですが、**現実にはトレードオフ**があります。

| 項目 | JVM | ネイティブイメージ |
|---|---|---|
| 起動時間 | 数秒 | **数ミリ秒** |
| メモリ使用量 | 数百 MB | **数十 MB** |
| スループット（ホット時） | **JIT 最適化で最高** | やや劣る |
| ビルド時間 | 数秒 | **数分** |
| リフレクション | フル対応 | **設定ファイル必須** |
| デバッグのしやすさ | 充実 | 限定的 |

要するに、

- **起動・メモリ**ではネイティブイメージが圧倒的
- **スループット**では JVM の JIT がやや勝る
- **ビルドと運用**はネイティブのほうが大変

ネイティブイメージは、用途次第で**正しく選ぶ**必要があります。
