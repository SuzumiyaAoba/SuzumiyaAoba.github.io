## ネイティブイメージのトレードオフ

ネイティブイメージは、起動とメモリで圧勝です。
ですが、**何でもネイティブ化すべき**かというと、答えは「**No**」です。
この節では、ネイティブイメージを選ぶか選ばないかの**判断軸**を整理します。

---

## ピークスループットでは JIT が勝つ

JVM の JIT は、**実行時情報**を見て最適化できます。

- どのメソッドが**よく呼ばれるか**
- 仮想呼び出しの**実装が実は 1 つ**かどうか
- ループの**実際の反復回数**

これらをもとに、**インライン化・分岐予測**を最適化します。
ネイティブイメージは、これを**ビルド時**にやるしかないので、ピーク性能では JIT に劣ることがあります。

| 用途 | 推奨 |
|---|---|
| **長時間動かす Web サーバー** | **JVM**（JIT で速くなる時間がある） |
| **短命のコマンド・関数** | **ネイティブ**（JIT が走る前に終わる） |
| **CPU-bound バッチ** | **JVM**（ピークスループットが効く） |
| **I/O-bound マイクロサービス** | **ネイティブ**（I/O 待ちが多いので JIT メリットが少ない） |

---

## ビルド時間と CI コスト

ネイティブビルドは**重い**です。

- 1〜2 分（小さなアプリ）
- 10〜30 分（Spring Boot 中規模アプリ）
- **CPU・メモリ**も大量に使う

CI で毎 PR ごとにネイティブビルドを走らせると、

- ビルド時間が**伸びる**
- CI コストが**膨らむ**
- 開発者の待ち時間が**増える**

対策として、

- **PR 段階では JVM ビルド + テスト**だけ
- **main マージ後・夜間バッチ**でネイティブビルド
- **本番デプロイ前**だけネイティブ

という、段階的な CI 設計が現実的です。

---

## デバッグのしにくさ

ネイティブイメージは、

- ふつうのデバッガ（IntelliJ・jdb）が**そのままでは使えない**
- スタックトレースは出るが、**変数の中身を覗く**のは難しい
- リモートデバッグの仕組みが**限定的**

特に**本番でしか出ない不具合**を追うとき、JVM なら `jmap`・`jstack`・JFR が使える一方、ネイティブイメージでは**できないことが多い**です。
GraalVM は GDB を使ったデバッグもサポートしていますが、慣れが必要です。

「**運用デバッグが必要なシステム**」は、JVM のままにする選択もあります。

---

## ライブラリ互換性

主要ライブラリ（Spring Boot 3、Jackson、Hibernate、Micronaut、Quarkus）は、ネイティブ対応が進んでいます。
ですが、まだ対応していないライブラリも多くあります。

たとえば、

- 古い JPA プロバイダ
- リフレクション中心の DI コンテナ（古い Spring 2 系）
- 動的なバイトコード生成（cglib のレガシー版）
- スクリプトエンジン（Nashorn は Java 15 で削除されたが、後継のスクリプト系）

これらを使う部分は、**ネイティブ化を諦める**か、**他のライブラリに置き換える**判断が必要です。

---

## 「**全部ネイティブ**」vs「**選んでネイティブ**」

現実的なシステム構成は、しばしばこんな組み合わせになります。

```text
[フロントエンド: React]
        │
        ▼
[API Gateway: ネイティブ Spring Boot]   ← 高速起動が重要
        │
        ▼
[業務サービス: 通常 JVM Spring Boot]    ← スループットが重要
        │
        ▼
[バッチ: 通常 JVM]                       ← 長時間動くので JIT 有効
```

- **API Gateway** のような I/O 中心・短時間動作 → **ネイティブ**
- **業務サービス** のような長時間動作・スループット重視 → **JVM**
- **バッチ** → **JVM**

このように、**役割で使い分ける**のが正解です。

---

## サーバーレスでの効果

サーバーレス（AWS Lambda・Cloud Run・Knative）では、ネイティブイメージのメリットが**特に大きい**です。

| | JVM Lambda | ネイティブ Lambda |
|---|---|---|
| コールドスタート | 2〜10 秒 | **数百 ms** |
| 課金時間（起動含む） | 長い | **短い** |
| ピーク性能 | JIT 後は速い | 安定して速い |

Lambda の課金は**実行時間**ベースなので、起動が速いだけで**コストが下がります**。
サーバーレスで Java を使うなら、ネイティブイメージは**ほぼ必須**と言ってよいでしょう。

---

## 「**いつネイティブにすべきか**」の判断

最後に、判断軸を要約した質問リストです。

1. **起動時間がユーザーから見えるか?**
   - はい → ネイティブ候補
   - いいえ → JVM のままで OK

2. **メモリ使用量を圧縮したいか?**
   - はい（小さなコンテナ、Lambda） → ネイティブ
   - いいえ → JVM のままで OK

3. **長時間連続動作するか?**
   - はい → JVM の JIT 最適化を活かす
   - いいえ → ネイティブ候補

4. **使うライブラリがネイティブ対応しているか?**
   - はい → ネイティブビルドを試す
   - いいえ → JVM のまま、または別ライブラリに置き換え

5. **CI ビルド時間に余裕があるか?**
   - はい → ネイティブを CI に含める
   - いいえ → 段階的な CI 設計

これらにすべて答えてから、判断する ―― それが現実的なネイティブ化の進め方です。

---

## まとめると

- ネイティブイメージは、**起動・メモリ**で圧勝、**ピーク性能**では JIT がやや勝つ
- ビルド時間・CI コスト・デバッグのしにくさはトレードオフ
- 用途に応じて、JVM とネイティブを**使い分ける**
- サーバーレスでは、ネイティブが**ほぼ必須**
- 判断軸: **起動・メモリ・寿命・ライブラリ・CI 余裕**
