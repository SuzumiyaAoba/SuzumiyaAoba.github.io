## エラーと向き合う ― はじめてのコンパイルエラー

プログラムを書いていると、必ず出会うのが**エラー**です。

「エラー」と聞くと、なんだか失敗したような、こわい印象があるかもしれません。
ですが、第1章で学んだとおり、エラーは**コンピュータが「ここに問題があります」と教えてくれる合図**です。
あわてず、メッセージを読み、落ち着いて直していけば大丈夫です。

このセクションでは、初心者がとくに出会いやすい**コンパイルエラー**を取り上げ、その読み方と直し方を学びます。

---

## コンパイルエラーとは

第1章で学んだように、人間が書いたソースコードは、そのままでは実行できません。
`javac` などによって**コンパイル**され、JVM が読めるバイトコードに変換されてから、実行されます。

このコンパイルの段階で、文法の間違いが見つかったときに出るのが、**コンパイルエラー**です。

```text
ソースコードを書く
   ↓
コンパイルする  ←── ここで文法の間違いがあると「コンパイルエラー」
   ↓
実行する        ←── ここで起きる問題は「実行時エラー」
```

コンパイルエラーは、**プログラムが実行される前**に見つかります。
言いかえれば、「実行する前に間違いを教えてもらえる」ということです。
これは、Java の心強い長所の一つです。

> **補足: IDE なら、書いたそばから教えてくれる**
>
> 第2章で入れた IntelliJ IDEA を使っていると、コンパイルする前から、間違っていそうな部分に**赤い波線**を引いて教えてくれます。
> マウスを近づけると、何が問題かの説明も表示されます。
> この節では、エラーの「中身」を理解するために、コマンドラインで `javac` を実行したときのメッセージを例に説明します。

> **補足: エラーメッセージは、環境によって言葉が変わる**
>
> Java のエラーメッセージは、**コンピュータの言語設定**によって、表示される言葉が変わります。
> 日本語環境では**日本語**で、英語環境では**英語**で表示されます。
> 本書では、日本語環境での表示を主な例とし、あわせて英語の表示も併記します。
>
> また、メッセージの細かい文言は、JDK のバージョンによっても少しずつ変わります。
> 大切なのは、文言を丸暗記することではなく、**どこに何が書かれているか**を読み取れるようになることです。

---

## エラーメッセージの読み方

エラーが出たら、まず**メッセージを読む**ことが何より大切です。
見慣れない言葉に身構えてしまうかもしれませんが、見るべき場所は決まっています。

たとえば、文の終わりの `;` を忘れたとき、`javac` は次のようなメッセージを表示します。

```text
Main.java:3: エラー: ';'がありません
        System.out.println("Hello, World!")
                                           ^
エラー1個
```

このメッセージは、次のように読み解けます。

| 部分 | 意味 |
|---|---|
| `Main.java` | エラーが起きた**ファイル名** |
| `:3` | エラーが起きた**行番号**（3行目） |
| `エラー: ';'がありません` | エラーの**内容**（「`;` があるはずなのに、ない」） |
| 問題の行＋`^` | 問題のある**場所**を指し示す（`^` が指している位置） |
| `エラー1個` | エラーの**総数**（全部で1件） |

つまり、「**`Main.java` の3行目あたりで、`;` が足りない**」と教えてくれているのです。

エラーメッセージを読むときのポイントは、次の3つです。

1. **どのファイルの、何行目か**（場所）
2. **何が問題だと書かれているか**（内容）
3. **直前に、何を変えたか**（きっかけ）

この3つを手がかりにすれば、たいていの間違いは見つけられます。

## よくあるエラーと、その直し方

ここからは、初心者がとくに出会いやすいエラーを、症状ごとに見ていきます。
「こういうメッセージが出たら、ここを疑う」という対応を、少しずつ覚えていきましょう。

以下では、日本語環境での表示を示し、英語環境での表示もあわせて記します。
（前述のとおり、文言は JDK のバージョンによって多少変わることがあります。）

### 1. セミコロン（`;`）の付け忘れ

文の終わりの `;` を忘れると、次のようなエラーが出ます。

```java
System.out.println("Hello, World!")   // ← ; を忘れている
```

```text
Main.java:3: エラー: ';'がありません
```

> 英語環境では `error: ';' expected` と表示されます。

**「`;`がありません」**（`;' expected`）と出たら、その行か、その少し手前の行に、`;` の付け忘れがないか確認しましょう。

### 2. 波かっこ（`{ }`）の対応ミス

`{` を開いたまま、`}` で閉じ忘れると、次のようなエラーが出ます。

```java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
// ← クラスを閉じる } を忘れている
```

```text
Main.java:4: エラー: 構文解析中にファイルの終わりに移りました
```

> 英語環境では `error: reached end of file while parsing` と表示されます。

**「構文解析中にファイルの終わりに移りました」**（`reached end of file while parsing`）と出たら、`}` の閉じ忘れを疑います。
「ここで終わるはずがないのに、ファイルが終わってしまった」という意味です。
`{` の数と `}` の数が合っているかを、数えて確かめましょう。

### 3. つづり間違い・大文字小文字の間違い

`println` を `printline` と打ち間違えるなど、名前のつづりを間違えると、次のようなエラーが出ます。

```java
System.out.printline("Hello, World!");   // ← println が printline になっている
```

```text
Main.java:3: エラー: シンボルを見つけられません
        System.out.printline("Hello, World!");
                  ^
  シンボル:   メソッド printline(String)
  場所: タイプPrintStreamの変数 out
```

> 英語環境では `error: cannot find symbol` と表示されます。

**「シンボルを見つけられません」**（`cannot find symbol`）は、初心者がもっともよく出会うエラーの一つです。
「あなたが書いた名前を、Java が知らない」という意味で、つづりの間違いや、大文字・小文字の取り違えが原因のことがほとんどです。
`シンボル:`（`symbol:`）のうしろに、見つからなかった名前（ここでは `printline`）が書かれているので、そこが手がかりになります。

> **補足: `System` 自体を間違えたとき**
>
> `System` を `system`（小文字）や `Sytem` のように間違えると、同じ「名前の間違い」でも、次のような別のメッセージが出ます。
>
> ```text
> Main.java:3: エラー: パッケージsystemは存在しません
> ```
>
> （英語環境では `error: package system does not exist`）
>
> これは、Java が `system.out` の `system` を「パッケージ（部品の置き場所）の名前」だと解釈しようとして、「そんな置き場所はない」と言っているためです。
> メッセージの言葉は違いますが、**原因は同じ「名前の書き間違い」**です。第5節で学んだとおり、`System` の `S` は大文字、と確認しましょう。

### 4. 全角の記号が混ざっている

記号を全角で打ってしまうと、次のようなエラーが出ます。

```java
System.out.println（"Hello, World!"）;   // ← かっこが全角（ ）になっている
```

```text
Main.java:3: エラー: '\uff08'は不正な文字です
        System.out.println（"Hello, World!"）;
                          ^
```

> 英語環境では `error: illegal character: '\uff08'` と表示されます。

**「不正な文字です」**（`illegal character`）と出たら、**全角の記号が混ざっていないか**を疑いましょう。
メッセージの `\uff08` という見慣れない表示は、問題の文字（ここでは全角の `（`）を表す番号（文字コード）です。
番号で示されるため一見わかりにくいですが、「全角の記号が紛れ込んでいる」というサインだと考えてください。
全角の記号が混ざると、その1か所が原因で「`文ではありません`」など別のエラーがいくつも連なって出ることもあります。
あわてず、まずは全角の記号を半角に直してみましょう。
第2章・第5節でも注意したとおり、プログラムの記号はすべて半角で入力します。

### 5. ダブルクォート（`"`）の閉じ忘れ

文字列を囲む `"` を閉じ忘れると、次のようなエラーが出ます。

```java
System.out.println("Hello, World!);   // ← 閉じる " を忘れている
```

```text
Main.java:3: エラー: 文字列リテラルが閉じられていません
```

> 英語環境では `error: unclosed string literal` と表示されます。

**「文字列リテラルが閉じられていません」**（`unclosed string literal`）と出たら、`"` を開いたまま閉じ忘れていないかを確認します。
（「リテラル」とは、`"Hello, World!"` のように、コードに直接書いた値のことです。）
`"` は、必ず開いたら閉じる、とセットで覚えておきましょう。

### 6. ファイル名とクラス名の不一致

第2節で学んだ「`public` なクラスは、クラス名とファイル名を同じにする」というルールを破ると、次のようなエラーが出ます。

たとえば、`public class Main` と書いたコードを、`Hello.java` という名前で保存した場合です。

```text
Hello.java:1: エラー: クラス Mainはpublicであり、ファイルMain.javaで宣言する必要があります
```

> 英語環境では `error: class Main is public, should be declared in a file named Main.java` と表示されます。

メッセージに、**「ファイルMain.javaで宣言する必要があります」**と、正しいファイル名がそのまま書かれています。
言われたとおりにファイル名を `Main.java` に直せば解決します。

### 7. main が見つからない

コンパイルは通っても、`main` メソッドがなかったり、形が間違っていたりすると、**実行する段階**で次のようなエラーが出ます。

```text
エラー: メイン・メソッドがクラスMainで見つかりません。次のようにメイン・メソッドを定義してください。
   public static void main(String[] args)
```

> 英語環境では `Error: Main method not found in class Main, ...` と表示されます。

**「メイン・メソッドが…で見つかりません」**（`Main method not found`）と出たら、第3節で学んだ `public static void main(String[] args)` の形が、正しく書けているかを確認しましょう。
親切なことに、メッセージのなかに、あるべき正しい形まで書いてくれています。
これは、コンパイルエラーではなく、**実行時のエラー**（実行しようとした段階で出るエラー）です。

## 直すときの心がまえ

たくさんエラーが出ると、不安になるかもしれません。
ですが、次のことを心にとめておけば、落ち着いて対処できます。

- **いちばん上のエラーから直す** … 1つの間違いが、いくつものエラーを引き起こすことがあります。まずは最初のエラーを直して、もう一度コンパイルしてみましょう。
- **直前に変えたところを疑う** … さっきまで動いていたなら、原因は直前の変更にあることがほとんどです。
- **1つ直したら、すぐ試す** … まとめて直そうとせず、1か所直すごとにコンパイルすると、原因を切り分けやすくなります。
- **エラーは「失敗」ではない** … 第1章でも学んだとおり、エラーは問題の場所を教えてくれる合図です。経験豊富なプログラマーも、毎日エラーと付き合っています。

> **コラム: エラーメッセージは、いちばん親切な先生**
>
> 慣れないうちは、エラーメッセージを読み飛ばして、やみくもにコードをいじってしまいがちです。
> ですが、エラーメッセージには、「どこで」「何が」問題なのかが、きちんと書かれています。
>
> 日本語環境なら、まずはそのまま日本語として読んでみましょう。
> 英語環境でも、`expected`（〜があるはず）、`cannot find`（見つからない）、`illegal`（使えない）といった単語に注目するだけで、ぐっと読み解きやすくなります。
> エラーメッセージは、あなたのいちばん近くにいる、親切な先生だと思って、まずは落ち着いて読んでみましょう。

> **コラム: それでも詰まったときの「調べ方」**
>
> メッセージを読んでも、どうしても意味がわからない ―― そんなときの、いちばん確実な解決法をお伝えします。
> それは、**エラーメッセージを、そのまま検索する**ことです。
>
> メッセージの中心となる部分（たとえば `';'がありません` や `cannot find symbol`）をコピーし、`Java` という言葉を添えて検索してみましょう。
> 同じエラーに悩んだ人の質問や、その解決法が、たくさん見つかります。
>
> このとき、**英語のメッセージのほうで検索する**と、見つかる情報がぐっと増えます。
> 本書が各エラーに英語表記も併記しているのは、このためです。日本語環境の人も、英語のほうをコピーして検索すると探しやすくなります。
>
> 公式ドキュメントを調べたり、まわりの人や AI に質問したりするのも、とてもよい方法です。
> 「調べて解決する」のは、決して悪いことではありません。むしろ、**経験豊富なプログラマーほど、上手に調べ、上手に人に聞きます。**
> 自分の力で詰まりを抜け出せるようになることは、コードを書く力と同じくらい大切な技術です。

## まとめ

このセクションでは、コンパイルエラーの読み方と、その直し方を学びました。

- **コンパイルエラー**は、コンパイルの段階で文法の間違いが見つかったときに出る。実行する前に教えてもらえる
- エラーメッセージは、**ファイル名・行番号・内容・場所（`^`）** を見れば読み解ける
- メッセージの言葉は、コンピュータの**言語設定**（日本語／英語）や JDK のバージョンによって変わる
- よくあるエラーと対応：
  - `';'がありません`（`';' expected`） → セミコロンの付け忘れ
  - `構文解析中にファイルの終わりに移りました`（`reached end of file while parsing`） → `}` の閉じ忘れ
  - `シンボルを見つけられません`（`cannot find symbol`） → 名前のつづり・大文字小文字の間違い
  - `パッケージ…は存在しません`（`package … does not exist`） → `System` などの名前の書き間違い
  - `…は不正な文字です`（`illegal character`） → 全角の記号が混ざっている
  - `文字列リテラルが閉じられていません`（`unclosed string literal`） → `"` の閉じ忘れ
  - `…で宣言する必要があります`（`should be declared in a file named …`） → ファイル名とクラス名の不一致
  - `メイン・メソッドが…で見つかりません`（`Main method not found`） → `main` の形の間違い（実行時エラー）
- 直すときは、**上のエラーから・直前の変更を疑い・1つずつ**を心がける

エラーと向き合う準備ができたところで、この章の最後のセクションでは、Java 25 で使えるようになった、もっとシンプルなプログラムの書き方を紹介します。
