## 列挙型とは ― 決まった選択肢を表す

**列挙型**（Enum）は、「**取りうる値が、あらかじめ決まっている**」データを表す型です。
この節では、なぜ列挙型が、文字列や数値よりも優れているのかを学びます。

---

## 文字列で表すと、何が困るのか

信号の色を、文字列で表す場合を考えます。

```java
String signal = "RED";

if (signal.equals("RED")) {
    IO.println("止まれ");
}
```

これでも動きますが、次のような危険があります。

```java
String signal = "red";    // 小文字で書いてしまった
String signal = "RDE";    // 打ち間違えた
String signal = "PURPLE"; // 存在しない色を入れてしまった
```

これらは、どれも**コンパイルエラーになりません**。
`String` には「どんな文字列でも入る」ので、「信号の色として正しいか」を Java はチェックできないのです。
そのため、打ち間違いや、ありえない値が紛れ込んでも、**実行して初めておかしくなる**、という事態になります。

---

## 列挙型なら、取りうる値を限定できる

そこで、列挙型の出番です。
「信号の色は、`RED`・`YELLOW`・`GREEN` の3つだけ」と、`enum` で定義します。

```java
enum Signal {
    RED, YELLOW, GREEN
}
```

これで、`Signal` 型の変数には、`Signal.RED`・`Signal.YELLOW`・`Signal.GREEN` の**3つしか入れられなくなります**。

```java
Signal signal = Signal.RED;     // ◯ OK
Signal signal = Signal.PURPLE;  // ✕ そんな値はない → コンパイルエラー
```

存在しない `PURPLE` を入れようとすると、**コンパイルの段階でエラー**になります。
「信号の色として、ありえない値」が入る余地が、そもそもなくなるのです。
打ち間違いも、入力補完（IDE が候補を出してくれる）で防げますし、つづりを間違えればエラーで分かります。

## 列挙型の利点

文字列・数値ではなく列挙型を使うと、次の利点があります。

| | 文字列・数値 | 列挙型 |
|---|---|---|
| ありえない値 | 入ってしまう | **入れられない**（コンパイルエラー） |
| 打ち間違い | 実行時まで気づけない | **コンパイル時に気づける** |
| 入力補完 | 効きにくい | **効く**（候補が出る） |
| 意味の分かりやすさ | `0`・`1` では不明 | `RED`・`GREEN` で明確 |

ひとことで言えば、列挙型は「**取りうる値を、型のレベルで限定する**」ことで、安全性と分かりやすさを高めてくれます。
「選択肢が決まっているもの」は、文字列や数値ではなく、列挙型で表すのが、Java の定石です。

> **補足: 「列挙」という言葉**
>
> 「列挙（れっきょ）」とは、「1つひとつ並べて挙げること」という意味です。
> `enum Signal { RED, YELLOW, GREEN }` は、まさに「信号の色を、1つひとつ並べて挙げている」わけです。
> `enum` は英語の enumeration（列挙）の略です。

---

## まとめ

- **列挙型**（`enum`）は、「取りうる値が決まっている」データを表す型
- 文字列・数値で表すと、**ありえない値や打ち間違いを防げない**（実行時まで気づけない）
- 列挙型なら、定義した値以外は**コンパイルエラー**になり、安全
- 入力補完も効き、`RED`・`GREEN` のように**意味も明確**になる
- 「選択肢が決まっているもの」は、列挙型で表すのが定石
