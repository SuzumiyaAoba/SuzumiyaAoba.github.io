# Java 入門 執筆ガイド

このディレクトリは `content/books/java-abc/` に置かれた Java 入門書のコンテンツです。
Claude はこのガイドに従って執筆をサポートしてください。

## 書籍の概要

- **タイトル**: Java 入門
- **対象読者**: プログラミング完全初心者（プログラミング経験ゼロを想定）
- **ターゲット Java バージョン**: Java 25
- **目的**: プログラミングの基礎から始め、Java の最新機能・実務で必要な知識（設計・テスト・API 設計など）まで幅広く解説する

## 執筆方針

### 1トピックずつ確認しながら進める

- 一度に大量に書かない。1トピック（1節〜1章）ずつ執筆し、ユーザーが内容を確認してから次へ進む
- 執筆前に「このトピックで説明する内容の概要」を提示し、合意を得てから本文を書く
- 誤りが混入しやすい技術的な細部（仕様の挙動、バージョン差異、API の動作）は特に慎重に記述する

### 正確性の確認ポイント

コードや技術的な説明を書く際は以下を確認すること：

- **コンパイル可能か**: サンプルコードは Java 25 でコンパイル・実行できる形にする
- **JLS 準拠**: Java Language Specification に沿った説明をする
- **最新 API の使用**: 古い・非推奨の API は使わない。Java 25 時点の標準的な書き方を採用する
- **バージョン差異の明示**: 特定バージョンで導入された機能は明示する（例: Records は Java 16+、Virtual Threads は Java 21+）
- **一般的な誤解の排除**: よくある誤解（例: `==` と `.equals()` の混同、null の扱い）を助長しない

### 文体・スタイル

- 敬体（です・ます調）で統一する
- 専門用語は初出時に日本語と英語を併記する（例: 参照型（Reference Type））
- コードブロックには言語を指定する（` ```java `）
- 長い説明は箇条書きや表を活用して読みやすくする
- 完全初心者向けなので「なぜそうなのか」の理由を丁寧に説明する
- 難しい概念は身近な例えを使って説明してから技術的な解説に入る
- 前の章で学んだ内容を参照・活用しながら積み上げ式で解説する

## ディレクトリ構成

書籍は論理的に「部（Part）＞章（Chapter）＞節（Section）」の3階層で構成する。
ただしディレクトリは「章＞節」の2階層で管理し、部（Part）はディレクトリには表さず `index.md` の目次でまとめる。
節は1ファイルずつに分割して管理する。

```
content/books/java-abc/
├── CLAUDE.md                       # このファイル
├── index.md                        # 書籍トップ（概要・目次）
├── images/                         # 画像・アセット（ブック全体で共有）
│   └── *.svg / *.png / *.jpg …
└── parts/
    └── {NN}/                       # 章（通し番号。第1章 = 01、第2章 = 02 …）
        └── chapters/
            ├── 00-introduction.md   # 章の導入（章扉。任意）
            ├── {NN}-{slug}.md       # 各節（01 から連番。slug は英語ケバブケース）
            └── {NN}-glossary.md     # 章末の用語集（任意）
```

- `parts/{NN}/` の `NN` は章の通し番号（部をまたいで `01`〜`40`）をゼロ埋め2桁で表す。部（Part）はディレクトリでは分けない。
- 各節のファイルは `parts/{NN}/chapters/` の直下に置く（章ごとのサブディレクトリは作らない）。
- 節ファイルは `01-` から連番。導入は `00-`、用語集など章末の付録は最後の番号に置く。
- ファイル名の `slug` は内容を表す英語のケバブケース（例: `what-is-program`）。

## フロントマター形式

`index.md`（書籍トップ）:
```yaml
---
title: Java 入門
date: YYYY-MM-DD
category: プログラミング
tags: ["Java", "プログラミング", "入門"]
---
```

各節のファイル（`parts/{NN}/chapters/*.md`）:
```yaml
---
title: 節のタイトル
llm: true   # 執筆メタデータ。AI 生成の度合いを示す（true / partial）
---
```

- `title` は節の見出しに使う。本文の冒頭にも同じ文言で `## {title}` を置く。
- `llm` はこの書籍内の執筆メタデータ（`true`＝AI 生成、`partial`＝一部 AI 生成）。サイト表示には現時点で未使用。

## 画像・アセット

図版や画像は、**ブック直下の `images/` ディレクトリにまとめて配置する**（`content/books/<book>/images/`）。
章・節ごとには分けない。

- **参照のしかた**：節の Markdown からは、階層の深さに関わらず `./images/<file>` で参照する。

  ```markdown
  ![入力・処理・出力の流れ](./images/input-process-output.svg)
  ```

- **配信のしくみ**：`content/` 配下の画像は `app/contents/[...path]` ルートが
  `/contents/<相対パス>` で静的配信する。上記の `./images/foo.svg` は、レンダリング時に
  `/contents/books/<book>/images/foo.svg` へ解決される。
- **解決の基盤**：MDX をレンダリングする際は、`basePath` に
  `bookContentBasePath(<bookSlug>)`（`src/foundation/shared/lib/books/content-path.ts`）を渡す。
  これにより、どの節からでも `./images/...` がブック直下の `images/` を指す。
  （※ 書籍ページのレンダラ本体は未実装。実装時にこの basePath を渡せばよい。）
- **図版の方針**：図は原則 **SVG** で作成する（和文対応フォント・角丸・統一配色）。
  アクセシビリティのため `<title>` / `<desc>` を付け、Markdown 側にも代替テキスト（alt）を書く。
- `.png` / `.jpg` / `.jpeg` は配信時に `.webp` も自動生成される（SVG は対象外で、そのまま配信）。

## 執筆の進め方（ワークフロー）

1. ユーザーが「次の章（節）を書いて」と指示する
2. Claude がそのトピックで扱う内容の概要を箇条書きで提示する
3. ユーザーが概要を確認・修正する
4. Claude が本文を執筆する
5. ユーザーがレビューし、修正点を指摘する
6. 修正後、次のトピックへ進む

## 注意事項

- AI が生成したコードであっても必ずユーザーが動作確認する
- 仕様が曖昧な場合は断言せず「仕様では〜とされています」と出典を示す
- Java SE の公式ドキュメント（docs.oracle.com）や JLS を根拠にする
- 完全初心者が読むため、前提知識をできる限り持ち込まない
- 初心者がつまずきやすいポイント（null、型変換、参照渡しなど）は丁寧に解説する
